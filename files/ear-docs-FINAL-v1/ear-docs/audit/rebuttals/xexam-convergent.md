## GROUP A — Field-drift: R1/L1 prose (`time`, `direction`) vs v5 signal schema (`observed_at`, no `direction`, `additionalProperties:false`)
### Targets: FINDING-LEAD.2, FINDING-DE.11, FINDING-QA.8 (adjudicated together — same underlying defect, three lenses)

**rebuttal (the strongest kill attempt).** This is the finding most exposed to a `rationale_addressed` dismissal, because D-052 is almost custom-built to answer it: *"Legacy names (`time`, `power`, `agent`) are NOT valid v5 — the fixture/adapter maps them. Schemas are the tie-breaker over prose on field questions"* (DECISION-LOG L65). On its face the tie-breaker rule fully resolves the conflict: a disciplined build agent hitting `time` in R1 and `observed_at` in the schema applies D-052 and picks `observed_at`. The reviewers even grant this in their own steelmen. So the naive move is: mark all three `rationale_addressed`, quote D-052, done.

**why the kill fails.** The tie-breaker rule resolves the *field-name* collision (`time` vs `observed_at`) but does **not** resolve the two harder halves of the finding, and the reviewers correctly separated them:

1. **`direction` is not a name collision — it is a presence/validity collision that the tie-breaker makes WORSE, not better.** The schema sets `additionalProperties:false` and has no `direction` property. R1 L43 and L1 L93/L102 mandate `direction ∈ {ul,dl}` as an emitted-exactly field and a validated invariant. Applying D-052 (schema wins) does not gently downgrade the prose — it means **every row an R1-faithful agent emits is quarantined 100%** (forbidden extra property + wrong timestamp name). D-052 does not *say* to drop `direction`; it just makes the prose that requires it fatal. So "schema is the tie-breaker" is the mechanism by which the defect becomes a total-quarantine failure, not the mechanism that prevents it. The DECISION-LOG resolves the *authority question* but leaves the *content contradiction* live and now weaponized.

2. **R1 is a `[LAW]` producer-guarantee document and its ENFORCED-BY clause points the builder at a schema filename that does not exist** (`observation.schema.json` — actual files are `observation-signal.v5.schema.json` / `observation-context.v5.schema.json`, QA.8/DE.11 both cite L56). A build agent that trusts the ENFORCED-BY pointer cannot even reach the tie-breaker schema. D-052 says "schemas win" but the one contract that is supposed to route the builder to the schema mis-names it. That is not addressed anywhere in the log.

3. **The tie-breaker is a *disposition rule for humans/careful agents at conflict-resolution time*, not a *build-time guardrail*.** The reviewers' shared point — and it is correct — is that the fixture *defines truth* (D-057/D-037). Whichever field name the S03 fixture-generator author happens to type becomes the golden expected-output that every downstream persona is diffed against. If they read R1 first (it is *the* producer guarantee, read first by construction) they encode `time`, the schema test goes red, and the "fix" an agent reaches for could as easily be "relax the schema" as "fix the prose." D-052 tells you who wins a *noticed* conflict; it does nothing to guarantee the conflict is noticed before it is cemented into the fixture. The reviewers' proposed CI check (diff R1's enumerated field list ⊆ schema properties) is exactly the missing guardrail, and it is *not* in the DECISION-LOG.

**cross-lens convergence hardens this.** Three independent lenses — Lead (whole-system consistency), Data-Eng (loader/quarantine), Test-QA (fixture-as-truth) — reached the same defect from three vantages and all three explicitly engaged D-052 in their steelman and still filed it. When the reviewers who *know* the tie-breaker rule and quote it still conclude the finding stands, the tie-breaker is not a refutation; it is the finding's premise. DE.11 filed it `minor` (its lens sees only the loader mechanically quarantining), but QA.8 and LEAD.2 filed it `major` because they see the fixture-cements-wrong-truth and 100%-quarantine consequences. The correct severity is the **major** the two truth-aware lenses assigned, not DE.11's minor — the cross-exam *raises* DE.11's severity to match.

**verdict: hardened** (all three; DE.11's severity is corrected upward from minor→major in the process).
The DECISION-LOG D-052 tie-breaker is real and should be quoted, but it resolves only 1 of the 3 sub-conflicts (the name), and it does so by *escalating* the `direction`/`additionalProperties:false` sub-conflict to a 100%-quarantine failure rather than curing it. The ENFORCED-BY dangling filename and the absence of a schema↔prose CI diff are wholly un-addressed. This is not `rationale_addressed`; it is a live pre-repo contradiction that D-052 arbitrates the *authority* of but does not *reconcile the content* of.

**residual_claim (survives at full strength, hardened):** Before WP-F1/WP-A2/S03: in R1 and L1 rename `time`→`observed_at`; delete `direction` from R1/L1 and the validator description (or add it to the schema — but uplink-only makes it a droppable constant); fix R1's ENFORCED-BY filename to the two real v5 schema files; align `duty_cycle` optionality and add `config_version` to R1's list. Add the CI meta-check (prose field list diffed against schema `properties`). Added harden: state explicitly, once in R1, "the JSON Schema is authoritative for field name/presence; this prose list is descriptive," so the tie-breaker is *visible at the point of authorship*, not buried in D-052.

**downstream:** R1 + ENFORCED-BY, L1 (L93/L102), D-052, D-034 (schemas imported by both suites), the S03 fixture generator and every signal-row persona (P1/P2/P6/P8/P15/P17), P9 (malformed-row: a `time` row *should* now quarantine — intended, but assert it), the S05 legacy adapter (`time`→`observed_at` target must be correct). No cross-lens conflict — this is a clean 3-lens convergence.

## GROUP B — presence_ratio 0.95 (methodology/D-028/scope) vs 0.8 (PARAMETER-TUNING L42)
### Targets: FINDING-LEAD.4, FINDING-RF.4, FINDING-QA.2

**rebuttal (kill attempt).** The strongest rebuttal is the hierarchy argument the reviewers themselves raise and then reject: (a) PARAMETER-TUNING's own header declares it "forward-looking … not yet built," making its table illustrative; (b) the methodology (which D-028 points at) is the binding contract and states 0.95 in two places (L105, L127); (c) RFANALYSIS-SCOPE L66 externalizes these into `rfanalysis.toml` — "the whole point is the number can move," so a doc disagreeing on a *tunable default* is not a contract violation, just stale illustration; (d) even D-052's schema-wins principle, extended, says the machine-readable value governs. So: authoritative doc says 0.95, the "not yet built" doc says 0.8, hierarchy resolves it, downgrade to minor/cosmetic.

**why the kill fails.** The "it's just a tunable" defense is precisely the failure class the R0 discipline exists to prevent, and three facts defeat it:

1. **0.8 is not an arbitrary stale illustration — it is the *specific rejected value* from earnest-v3.** LEAD.4 nails this: the methodology (L118, L138) singles out the "≥80% hours ⇒ infrastructure" rule as the earnest-v3 version that "would filter out hourly-beaconing trackers." So PARAMETER-TUNING doesn't hand the operator a harmless different number; it hands them *the exact discarded value whose rejection is the reason this filter was redesigned*, misfiled into the "Analysis/filter" layer. This is worse than drift — it is a resurrection of the known bug in the one document an operator opens *specifically to tune this knob*.

2. **The value changes the filter PATH, not just the sensitivity — and RF.4 proves it reproduces.** RF.4's failure-scenario is concrete and computable: at 0.95, `infra_hours = ceil(24×0.95)=23`; at 0.8, `=20`. A 90-minute-cadence G1 interval-reporter touches ~16 distinct hour-buckets → below both thresholds, so it never reaches the low-duty *exception* path and is evaluated only against MIN_OBSERVATIONS. The *same device* takes a different branch under the two documented values. This is not a sensitivity knob you can A/B later — it is a routing decision, and the low-duty exception is D-028's NEVER-remove safety property. Getting the branch wrong for a sub-95%-presence tracker is exactly the miss-a-tracker outcome the system's stated bias (Q-05) ranks as the worst error.

3. **The A/B framing makes 0.8 self-defeating in a way the hierarchy argument misses (LEAD.4's sharpest point).** PARAMETER-TUNING *is* the A/B surface. If 0.8 lives there, the A/B framework varies presence_ratio against a *fixed event dataset* — but at 0.8 the event dataset would already have dropped the sub-daily trackers during Level-1 prep, so the A/B sweep can never recover them. The knob is varied downstream of the layer where it already did its damage. That is a structural defect in the tuning apparatus, not a stale number.

**QA.2 is adjacent, not duplicate — do not fold it in.** QA.2 is a *different* defect that shares the "most important knob" target: it flags that **no persona straddles the exception/infrastructure activity boundary** (present-~100% with activity at `timeframe×40 ± ε`), and it surfaces a *second* live inconsistency — the classifier uses `timeframe × ACTIVITY_PER_HOUR` (L110) while the presence threshold uses `effective_hours × presence_ratio` (L106), two different time bases. That time-base inconsistency is independent of the 0.95/0.8 drift and survives on its own. QA.2 should be tracked as its own missing-boundary-persona finding (add P2b), not merged into the presence_ratio reconciliation.

**cross-lens convergence.** RF (the physics of which device trips presence), Lead (cross-doc value + layer + rejected-value provenance), and QA (the boundary the personas fail to pin) converge. The RF lens *computes* the reproduction; the Lead lens *sources* the 0.8 to the rejected rule; the QA lens shows the fixture *cannot catch it* because no persona exercises sub-95% presence (P2 is at ~100%). Each lens supplies a piece the others lack — this is genuine convergence, and it hardens.

**verdict: hardened** (LEAD.4, RF.4). **survives** (QA.2, as a distinct adjacent finding).
The hierarchy/"it's tunable" rebuttal fails because 0.8 is the provably-rejected earnest-v3 value, it changes the filter *path* not just sensitivity (RF.4 computes the reproduction), and it sits on the A/B surface where it would sweep against an already-damaged dataset. Harden: the finding is worse than "two docs disagree" — it is *the discarded bug value, misfiled into the wrong layer, on the tuning surface.* Q-05's stated bias (miss-a-tracker is worst) makes 0.95 the correct default unambiguously.

**residual_claim:** Correct PARAMETER-TUNING L42 to `0.95` and layer `Prep (Level 1)`; add the one-line note "0.8 was earnest-v3's rejected presence-only value — do not use"; add the low-duty exception multiplier (40) as its own PARAMETER-TUNING row so the tracker-protection knob is visible on the tuning surface; make min-events explicitly a Level-2 analysis filter never a Level-1 handoff filter (RF.4); resolve the `timeframe` vs `effective_hours` time-base inconsistency (QA.2, same commit); add fixture persona for a sub-95%-presence tracker (~90-min cadence, ~16/24 hrs) that must survive, and P2b straddling the activity boundary (QA.2). Best home for the single authoritative table: machine-readable defaults in `contracts/schemas/` imported by both prose docs (per D-052/D-034), so prose can never drift again.

**downstream:** D-028, low-duty exception, HANDOFF "noise/infra already removed," PARAMETER-TUNING + A/B framework design, WP-S7, P2/P2b/P3/P15, Q-05, taxonomy G1/G5 discrimination, L1/L2 boundary (D-029). No cross-lens dissent — the three lenses reinforce.

## GROUP C — Pass-2 stub honesty
### Targets: FINDING-RF.7 (no-op vs population-fold stub bias), FINDING-QA.3 (whole-run on stub-binned, mislabeled-authoritative power_bin)

This is the group the cluster brief flags for adjudication: **RF.7's no-op stub vs RFANALYSIS-SCOPE L79's population-fold placeholder — which is safer given Q-05?** I take the two findings separately because they part ways on exactly this point.

### FINDING-QA.3 — the pipeline runs the whole build on stub-binned `power_bin` that the handoff mislabels "authoritative"

**rebuttal (kill attempt).** Seam discipline is exemplary and the DECISION-LOG plus fixture handle this about as well as an open algorithm can be: D-027 LOCKS the *structure* (load-bearing for processing time) and leaves only the *calc* OPEN; P5 is committed `xfail` from day one "so the stub can never be mistaken for done" (fixture L20/L47); RFANALYSIS-SCOPE L79 forbids the placeholder becoming canonical ("mark it clearly"). So the honesty is structural and the reviewer is extending, not refuting. One could argue: the xfail *is* the flag, no further mechanism needed.

**why the kill fails.** QA.3's precise point survives the seam-discipline defense because the honesty rides the **test file**, not the **data**. The `xfail` on P5 tells a *developer reading the test suite* that Pass-2 is stubbed. It does **not** travel with the handoff dataset. Meanwhile HANDOFF L30/L52/L70/L72 make four unconditional guarantees that `power_bin` is "Authoritative, post-boundary-reconciliation," "boundary reconciliation (Pass 2) has been applied," "Level 1 never emits an event that skipped … boundary reconciliation." During the entire build (S15 stub → S17 whole-run-green → S18 human-oracle sign-off on the 20595 corpus → S19 classification), the pipeline emits a dataset that *asserts in its own contract a property it does not have*. Guy's S18 "looks sane" blessing is applied to provisionally-binned data whose contract claims finality. RFANALYSIS-SCOPE L79's "mark it clearly" is satisfied for the *code seam* but not for the *data artifact* — nothing in the handoff row or `dataset_version` stamps `pass2=stub`. That gap is real, reproduces on every build run, and is not covered by the xfail mechanism.

Second surviving point: **P6 gives false merge-guard confidence under any stub.** P6 asserts "two distinct power_bins, never merged." A no-op OR a far-apart-only fold trivially passes it because the merge step never runs on far-apart bins — so P6 is green whether or not merge logic exists. QA.3's fix (assert the merge stage was *invoked and declined*, e.g. a boundary-candidates-considered counter) is a genuine mechanism-vs-output gap, exactly the QA lens's remit, and untouched by D-027.

**verdict: survives.**
The DECISION-LOG and fixture handle the *code-seam* honesty (rightly praised), but QA.3's claim is narrower and lands: the *data* is mislabeled authoritative for the whole build, the human-oracle sign-off is applied to it, and P6's merge-guard is unfalsifiable under a stub. None of these three is answered by D-027 or the xfail.

**residual_claim (survives as filed):** (1) Stamp `pass2=stub` on the handoff `dataset_version`/run-flag whenever P5 is xfail, so the honesty rides the data; downstream consumers and Guy's S18 review cannot mistake stub-binned for reconciled. (2) Mark HANDOFF's "power_bin authoritative" guarantee as *provisional-while-P5-xfail*, mirroring the correlation-provisional discipline already used for D-055 item (4). (3) Make P6 assert the merge stage was invoked-and-declined, not merely "2 bins." (4) Record S18's human-oracle sign-off as PROVISIONAL until P5 flips. (5) Pre-author the exact −40.1±0.4 acceptance derivation that flips P5 so "done" is machine-checkable.

**downstream:** D-027, Q-03/Q-05, HANDOFF power_bin guarantee (L30/L52/L70/L72), P1/P6/P15, S15/S17/S18 gates, the five human-oracle points, PARAMETER-TUNING Pass-2 row.

### FINDING-RF.7 — stub *bias*: no-op (accept false-split) vs population-fold (can false-merge); + Q-05 undecided

**rebuttal (kill attempt — and this one partly lands).** RF.7 makes a specific prescriptive claim: the *safe* stub is **identity / no reconciliation** because that "accepts false-splits, which are the known-tolerable error per the working assumption." **This rationale is factually inverted, and the inversion is checkable against the primary sources.** Q-05 states the operator's working assumption verbatim: *"confirm my working assumption that a false split (missed tracker) is worse than a false merge, so Pass-2 leans toward **merging** boundary cases."* The methodology L147–148 restates it: *"False split … missed trackers (**the primary failure — under-detection**) … my assumption is false-split/under-detection is worse for a tracker-hunting system, arguing for a Pass-2 that leans toward **merging**."*

So the working assumption is the **opposite** of what RF.7 asserts: false-split is the **worse** error, not "the known-tolerable error." A no-op stub (Pass-1 only, never merges) errs *entirely* toward false-splits — i.e. toward the error the operator ranks as **worst** (missed trackers). RF.7's own recommended stub therefore biases the system in the *most dangerous* direction under the documented assumption, and RF.7 justifies it with a sentence that gets the assumption backwards. The cluster brief inherits the same inversion ("the tolerable false-split").

**adjudication: RF.7 (no-op) vs RFANALYSIS-SCOPE L79 (population-fold) — which is safer given Q-05?**
- **Under the operator's stated assumption (false-split worse → lean merge):** the population-fold, despite being a crude whole-bin remap that *can* false-merge near-but-distinct devices, at least errs in the operator's *preferred* direction (merging). A pure no-op errs in the *dispreferred* direction (splitting). So on the *bias* axis alone, L79's placeholder is *more* aligned with Q-05 than RF.7's no-op — the reverse of RF.7's conclusion.
- **BUT** RF.7's *concrete hazard* is still real and independent of the bias axis: the population-fold triggers on **bin population, not proximity to a boundary** (methodology L87), so it can fold two genuinely distinct devices at −20/−22 with unequal counts into one bin — a false-merge that is *not* a controlled "lean toward merge at a boundary" but an *uncontrolled* merge driven by count asymmetry anywhere. That is a different, worse thing than "lean toward merging *boundary* cases." Q-05's "lean toward merging" means *at the boundary*; the population-fold merges *by population regardless of boundary*. So neither stub is clean: no-op violates the bias direction; population-fold honors the direction but by a mechanism that over-merges uncontrollably.

**verdict: softened.**
RF.7 **survives in its structural half** (the stub bias is undefined, Q-05 is undecided, and the population-fold's uncontrolled false-merge hazard is real and un-guarded — P5 is xfail, P6 only guards far-apart bins, so nothing catches near-but-distinct fusion). But RF.7's **specific prescription — "the safe stub is a no-op because false-split is tolerable" — is refuted**: it inverts the documented working assumption (Q-05, methodology L148), under which false-split is the *worse* error and a no-op stub biases toward it. The reduced, corrected claim is below.

**residual_claim (softened / corrected):** The live defect is that **the stub's bias is unspecified and Q-05 is unresolved**, and that *whichever* stub ships, an uncontrolled-merge hazard (population-fold) or an against-the-bias split hazard (no-op) goes unguarded. The correct pre-code action is: **(1) Resolve Q-05 first — it is the spec for the stub's conservatism and blocks the stub choice more than the docs admit.** (2) *If* Q-05 confirms false-split-worse (the working assumption), the stub should lean toward merging, but via a **boundary-scoped** rule (merge only observations within X dBm of a bin edge), NOT the population-fold's boundary-agnostic whole-bin remap — this honors the bias direction without the uncontrolled-fusion hazard RF.7 correctly identified. (3) Add a **green** stub-guard persona for near-but-distinct devices (−20/−22, unequal counts): under a population-fold they would wrongly fuse; a boundary-scoped merge or no-op would keep them separate — this catches the exact hazard P5's xfail and P6's far-apart bins both miss. (4) Restrict any stub merge to **one-hop** (no transitive chaining across >1 boundary). **Do NOT** adopt RF.7's framing that "no-op = safe because splits are tolerable" — that is contradicted by the operator's own working assumption; the stub-bias question must be answered *from* Q-05, not pre-empted by an inverted reading of it.

**downstream:** D-027, Q-03, **Q-05 (now shown to be blocking, not merely informative)**, methodology L87/L147-148, RFANALYSIS-SCOPE L79, P5/P6 + new stub-guard persona, HANDOFF power_bin (coordinate with QA.3's provisional-marking fix — same seam), PARAMETER-TUNING corpus seeding (RF.6).

**author-flag (cross-finding, for adjudication):** RF.7's residual and the cluster-brief framing both assert "false-split is the tolerable error." Q-05 and methodology L148 assert the opposite. The author must reconcile: either (a) Q-05's working assumption stands (false-split worse → stub leans merge, boundary-scoped), and RF.7's no-op recommendation is dropped; or (b) the operator, on review, *inverts* Q-05 (decides false-merge is actually worse — plausible for a periodicity-driven system where a fused stream corrupts CV), in which case a no-op/split-leaning stub becomes correct and RF.7's prescription is vindicated *but only after Q-05 is answered that way*. This is a genuine open decision that RF.7 prematurely resolved in the direction the current docs contradict. Flagging as the one place in this cluster where a reviewer's prescription conflicts with logged operator rationale.

## GROUP D — Stage-order contradiction across four contracts
### Target: FINDING-RF.5

**rebuttal (kill attempt).** The steelman RF.5 itself raises is strong: all four docs agree on the *set* of stages and on the two operationally load-bearing invariants — filtered devices don't flow forward (`INNER JOIN … filter='pass'`) and events are built only from passing FORWARD signal. The methodology *prose* (L100) states the intent unambiguously: noise filter sits "After power-bin grouping, before event construction." One could argue the numbered-list ordering is presentational scaffolding and a build agent follows the prose + the invariants, which are consistent. Also: is this reproducible, or does every sensible implementer just filter-before-building because that is obviously cheaper (L100's own rationale: "so noise is removed before event-building effort is spent")?

**why the kill fails.** Two facts defeat the "presentational only" defense:

1. **The methodology contradicts ITSELF on the page** — verified directly. The numbered "Event construction … in order" list (L19–24) is: power-bin (1) → sweep-collapse (2) → **burst-group = THE EVENT (3)** → **noise-filter (4)**. That puts noise-filtering *after* event construction. The prose at L100 says noise-filter sits *before* event construction. Same document, two orderings, one of which every other doc (R7 L196, HANDOFF §8, RFANALYSIS-SCOPE §54) contradicts. A build agent handed "the methodology is authoritative for this algorithm" (D-028 points at it) reads the numbered recipe as the spec — and the numbered recipe is the outlier.

2. **The order is not cosmetic — it changes the noise filter's *input domain* and therefore its results.** The noise filter is computed on `hours_present × total_activity` where `total_activity = SUM(observation_count)` over the *aggregate*. If sweep-collapse and burst-group run *first* (numbered list), that sum is over *collapsed detections*; if noise-filter runs first (prose/R7/HANDOFF), it is over *raw enriched observations*. RF.5's P15 example reproduces concretely: a deep-sleep tracker with 12 raw sweep-hits collapsing to 4 detections can pass a raw-count MIN_OBSERVATIONS but fail a detection-count one. So the P15 "survives noise filter" assertion — a *primary-target* guarantee — is order-dependent, and the fixture would encode whichever the author computed by hand. That is the silent-contract-via-a-number failure the whole R0 effort targets, and it is order-driven, not value-driven.

**does it reproduce?** Yes. Unlike a purely stylistic reordering, this one silently redefines what `SUM(observation_count)` ranges over, which the docs *never state explicitly* — RF.5's sharpest observation is the absence of any statement of "what the noise filter's activity count is computed over." Two competent implementers reading the two orderings compute two different filter inputs and two different pass/fail sets for exactly the boundary devices (deep-sleepers) the system most cares about.

**DECISION-LOG check.** D-026 records "sweep-collapse ≠ burst-grouping (two scales)" but is silent on where noise-filter sits relative to them. So the log does *not* resolve this — no `rationale_addressed` escape.

**verdict: hardened.**
The "presentational" rebuttal fails: the methodology self-contradicts (numbered list vs its own prose), the order changes the noise filter's input domain (raw vs collapsed observation counts), and P15's primary-target survival guarantee is order-dependent. The cross-exam *adds* the reason it is worse than filed: the docs nowhere state what `total_activity` is summed over, so even fixing the *order* is insufficient — the fix must also pin the *input domain* of the noise metric, or the same ambiguity re-enters through a different door. RF.5 correctly identifies this but it deserves emphasis as a hardening: this is two coupled defects (order + unstated count basis), not one.

**residual_claim (hardened):** Fix the canonical order in ONE place — R7 (the pipeline-stage contract) is the natural home — as **power-bin → noise-filter (on raw observation counts) → sweep-collapse → burst-group**, matching R7/HANDOFF/RFANALYSIS-SCOPE and the methodology *prose*; correct the methodology's numbered list (L19–24), the outlier. **Additionally and non-optionally:** state explicitly that the noise filter's `total_activity` is `SUM(observation_count)` over **raw enriched observations** (R2-weighted), computed *before* collapse, and add that assertion to P15's derivation so the survival guarantee is order-independent. Make the other three docs reference R7 rather than restating the order. Pure-doc edit now; post-build it is a pipeline rewrite + re-derivation of every fixture expected-output.

**downstream:** R7, methodology L19-24 (numbered list) + L100 (prose), HANDOFF §8, RFANALYSIS-SCOPE §54, D-026, P15 derivation, MIN_OBSERVATIONS definition (its input basis), the noise-filter component (WP-S6/S7). No cross-lens conflict — RF.5 is single-lens but multi-doc.
