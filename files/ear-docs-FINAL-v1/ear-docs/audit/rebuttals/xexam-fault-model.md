# Adversarial Cross-Examination · Cluster: fault-model & disposability

Cross-examiner mandate: refute, not agree. Findings DE.6, REL.6, REL.5.
Anchors: D-023 (per-agent disposable L1), D-044 (ephemeral run-to-completion, halt-on-failure = exit code, run lock), D-045 (ephemeral uploader oneshot, `pending/` = retry queue, DLQ = poison only), RECOVERY-RETENTION stale-lock rule.

The pivotal question the whole cluster turns on, and which I test each finding against:

> **Does the D-044 rationale — "run-to-completion gives clean memory, pinned image per run, fewer moving parts for a single operator" — actually REQUIRE whole-run halt, or only run-to-completion?**

Read literally, D-044's rationale (DECISION-LOG D-044) justifies three things: (a) *ephemeral* execution (fresh container per run → clean memory), (b) a *pinned image* per run, (c) *fewer moving parts* (no resident Celery/Redis, pool + function-order completion-gating). None of (a)/(b)/(c) is a statement about **failure granularity**. "Halt-on-failure maps to exit codes" is the *alerting* mechanism, not a claim that one agent's failure must abort the other 59. So the rationale that the cluster is asked to refute the findings *with* does not, on inspection, defend the whole-run-halt behavior the findings attack. This materially weakens the rebuttals available to me — and I say so honestly below rather than manufacture a defense the log does not supply.

---

## target: DE.6 — halt-on-failure undoes per-agent isolation (one corrupt disposable file fails the whole 60-agent run)

**rebuttal (strongest available).**

Three distinct lines of attack, taken in order of force.

1. **The rationale-addressed attack (does D-044 already answer this?).** D-044 says "Non-zero exit = alert; next scheduled run retries … L1 DuckDB files disposable." One could argue the design *intends* whole-run halt as a deliberate fail-loud posture: for a *first-facility* deployment with a handful of sensors, a single operator, and a periodic batch job, aborting the whole run on any error and paging is the minimum-moving-parts choice, and "next run retries" self-heals the *transient* case (a torn read, a transient R2 hiccup, a one-off OOM). This is the steelman the reviewer themselves granted. **But this attack fails on the finding's own decisive sub-case:** the *deterministic-poison* file. The reviewer explicitly carved it out — "if that one file corrupts *deterministically* … **every** retry fails at the same agent, and the fleet never produces a handoff until a human intervenes." D-044's "next run retries" is exactly the clause that does *not* cover this, and RECOVERY-RETENTION's "Just re-run" (line 8) is silent on it. So the rationale does not address the load-bearing half of the finding.

2. **The disposability-cuts-the-other-way attack (SRE's prior).** The SRE lens protects per-agent disposability (D-023) as "the single best operational decision" — recovery = delete + rebuild, never a 3am surgical repair. One could argue DE.6's proposed fix (per-agent quarantine, per-agent status on AnalysisRun, K-consecutive-failure auto-disable) *adds moving parts* and thus violates the D-044 "fewer moving parts" ethos the SRE lens also prizes — i.e. DE.6 is in tension with "protect disposability / keep it simple." **This attack fails too, and the lead already adjudicated it (lead-seams SEAM-4):** disposability and per-agent fault-isolation are *the same property*, not opposed. D-023 already *sells* the containment ("a wedged/corrupt file affects one agent and is disposable"); DE.6 merely asks the execution model to *honor* the containment D-023 advertises. Making the pool continue past one agent's failure is not new machinery — it is a try/except boundary around the per-agent chain that already exists as a unit of work (D-025: "per-agent chains … correctness boundary"). The SRE "fewer moving parts" prior does not defend whole-run halt; it defends *not adding a resident supervisor / Celery back*, which DE.6 does not propose.

3. **The reproduces-or-speculative attack.** Does the whole-fleet-halt-from-one-file scenario actually reproduce? Yes, and trivially, under the design as written: `compose run --rm … run_level1` exits non-zero if any pooled worker raises; a non-zero process exit *is* the run's failure (D-044); the 59 healthy agents' handoff exports either never run (if the failing agent is early and the pool short-circuits) or are discarded as a failed run. The reviewer's DE.1 (quarantine tripping the post-load assertion) supplies a *concrete, every-run* trigger for the raise, and DE.4 (OOM) supplies a second. So the trigger is not hypothetical — the finding names two independently-filed mechanisms that produce the non-zero exit. This is the opposite of speculative.

**Where the rebuttal partially lands (the one concession the finding must absorb):** the finding slightly overstates *guaranteed* fleet-wide blast radius. Whether the 59 healthy agents lose their handoff depends on an unstated implementation detail: if the pool collects per-agent results and only the *job's* final exit code is non-zero (agents having already written their handoff partitions atomically, D-024), then the healthy agents' *data* survives and only the *run status* is red. The finding assumes the worse variant ("no handoff is produced for the 59 healthy agents"). Since the export step is per-agent and atomic (D-024), the more likely real behavior is: healthy agents' handoff partitions are on disk, but the run is *marked failed* and — critically — the L2/dashboard consumers may treat a failed run as "produce nothing new" (per REL.12's partial-fetch-looks-complete concern and RUNBOOK's clean-fail assumption). So the residual harm is not necessarily "59 agents lose data" but "59 agents' fresh data exists yet the run is red and may be treated as no-progress, and the poison agent blocks the run status indefinitely." That is still a real defect — the fail-loud posture cannot distinguish "one disposable file is poison" from "the fleet is down," and the deterministic-poison case never clears without a human. The finding survives; only its worst-case data-loss phrasing softens to a status/visibility harm plus the unbounded poison stall.

**verdict.** survives

**residual_claim.** Whole-run halt-on-failure (D-044) contradicts the per-agent fault-containment D-023 advertises; the load-bearing, non-speculative harm is the **deterministic-poison stall** — a single agent whose chain raises identically every run (DE.1 quarantine-assertion or DE.4 OOM as concrete triggers) makes the run red on every retry, so the fleet never reaches a clean run and "next run retries" never self-heals until a human intervenes. The fix (per-agent quarantine + per-agent status on AnalysisRun + K-consecutive-failure auto-disable, whole-run halt reserved for genuinely-shared-resource failures) does **not** violate D-044's rationale, which justifies run-to-completion and clean memory but says nothing requiring whole-run halt. Downgrade only the "59 agents lose their data" phrasing to "59 agents' atomic handoff may survive on disk but the run is red / treated as no-progress and the poison agent blocks run status unboundedly."

**downstream.** Confirms the lead's Phase-2 question in the operator's favor: D-044's rationale does **not** require whole-run halt. Couples to DE.1 (supplies the every-run trigger) and DE.4 (supplies the OOM trigger). Needs AnalysisRun per-agent status field; D-023/D-044 fault-model wording reconciled once. HEALTH-CHECKS run-success check must distinguish "partially succeeded" from "failed."

## target: REL.6 — wall-clock stale-lock steals the lock from a live long run → thundering-herd self-stealing rebuild loop

**rebuttal (strongest available).**

1. **The "hourly is a strawman / Q-19 provisional forbids it" attack.** REL.6's loop requires the timer period to be *shorter than a legitimate run's duration*. The design's provisional cadence is "2–4×/day" (Q-19), i.e. a 6–12h period. A run that legitimately exceeds 6h at first-facility scale (a handful of sensors, D-023 embarrassingly-parallel, pool 4×4GB on a 128GB/2TB-NVMe box) is implausible. So one could argue: at the *actual* provisional cadence, the stale-lock threshold ("older than the timer period", RECOVERY-RETENTION line 12) comfortably exceeds any real run time, and the loop cannot arm. **This attack fails on two counts.** First, the reviewer pre-empted it: the trigger is not steady-state but a *post-outage backlog* (5 days × 60 sensors) colliding with an operator who *tightened* cadence chasing dashboard freshness — and Q-19 is OPEN, so "hourly is on the table" is a fact about the contract, not a strawman. Second, and more decisively, DE.5 (full-rebuild-over-window as windows grow) and REL.2 (undefined window) independently show rebuild wall-clock is *unbounded and grows with deployment age and backlog*, while Q-22 (the load gate that would measure max run time) is **deferred**. So nobody has measured whether a run *can* exceed the period — the design sets a stale-lock threshold against a run duration it has explicitly declined to measure. The wall-clock criterion is unsafe *by construction* the moment either cadence is tightened or a backlog lands, and neither is forbidden.

2. **The "does D-023 disposability rebut REL.6 or enable it?" attack (the assigned pivot).** The tempting rebuttal: D-023's disposability is a *strength* — a stolen lock corrupts one DuckDB file, the file is cheerfully deleted and rebuilt, no harm. **This is exactly backwards, and the reviewer nailed it: disposability is the *enabler*, not the mitigant.** Because a corrupted file is *cheerfully* deleted and rebuilt, the second (lock-stealing) run does not fail hard and stop — it re-enters delete-and-rebuild over the *same huge backlog*, *also* outlives the period, and *also* gets its lock stolen by the next tick. The property that makes single-file recovery safe (delete + rebuild is free) is precisely what removes the natural circuit-breaker (a hard error that would halt the loop). Lead-seams SEAM-4 states this explicitly and I concur under adversarial scrutiny: disposability *enables* REL.6. The pivot resolves against the operator.

3. **The "systemd already prevents overlap" attack.** One could argue D-044 already says "overlap prevented by a run lock," so overlapping runs cannot happen. **This fails because the lock's *break* rule is the bug**, not the lock's existence. The lock correctly prevents overlap *until* the auto-break fires; the auto-break criterion ("older than the timer period") is a *liveness proxy that is wrong when a live run legitimately outlives the period*. The mechanism meant to recover from a *dead* run (host crash mid-run) cannot tell a dead run from a slow-but-healthy one, because it uses wall-clock age instead of process liveness. So the lock does not save the design here; its recovery clause is the defect.

**Can I find *any* softening?** The one honest reduction: the loop is only *armed*, not *guaranteed*. It requires (a) run duration > stale-lock threshold AND (b) the next tick to actually fire and steal rather than the systemd unit refusing to start a second instance. In real systemd, a `Type=oneshot` service triggered by a timer will *not* start a second instance while the first is still active (systemd serializes a given unit) — so if the pipeline is one systemd service, the *next timer tick that finds the service still Active is itself a no-op* and no second `run_level1` starts. The lock-steal-and-race requires the *application-level* lock to be the only guard AND the job to be re-invoked (e.g. cron, or `compose run` fired by a fresh timer instance not bound to the first's unit state). Whether REL.6 reproduces therefore depends on an unspecified deployment detail (systemd unit binding vs cron vs `compose run` per tick). The reviewer's own proposed fix even names this (`Type=oneshot` + `RefuseManualStart` + flock so an overrun *blocks* the next tick). So the *catastrophic self-stealing loop* is contingent on the wall-clock break firing against a live run, which a correct systemd unit binding would prevent independently of the lock. This narrows the finding from "will loop forever" to "the *documented* stale-lock rule (wall-clock age) is the wrong criterion and will steal a live run's lock under the exact conditions D-044/Q-19/Q-22 leave open — unless an unspecified systemd binding happens to save it."

**verdict.** survives

**residual_claim.** The stale-lock staleness criterion in RECOVERY-RETENTION line 12 ("older than the timer period") is wrong: it cannot distinguish a crashed run from a healthy run that legitimately outlives the period, and the design has *deferred (Q-22) the very measurement* that would tell you whether that can happen, while leaving cadence (Q-19) open including hourly. Under a post-outage backlog (REL.2/DE.5 make rebuild wall-clock unbounded) the auto-break steals a live run's lock; D-023 disposability *enables* rather than mitigates the resulting loop because corrupt files are cheerfully rebuilt, removing the natural halt. **Fix:** break on process liveness (pid + heartbeat-mtime the running job touches), not wall-clock age; and/or make the timer/unit refuse a second instance so an overrun *skips* the next slot rather than racing it; and set the stale-lock threshold ≥ measured P95 run time. The one honest narrowing: whether the *full self-stealing loop* materializes depends on an unspecified systemd-unit-binding-vs-cron detail — but the *documented rule* is defective regardless of that detail, so the contract must change.

**downstream.** Directly hardens the panel's dissent (lead-seams Section F item 3 / this cluster's Q-22 question): you **cannot** safely choose Q-19 cadence or the stale-lock threshold without measuring max run time, so a *narrow* single-agent worst-case run-time probe must precede first facility even if the full Q-22 fleet gate stays deferred. Couples to DE.4 (RSS/OOM extends run time), DE.5/REL.2 (unbounded window extends run time), D-023 (disposability = enabler). RUNBOOK needs a "run outran its timer" response distinct from "host crashed mid-run."

## target: REL.5 — uploader poison-file starvation + missing uploader run-lock + halt-on-failure asymmetry with D-044

**rebuttal (strongest available).** This finding is three sub-claims; they do not all survive equally, so I adjudicate each.

**(5a) Poison-file starves the whole batch (halt-on-first-error on the uploader).**

1. *The rationale-addressed attack.* D-045 says "DLQ narrows to genuinely poison files only" and "`pending/` IS the retry queue by construction (failed run exits leaving files)." One could read this as *already* specifying continue-past-poison: poison → DLQ, everything else stays in `pending/` and drains next tick. **This attack partly lands but does not kill the finding.** D-045 *names* the DLQ-for-poison outcome but never states the *intra-run isolation* invariant — "one poison file is routed to DLQ and the run **continues** uploading the rest of the batch." The reviewer's decisive observation: D-044's *sibling* rule for the pipeline is explicitly "halt-on-failure = non-zero exit," and the two jobs "look symmetric." An implementer copying D-044's halt-on-failure onto the uploader loop (a natural, even likely, reading given the symmetric ephemeral-oneshot framing) gets exactly the starvation bug: file on device `a` throws → oneshot exits non-zero → `b/c/d`'s good files never upload → all four `pending/` grow → disk ceiling eventually evicts *good* data while the poison file sits there. So the gap is real: the *asymmetry* between D-044 (halt-on-failure, correct for the pipeline) and D-045 (must be best-effort-per-file, the *opposite*) is undocumented, and the docs' symmetric framing ("Symmetric with D-044", D-045) actively invites the wrong copy.

2. *Reproduces-or-speculative.* This reproduces only if the implementer chooses halt-on-first-error. But the finding is a *pre-code design audit*: the contract does not forbid the wrong choice and the sibling decision models it. For a spec whose entire ethos is "make contract violations impossible, not merely forbidden" (R0), leaving the isolation invariant unstated *is* the defect. Survives as a spec-completeness finding.

**(5b) Missing uploader run-lock → overlapping oneshots race `pending→processing→completed`.**

1. *The "D-044 lock covers it" attack.* D-044's run lock is explicitly scoped to `run_level1` (the pipeline). D-045 says nothing about an uploader lock. **The attack fails: the lock is the wrong lock.** However —
2. *The "systemd serializes a oneshot" attack (genuinely strong here).* A systemd `.timer` firing a `Type=oneshot` service will **not** start a second instance while the first is still Active — systemd serializes instances of the same unit by default. D-045 explicitly describes `uplink-upload.timer → oneshot`. So under the *documented* systemd model, the "two overlapping upload oneshots on the same `pending/`" scenario **does not reproduce**: the next timer tick that finds the oneshot still running is a no-op, which is *exactly the behavior the reviewer asks for* as the fix ("overlapping-tick attempt is a no-op"). The reviewer's own evidence hedges ("systemd timers with `OnUnitActiveSec` *can* overlap a long-running oneshot **unless** the unit is `Type=oneshot`") — but D-045 *does* specify `Type=oneshot`. So the missing-run-lock sub-claim is largely **already handled by the stated systemd model**; the residual is only that the *contract does not state the invariant* ("overlapping tick is a no-op") and does not cover the non-systemd path (a cron-driven or `compose run`-per-tick uploader, which the docs do not use for the agent). This sub-claim softens hard: it is a documentation nicety, not a live race, given `Type=oneshot`.

**(5c) Halt-on-failure asymmetry with D-044 must be documented.**

This is the strongest and most durable part of the finding, and it is *not* rebuttable — it is precisely the answer to this cluster's assigned pivot. D-044 halt-on-failure is *correct* for the pipeline (an integrity-critical batch where a partial run is a lie). D-045's uploader must be the *opposite* (best-effort-per-file; one poison file must never block delivery of good field data, because un-uploaded `pending/` is the *only true data-loss window* per BACKUP-DR). The two decisions are framed as "symmetric" (D-045) yet require *opposite* failure semantics. Documenting this asymmetry is free and prevents the 5a starvation bug at its root.

**Cross-domain adjudication (DE.6 vs REL.5 — do they conflict?).** They *appear* to pull opposite ways: DE.6 attacks the pipeline's whole-run halt (wants per-agent isolation), REL.5c defends the pipeline's halt-on-failure as correct while attacking its *copy onto the uploader*. This is **not a conflict — it is the same principle applied at two granularities.** DE.6 says: within the pipeline, halt should be per-*agent*, not whole-run. REL.5 says: the uploader should be per-*file*, not whole-batch. Both findings converge on one design rule: **failure isolation should be at the unit-of-independent-work (agent for the pipeline, file for the uploader), and only genuinely-shared-resource failures should halt the whole job.** They reinforce rather than contradict. The one subtlety the author must hold: the pipeline's *integrity* argument (a partial event dataset is a lie → REL.12) means a pipeline agent-failure should still *page* and mark the run degraded even as the 59 continue — whereas the uploader's *availability* argument means a poison file should page *quietly* (DLQ + one alert) and never degrade the delivery of good files. Same isolation principle, different halt-vs-continue-vs-degrade tuning because one job guards *integrity* and the other guards *delivery*.

**verdict.** softened

**residual_claim.** The durable core is **5a + 5c**: the uploader must be specified as **best-effort-per-file** (poison → DLQ + one alert, batch continues), and the **halt-on-failure asymmetry between D-044 (pipeline: halt/degrade to protect integrity) and D-045 (uploader: continue to protect delivery) must be documented explicitly** because the docs' "symmetric with D-044" framing invites the wrong copy and the resulting starvation lets the disk ceiling evict *good* field data around a stuck poison file. **5b (missing run-lock / overlapping oneshots) softens to a documentation-only item**: D-045's stated `Type=oneshot` systemd model already serializes instances, so the overlapping-drain race largely does not reproduce as written; the contract need only *state* "an overlapping tick is a no-op; the queue persists" and note it does not hold for a non-oneshot/cron path. Net severity drops from major toward major-minus (5a/5c hold; 5b is minor/doc).

**downstream.** Reinforces DE.6 (same isolation-at-unit-of-work principle, applied to files vs agents; no conflict — mutually confirming). Couples to REL.7 (disk-ceiling eviction policy — the starvation makes the eviction path fire on good data, so the eviction-policy decision matters more). P17 conservation persona should assert a poison file never silently vanishes from the count. Q-09 (DLQ retention) extended with the isolate-and-continue requirement. The asymmetry note is the direct, affirmative answer to the cluster's pivot: D-044's run-to-completion rationale does **not** require whole-job halt for *either* job — the pipeline needs per-agent isolation, the uploader needs per-file isolation, and whole-job halt is reserved for shared-resource failure.

## Cluster synthesis (for the author)

- The assigned pivot resolves cleanly and against the current wording: **D-044's rationale ("clean memory, pinned image, fewer moving parts") justifies run-to-completion and ephemeral execution, but contains no premise that requires whole-*job* halt.** Whole-job halt is an independent, undocumented choice. Both DE.6 (pipeline) and REL.5 (uploader) attack that choice at their respective granularities and both are right to; SRE's protect-disposability stance is **complementary, not in conflict** (lead-seams SEAM-4 already adjudicated: disposability and per-agent isolation are the same property).
- **D-023 disposability *enables* REL.6's loop** (cheerful rebuild removes the natural halt) — this is the one place where the design's best operational decision is also a failure amplifier, and it must be held as both at once.
- **DE.6 vs REL.5 is not a dissent** — it is one isolation principle (isolate at the unit of independent work) applied twice, with the halt/continue/degrade tuning differing because the pipeline guards *integrity* and the uploader guards *delivery*.
