### FINDING-QA.1 · severity: **blocker** · target: Persona P7 ("Cadence trap") + GOLDEN-FIXTURE-SPEC §Personas

**steelman.** The whole point of D-012 (feed split) is to make the historic false-CRITICAL bug *structurally impossible*, not merely forbidden — LOCAL-HOST-DATA-FLOW L4 line 160: "It is thereby **structurally impossible** to construct an event from a summary row." P7 is the guardian of that structural claim: AGGREGATE/FILTER rows arriving at 5-min cadence with observation_count=350 in the **context feed** must yield zero events. The design's logic is sound: if the event path reads only `obs_` files and P7's rows are all in `ctx_` files, zero events is the *correct* and *inevitable* output. The steelman is strong — this is the single most-scarred bug in the system's history (R2 NEVER, line 75) and the persona exists precisely to regression-guard it.

**failure_scenario.** P7's expected output is "**Zero events**." But *zero events is the default state of an empty signal feed*. Concretely: a future refactor (say S16, WP-S7/S8, event construction) accidentally makes the event stage read from the context table instead of the signal feed — OR, more insidiously, the fixture generator for P7 simply writes P7's rows only to `ctx_` files and the signal feed for P7's EARFCN is genuinely empty. In BOTH cases P7 is green. In case one the contract is *violated* (event path now reads ctx) but P7 can still pass if the specific ctx rows it reads don't happen to sessionize into a burst under the 600 s rule — and 350-count AGGREGATE rows at exact 5-min cadence, if mis-ingested, are EXACTLY the shape that DID produce false CRITICALs historically. The persona asserts an absence, and an absence is unfalsifiable as a mechanism check: you cannot tell "zero because the split works" from "zero because there was nothing to find." A green P7 does not prove the feed split routes correctly; it proves only that no event came out.

**evidence.** GOLDEN-FIXTURE-SPEC line 22 (P7): "**Zero events** — structurally impossible (event path reads signal feed only)." The expected output is a null. There is no positive assertion that (a) the ctx rows *landed in the context table* with correct observation_count, or (b) that a **negative-control twin** exists — the same 350-count rows placed in the *signal* feed to prove the event path WOULD build events from them if it read them, so that P7's zero is meaningful by contrast. The R2 ENFORCED-BY (DATA-WORKFLOW-RULES line 78) says "A regression fixture (AGGREGATE-heavy file) asserts zero false CRITICALs" — same null-assertion weakness. P7's contract-proven column does list "presence metrics weighted by observation_count" and "ctx rows land in the context table," but the *expected output* row only commits to "Zero events."

**proposed_alternative.** Make P7 a two-armed differential test: (arm A) 350-count AGGREGATE rows in `ctx_` → assert zero events AND assert the ctx rows are present in the context table with observation_count=350 AND presence metric reflects that weight; (arm B — the negative control) the *same rows deliberately mis-routed into the signal feed* → assert the event path WOULD build a (false-CRITICAL-shaped) event, proving the event stage is not simply inert. If arm B produces zero events, the event stage is broken/empty and P7's green is meaningless. Then assert the routing layer sends P7's real rows to arm A not arm B. This converts P7 from "asserts an absence" to "asserts the split *causes* the absence." Additionally, add a positive assertion that observation_count survives into the presence weighting (a value check, not a count-of-events check).

**downstream_impact.** R2 ENFORCED-BY row; the R2 regression-fixture clause; S16 gate (P1–P3/P8 green, P2 protection); the conservation whole-run assertion (which also depends on ctx obs_count being correctly captured); D-012 structural claim.

**confidence.** high

### FINDING-QA.2 · severity: **major** · target: Persona P4, P3 vs P8 vs P2 — output-pinned filter_decision without mechanism separation

**steelman.** Each of the four filter personas pins a distinct `filter_decision` value (P2→pass/PROTECTED, P3→infrastructure, P8→noise) and P2 additionally pins the exact reason string 'PROTECTED: Low duty cycle'. The design intent (EVENT-CONSTRUCTION-METHODOLOGY lines 108–116) is a three-way classification with the low-duty exception as the critical branch; the personas map one-to-one onto the three branches plus the noise floor. That is a defensible test partition — one persona per decision path — and P2's reason-string pin is exactly the "green persona constrains what it claims" discipline the lens demands: it's not enough to be 'pass', it must be 'pass because PROTECTED', which pins the *branch taken*, not just the outcome.

**failure_scenario.** The classification has TWO conditions gating the exception (methodology line 110): "present in ≥ infra_hours **AND** total activity < timeframe × ACTIVITY_PER_HOUR → 'pass'". P2 is built as "present in ~100% of hours, total < timeframe×40" — it exercises the exception branch. But there is no persona that sits on the *boundary between the exception and the infrastructure verdict*: a device present in ~100% of hours whose activity is *just above* timeframe×40. If the implementation gets the comparison operator wrong (`<` vs `≤`, or compares against timeframe×40 vs effective_hours×40 — note the methodology uses `timeframe × ACTIVITY_PER_HOUR` in the classifier but `effective_hours` in the presence threshold, lines 106 vs 110, an inconsistency), P2 still passes (it's well below the threshold) and P3 still passes (it's well above), but every device landing in the narrow band between them is misclassified — and that band is exactly where a *marginal power-saving tracker* lives. The two personas green; the knob that "separates power-saving tracker from infrastructure" (methodology line 128, "the single most important knob") is unconstrained at its decision boundary.

**evidence.** EVENT-CONSTRUCTION-METHODOLOGY line 110 uses `timeframe × ACTIVITY_PER_HOUR`; line 106 defines the presence threshold with `effective_hours × presence_ratio`; the two use different time bases with no persona pinning which is correct at the margin. GOLDEN-FIXTURE P2 (line 17): "total < timeframe×40" — comfortably inside, not at the edge. No persona named for the ACTIVITY_PER_HOUR boundary. Contrast P5, which explicitly IS a boundary-straddler for power binning — the design KNOWS boundary personas matter but built one only for the power-bin edge, not the duty-cycle edge.

**proposed_alternative.** Add persona **P2b "Duty-cycle boundary straddler"**: a device present ~100% of hours with total activity at timeframe×40 ± epsilon, split so half the derivations land just under and just over. Assert the just-under case → pass/PROTECTED and the just-over → infrastructure, pinning the operator, the time base (timeframe vs effective_hours), and the comparison direction. This is the duty-cycle analogue of P5 and guards "the single most important knob" at the only place it can silently break. Also resolve the timeframe-vs-effective_hours inconsistency in the methodology in the same commit (R0.3).

**downstream_impact.** EVENT-CONSTRUCTION-METHODOLOGY classification block; PARAMETER-TUNING ACTIVITY_PER_HOUR row; taxonomy G1/G5 discrimination (the margin device is exactly the G1-vs-G5 confusion class, DEVICE-BEHAVIOR-TAXONOMY line 27); S16 gate.

**confidence.** high

### FINDING-QA.4 · severity: **blocker** · target: Whole-run "Determinism: byte-identical events Parquet" — physically unachievable as stated

**steelman.** Determinism is the foundation of the whole test model: R7 (DATA-WORKFLOW-RULES line 202) guarantees "Two runs on identical data+config produce identical events," and P16 (late arrival) depends on "full-rebuild semantics" producing deterministic updates. Byte-identical Parquet is the strongest possible statement of determinism and a clean CI assertion — either the bytes match or they don't, no fuzzy tolerance to argue about. The intent is unimpeachable: a rebuild-per-run pipeline (D-044) with config snapshots is *designed* to be reproducible, and reproducibility is the precondition for A/B testing (PARAMETER-TUNING line 11).

**failure_scenario.** Parquet byte-identity across two independent runs is defeated by at least four mechanisms none of which the docs address: (1) **DuckDB does not guarantee row ordering** without an explicit total `ORDER BY`; a parallel process-pool pipeline (D-044) exporting per-agent chains will interleave/emit rows in nondeterministic order unless every export is fully sorted — and even then hash-aggregation and window functions (`SUM(is_new_burst) OVER (…)`, methodology line 34) can vary. (2) **Parquet file-level metadata**: the format embeds a writer version, and DuckDB/Arrow may embed creation timestamps or per-column statistics whose encoding varies; two byte-identical logical tables can produce different Parquet bytes. (3) **Compression**: Parquet blocks are typically zstd/snappy-compressed; compression is deterministic *only* if codec + level + row-group boundaries are pinned, which the docs never specify. (4) **`run_id`**: EVENT-DATASET-HANDOFF requires every row carry `run_id` (line 39) and file_metadata carries `run_id` (handoff-events.v1.columns.json line 5). If run_id is a UUID or timestamp per run — which is its natural form — then run 1 and run 2 have DIFFERENT run_id in every row, so the Parquet CANNOT be byte-identical by construction. The determinism assertion contradicts the provenance requirement. Concretely at S17: the determinism test fails on first honest implementation, and the likely "fix" is to weaken it (compare logical rows, or exclude run_id) — but that weakening is done under gate pressure, undocumented, and now determinism means something softer than the contract claims.

**evidence.** GOLDEN-FIXTURE line 37: "run the full chain twice on the fixture → **byte-identical** events Parquet." EVENT-DATASET-HANDOFF line 39 requires `run_id` per row; handoff-events.v1.columns.json embeds `run_id` in file_metadata AND as an events column. D-044 (DECISION-LOG line 52): "per-agent chains … parallelized with a process pool" — parallel export, ordering-nondeterministic. No doc pins: ORDER BY on export, Parquet codec/level, row-group size, or how run_id is held constant across a determinism re-run. RFANALYSIS-SCOPE mentions no export-ordering rule.

**proposed_alternative.** Restate the determinism assertion at the level it can actually hold: "**logical determinism** — two runs produce the identical *event set* (same rows, same values) after a canonical total sort on `(agent_id, earfcn, power_bin, event_start)`, comparing all columns EXCEPT `run_id`." If byte-identity is genuinely wanted (it's a nice CI property), then additionally pin: a fixed `run_id` in determinism-test mode (inject a constant), an explicit `ORDER BY` on every Parquet export, a pinned compression codec+level+row-group size in the export config, and a documented DuckDB/Arrow version (the byte layout is version-sensitive, so the assertion must be version-scoped). Add this to the handoff ENFORCED-BY list. Decide this NOW — it changes the export code's contract and the meaning of P16.

**downstream_impact.** R7 determinism ENFORCED-BY; P16 (deterministic update depends on this); EVENT-DATASET-HANDOFF ENFORCED-BY + run_id field; D-044 parallel export; S17 gate; PARAMETER-TUNING A/B reproducibility precondition; the atomic-partition-export rename (a byte-mismatched re-run still replaces the partition, but P16's "update deterministically" claim needs the logical form).

**confidence.** high

### FINDING-QA.5 · severity: **blocker** · target: P15 min-events + PARAMETER-TUNING "Min events for analysis | Analysis | 3" — the starvation risk is assigned to a LAYER that has no persona

**steelman.** P15 is the crown-jewel persona: the primary target's hardest case (deep-sleep 20595-class, 4 bursts across 10 days), explicitly built to guard "the min-obs/min-events starvation risk" (D-057), and one of only three personas Guy hand-derives (D-057, double-derivation). The design took the starvation risk seriously enough to create a dedicated end-to-end persona AND put it in the double-derivation set. The steelman: this is the most-scrutinized single expected-output in the entire fixture, and it directly encodes "if this is red, the system misses 20595-class devices" (GOLDEN-FIXTURE line 30).

**failure_scenario.** P15's contract-proven list is "Low-duty exception × min-obs × **min-events** interplay; taxonomy G1." But **"min events" is a Level 2 / analysis parameter, not a Level 1 one**: PARAMETER-TUNING line 44 classifies "Min events for analysis" as layer **Analysis**, default **3**, "Below this, no meaningful CV." Level 1's job (the handoff contract) is to *build events and hand them off*; it does NOT drop a device for having too few events — the handoff explicitly preserves low-duty devices. So the "min-events erasure" P15 claims to guard against does not live in Level 1 at all; it lives in Level 2's feature-extraction/classification (S19a/S19b). But P15's expected output ("events built (min-events rules must not erase it); classified G1") straddles both layers, and the fixture is a **Level 1 fixture** (GOLDEN-FIXTURE line 2, "makes every Level 1 contract executable"). Concretely: at S16 (Level 1 events), P15 green means "4 events built, survived noise filter" — correct and meaningful. But the "classified G1, not erased by min-events starvation" half cannot be tested until S19b (Level 2 classifier), and at S19b the classifier's own "Min events for analysis = 3" rule may legitimately refuse to compute CV on a 4-event / 3-gap stream (a deep-sleep device with 4 events has only 3 gaps — right at the min-events=3 floor). So P15 could be GREEN at Level 1 and the device could STILL be starved at Level 2, because the erasure risk was located in the wrong layer and the persona that "guards" it is a Level 1 persona. The system misses the 20595-class device and P15 was green the whole time.

**evidence.** PARAMETER-TUNING line 44: "Min events for analysis | **Analysis** | 3 | Below this, no meaningful CV." DEVICE-BEHAVIOR-TAXONOMY line 13: deep-sleep "tolerates very few events." GOLDEN-FIXTURE P15 line 30 lists "min-events rules must not erase it" as a proven contract — but min-events is not a Level 1 contract; it is not in DATA-WORKFLOW-RULES, EVENT-CONSTRUCTION-METHODOLOGY, or the handoff. BUILD-RUNSHEET: P15 is gated at S04 (fixture authoring) and its Level-1 half at S16; its G1-classification half only exists at S19b ("P1→G1, P2→G1" — note **P15→G1 is NOT even listed** in the S19b gate, only P1/P2/P3). So P15's taxonomy half has no gate assertion at all. A 4-event stream yields 3 gaps = exactly the min-events=3 floor; the "≥ N vs > N" operator on that floor is untested and unpersonaed.

**proposed_alternative.** (1) Split P15 into P15-L1 (Level 1: 4 events built, survives noise filter — gated at S16) and P15-L2 (Level 2: the 4-event/3-gap stream is classified G1 deep-sleep and NOT sent to unclassified-by-starvation — gated at S19b, and ADD "P15→G1" explicitly to the S19b gate line, which currently omits it). (2) Move the "min-events" language out of the Level 1 fixture's P15 contract-proven list (it's a Level 2 rule) and instead pin, in Level 2, the deep-sleep sub-path exception: a very-few-event stream must reach G1 deep-sleep *without* requiring min-events≥3 for a CV computation — i.e., the taxonomy's "deep-sleep sub-path for very-few-event streams" (DEVICE-BEHAVIOR-TAXONOMY decision sketch line 53) needs its own assertion that the min-events floor does not apply on that path. (3) Add P15 to the double-derivation set for BOTH halves. (4) Resolve whether "Min events for analysis = 3" and the deep-sleep exception conflict — this is the starvation risk, and right now it is un-personaed at the exact layer where it bites.

**downstream_impact.** D-057 (the persona this decision created); PARAMETER-TUNING min-events row; DEVICE-BEHAVIOR-TAXONOMY G1 deep-sleep sub-path; S19b gate line; the double-derivation set; the handoff "low-duty preserved" guarantee; the four-factor scoring (PERIODIC factor needs gaps that a 3-gap stream barely supplies).

**confidence.** high
