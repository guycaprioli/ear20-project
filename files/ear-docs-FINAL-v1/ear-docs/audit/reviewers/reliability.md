## 2. FINDINGS

### FINDING-REL.1 · severity: blocker · target: D-016 / D-045 / L-chain "Deployment shape" / heartbeat.v1.schema.json `agent_id`

**steelman.** D-016 deliberately preserves the proven v4.3 topology ("Matches proven v4.3 model; per-device isolation at folder level"), and D-045 folds the uploader into a single ephemeral timer oneshot to kill the whole file-watcher/circuit-breaker bug class ("no resident uploader to crash or restart"). Consolidating four uploaders into one timer job is a *reliability improvement*: one systemd unit, one cadence, one place backlog is visible. The heartbeat "rides the upload run" so it is written exactly once per tick where the queue/disk numbers are already in hand. All coherent.

**failure_scenario.** The heartbeat object key is `_health/{agent_id}.json` and `agent_id` is regex-pinned to `^[a-z]{3}-rf[0-9]+[a-d]$` — it *must* carry a device letter (`ral-rf2a`). But D-045 and the L-chain say there is **ONE shared uploader per host** that "scans all four devices' folders … writes the heartbeat, and exits." A single shared process cannot author a device-lettered heartbeat: `ral-rf2a`, `-2b`, `-2c`, `-2d` are four distinct `agent_id`s and four distinct `_health/` objects, but there is one uploader run with one `queue_depth`/`disk_pct`/`dlq_count`. Now play it out on the server: HEALTH-CHECKS Layer-3 "Agent liveness" checks **"every active sensor's heartbeat fresh (< 3× upload cadence)"** *per sensor*. If the shared uploader writes only one heartbeat (say `ral-rf2a`), then `ral-rf2b/c/d` heartbeats are **permanently stale → dead-heartbeat alert fires forever** for three of every four sensors (D-049, P14). If instead the uploader writes four copies of the *same* body under four keys, then `queue_depth`/`disk_pct` are host-level but presented as per-sensor — and the "capture advancing per active sensor" cross-check (which *is* per-device, because `obs_` objects are per-`agent_id`) can disagree with a heartbeat that is actually host-aggregate. Either way the freshness contract is unsatisfiable as written.

**evidence.** LOCAL-HOST-DATA-FLOW-v2 L-chain line 11: "**plus ONE shared EPHEMERAL uploader per host (D-045)** … scans all four devices' folders, filters/splits/uploads, **writes the heartbeat**, and exits." Same paragraph, self-contradicting: "each device has its own scanner process, its own local folders, **its own uploader, and its own dead-letter queue**." Diagram line 26: "**(4 independent uploaders per host)**." L3 line 133: "PRODUCER GUARANTEES (**uploader, one per device**)." D-016: "1 shared uploader + 1 shared DLQ per host." D-045: "resident = 4 scanner processes … ephemeral = `uplink-upload.timer` → oneshot." heartbeat.v1.schema.json line 19-21: `agent_id` pattern requires a device letter. HEALTH-CHECKS-v1 line 20: "every **active** sensor's heartbeat fresh (< 3× upload cadence) … stale → **that sensor flagged**." The absent thing: no doc states whether there is one uploader (host-scoped) or four (device-scoped), and no doc reconciles a host-scoped uploader with a device-scoped heartbeat key.

**proposed_alternative.** Decide the cardinality explicitly and make the heartbeat match it. Cleanest: **one shared uploader per host, heartbeat keyed by a host-scoped id** — add a distinct `_health/{facility}-{host}.json` (drop the device letter for heartbeat only) OR keep four keys but have the *one* uploader emit four per-device heartbeats where `queue_depth`/`dlq_count`/`disk_pct` are computed **per device folder** (disk_pct is shared, so label it host-level in the schema). If instead you truly want four uploaders (matching the diagram and L3), then D-016/D-045 must say "4 ephemeral uploader timers per host" and accept four timer units — but then "shared DLQ" contradicts "its own dead-letter queue" and must also be resolved. The heartbeat schema needs an added `host` or `scope` field either way so the server knows whether `queue_depth` is a sensor number or a host number.

**downstream_impact.** HEALTH-CHECKS Layer-2 (`ear-agent check` reports "queue/DLQ depths"), Layer-3 (agent-liveness + agent-condition checks, "queue not monotonically growing across sweeps" — is that per-device or per-host?), P14 fixture (heartbeat fresh/stale semantics), D-049 dead-heartbeat alert, RUNBOOK "Dead heartbeat" response ("single agent: SSH → systemctl status uplink-scanner@x uplink-upload.timer" — note it names *one* timer, siding with host-scoped). Touches D-015, D-016, D-045, D-049, D-059, and the heartbeat schema. **Fix before repo creation** — it is a D-016-class structural call and the schema is already frozen (D-052).

**confidence.** high

### FINDING-REL.2 · severity: blocker · target: R7 "full rebuild over the configured window" / D-024 partition-per-date / P16 / RECOVERY-RETENTION raw-landing 90 d

**steelman.** The full-rebuild model is the spine of recovery-by-rebuild (D-023): each run is "a full rebuild over the configured window (deterministic given data + config)," so there is nothing to surgically repair — delete the L1 file, re-run, done. P16 asserts late arrivals reprocess cleanly: "Ledger ingests it; the window's events update deterministically (full-rebuild semantics); no duplicates." The ledger (R5.1) makes ingest idempotent by file identity, and atomic partition re-export (D-024) means a re-run replaces the whole `{date}` partition atomically. On its face, late arrival is solved by construction.

**failure_scenario.** "The configured window" is **never defined anywhere** — not in R7, not in RFANALYSIS-SCOPE, not in `rfanalysis.toml`'s enumerated params (line 66 lists bin size, sweep-collapse, burst gap, presence, duty multiplier, min-obs, safeguard, Pass-2 threshold — **no window length**). Two concrete breakages follow. (a) **Window shorter than late-arrival delay = silent permanent loss.** Q-19 leans "2–4×/day"; a natural window is "last 24–48 h." A warehouse WAN dies for 5 days (Q-14 admits WAN quality is unknown and per-site). On recovery the agent's `pending/` drains 5 days of backlog to R2. The ledger dutifully ingests every file (R5.1) into `raw_landing`. But if the pipeline only rebuilds events over "the last 48 h," the 5-day-old observations **land, enrich, and then are never grouped into events** — they fall outside the rebuild window. P16 only tests a file "e.g. WAN-outage backlog" with "older timestamps" but does not pin how old relative to the window; a persona with a 5-day-late file and a 2-day window would be **red**, and nothing in the spec says it must be green. (b) **Raw-landing 90-day prune vs 13-month R2 retention = the rebuild is a lie past 90 days.** RECOVERY-RETENTION says raw landing is "90 days rolling within the file; pipeline prunes at run start," but R2 `obs_` is kept 13 months "covers a full year + reprocessing window." "Delete the L1 file and rebuild" re-syncs from R2 and re-lands — fine within 90 days. But the *pruning* is "at run start," so a normal run that rebuilds a 90-day window will, on the same pass, prune landing older than 90 days that a *longer* reprocess would need. The 13-month R2 retention implies a 13-month reprocess capability that the 90-day landing prune silently defeats unless every reprocess first re-syncs — and re-sync is "skip-if-exists" (R5.1), so files already in the ledger are **not re-downloaded**, and if their landing rows were pruned, they are simply gone from the analytical layer until the ledger row is manually cleared.

**evidence.** DATA-WORKFLOW-RULES-v3 R7 line 198: "Each run is a full rebuild over the **configured window**." Line 202: "reflects the whole configured window." RFANALYSIS-SCOPE-v3 line 66: config param list — no window. GOLDEN-FIXTURE-SPEC P16 (line 31): "deliver a delayed file (older timestamps, e.g. WAN-outage backlog); run again … events update deterministically" — no bound on delay vs window stated. RECOVERY-RETENTION-v1 line 22-23: "Raw landing tables … **90 days rolling** … pipeline prunes at run start"; "R2 signal feed (`obs_`) **13 months** … covers a full year + reprocessing window." R5.1 line 156: "Sync downloads only **missing/size-mismatched** objects (skip-if-exists)." The absent thing: the relationship between window length, run cadence, max tolerated late-arrival delay, and the 90-day landing prune. No invariant of the form "window ≥ max_backlog_age" and no reprocess procedure that re-lands ledger-known-but-pruned files.

**proposed_alternative.** (1) Define `window` as a first-class, versioned config param and state the invariant **`rebuild_window ≥ raw_landing_retention ≥ max_expected_backlog_age`**, or decouple: make each run rebuild **all landing rows currently present** (landing retention *is* the window) rather than a separate time window — then late arrivals inside 90 days are always regrouped and there is one knob, not two that can disagree. (2) Add a persona P16b: file arriving *older than the routine window but within landing retention* must still produce/update its events. (3) For reprocess-from-R2 beyond landing retention, the procedure must **clear the ledger rows for that range** (so skip-if-exists re-downloads) — document it in RECOVERY-RETENTION; today "delete L1 file + re-run" does not touch the ledger, and the ledger's job is precisely to *prevent* re-ingest.

**downstream_impact.** R5.1 (ledger + skip-if-exists), R7 (rebuild semantics + determinism claim — "reflects the whole configured window" is only true if the window covers the data), D-023 (rebuild-from-R2 recovery), D-024 (partition re-export), RECOVERY-RETENTION (retention table + reprocess procedure), P16 (persona bounds), Q-14 (WAN quality sizes the required window), Q-19 (cadence). The 13-month R2 retention decision is spent money if nothing can reprocess past 90 days. **Structural; resolve before M3.**

**confidence.** high

### FINDING-REL.3 · severity: major · target: R5.1 ledger idempotency / R4 atomic PUT — the agent-side crash between PUT-confirmed and file-moved-to-completed

**steelman.** The delivery story is genuinely careful. R5.1 makes the *server* side crash-safe: "list → anti-join ledger → land new only → record in the **same transaction**," so a crash cannot record an unfinished load, and "post-load assertion: `observations` row delta == sum of ledger row_counts." R4 relies on S3/R2 PUT atomicity so no partial object is ever visible. L3 moves a file to `processing/` before reading it (write-complete barrier) and the OPEN item explicitly flags "crash-recovery of `processing/` on uploader restart." The brief's own steelman — atomic PUT + ledger keyed on filename+size/etag — means a re-uploaded identical file is a ledger no-op. So "exactly-once from antenna to events" is defensible *as a server-side dedup property*.

**failure_scenario.** The claim is at-least-once on the wire, dedup-to-exactly-once at the ledger — but the **agent-side** crash window is unspecified and the dedup key can miss it. Sequence: uploader run picks `obs_ral-rf2a_2026-07-02_000042.jsonl`, moves it `pending→processing`, gzips, PUTs to R2 (**PUT confirmed by R2**), and then — before the file is moved to `completed/` — the host loses power (warehouse breaker trips; this is the environment BACKUP-DR calls out as a real loss mode). Next timer tick: L3 crash-recovery "re-queues `processing/`" (the OPEN item, currently unbuilt). The file is re-gzipped and re-PUT. **gzip is not deterministic byte-for-byte** unless mtime/OS/name headers are zeroed — the default `gzip` header embeds a modification timestamp — so the second object has a **different size/etag** than the first even though the JSONL rows are identical. R5.1's ledger key is "filename(+size/etag)." The server now sees the same filename with a *different* etag: is that "size-mismatched" (R5.1 sync "downloads only missing/size-mismatched objects" — implying it re-downloads and **re-lands**, double-counting every row in that file) or a ledger-uniqueness violation on filename (rejected, so the newer bytes are dropped)? The contract says both things in different places (sync keys on size-mismatch → re-download; ledger uniqueness → filename). One path double-counts; the other silently drops. R5.1's own post-load assertion (`row delta == Σ ledger row_counts`) would *catch* the double-count only if the re-land is a *new* ledger row — but if it reuses the filename it violates uniqueness first.

**evidence.** DATA-WORKFLOW-RULES-v3 R5.1 line 156: "Sync downloads only **missing/size-mismatched** objects (skip-if-exists)." Line 157: "tracked in an `ingest_ledger` (**filename, size/etag**, row_count …)." Line 169 ENFORCED BY: "Ledger uniqueness constraint on **filename(+size/etag)**." R4 line 120: "re-upload of changed content uses a new key **or is treated as an error** (R5 keys on filename+size/etag)." L3 line 147 OPEN: "confirm files stuck in `processing/` after a crash are **re-queued**, not orphaned." R3 GREENFIELD NOTE line 86: "`ear-agent` v5 ships **real gzip**" — but nothing requires *deterministic* gzip (no `mtime=0`/`--no-name`). The absent thing: a rule that agent re-upload of the *same logical file* produces byte-identical objects (or that the ledger key ignores compression-layer bytes and keys on `(filename, row_count)` or an app-level content hash). The tension between "size-mismatched → re-download+re-land" and "filename uniqueness → reject" is never resolved for the same-filename-different-bytes case.

**proposed_alternative.** Pin **deterministic gzip on the agent** (`gzip --no-name`, mtime zeroed) as an R3 contract so a re-PUT of identical rows is byte-identical (same etag) → ledger no-op by construction; add it to the WP-A4 acceptance and a persona (extend P12 gzip). Independently, make the ledger key an **application-level content identity** (agent_id + filename + row_count + a content hash of the *decompressed* rows) rather than the R2 object etag, so the compression layer cannot perturb dedup. State explicitly which wins when filename matches but bytes differ: **reject as an integrity error + quarantine + alert**, never re-land. Add the crash-window persona: "file PUT-confirmed then host dies before `completed/` move → re-run → exactly one logical ingest."

**downstream_impact.** R4, R5.1 (the two conflicting keying statements), R3 (gzip determinism becomes a contract), L3 (crash re-queue must pair with deterministic re-PUT), P12/P16 personas, the "exactly-once" claim the brief leans on, BACKUP-DR "Warehouse host dies" loss-window analysis. Also touches R7's determinism test if a double-counted file inflates `observation_count`. **Fix during build (M2/M3), but pin the gzip-determinism contract before M2.**

**confidence.** high

### FINDING-REL.4 · severity: major · target: R1 "NTP-synced clock as provisioning requirement" / EVENT-CONSTRUCTION 600 s burst gap / WP-O3 time-sync (stub) / `ntp_offset_ms`

**steelman.** The design already *names* the risk, which is more than most do. R1's CHANGES note: "NTP-synced clocks become a *provisioning requirement*, because CV analysis of inter-event gaps across 60 sensors is meaningless if clocks drift." D-057 added `ntp_offset_ms` to the heartbeat for "continuous clock-drift visibility (time-sync validation hook)," HEALTH-CHECKS Layer-3 "Agent condition" checks "`ntp_offset_ms` within tolerance," and `ear-agent check` verifies "NTP synchronized." The event is per-sensor (burst grouping is within one `agent_id`), so a *single* sensor's clock skew shifts all its own timestamps uniformly and does **not** corrupt its own 600 s gap grouping (a constant offset preserves gaps). So the steelman is strong: intra-sensor gap analysis is offset-invariant, and the drift hook exists.

**failure_scenario.** Two places break, and both are *before* WP-O3 turns time-sync into a contract. (a) **Correlation is cross-sensor and gap-sensitive, and it ships in this build (D-055).** Correlation joins same-band sensors within a facility "overlapping time windows" (R8) — that is the whole point of two same-band antennas seeing one device. If `ral-rf2a`'s clock is +8 s and `ral-rf2c`'s is −5 s (13 s relative skew — trivially achievable on a host whose NTP daemon died weeks ago and whose only signal is an `ntp_offset_ms` field *nobody has wired a hard gate to*), a transmission seen by both at true-time T is stamped T+8 and T−5. The "time-overlap" correlation (D-055, provisional) either fails to overlap short bursts or mis-pairs them. (b) **The 600 s gap is offset-*invariant* but not drift-*invariant*.** If a sensor's clock **drifts** (rate error, not constant offset) — e.g. a cheap RTC gaining 2 s/hour with a dead NTP daemon — then two detections truly 590 s apart get stamped 590±drift; near the 600 s boundary this flips burst membership: what should be one event splits into two, or two split events merge. The primary target is a power-saving tracker beaconing at long intervals (P15: multi-day gaps) — precisely the regime where a slow clock drift accumulates between beacons and lands them on the wrong side of a burst boundary. And the *guard* is soft: `ntp_offset_ms` is "within tolerance" in a health check, but **no tolerance value is specified anywhere**, and there is no rule that says "reject/quarantine observations from a sensor whose offset exceeds X" — the data still flows into events. WP-O3 time-sync is explicitly "a stub not a contract" (RECOVERY-RETENTION line 2; plan WP-O3 "Remaining: time-sync tolerance + drift detection"). So the drift can corrupt burst grouping *before* the mechanism that would catch it exists.

**evidence.** DATA-WORKFLOW-RULES-v3 R1 line 58: "NTP-synced clocks become a *provisioning requirement*, because CV analysis of inter-event gaps across 60 sensors is meaningless if clocks drift." EVENT-CONSTRUCTION-METHODOLOGY line 26-28: 600 s burst gap. R8 line 218: correlation on "overlapping time windows." heartbeat.v1.schema.json line 60-66: `ntp_offset_ms` nullable, no bound. HEALTH-CHECKS line 21: "`ntp_offset_ms` within tolerance" — tolerance undefined. GREENFIELD-PLAN WP-O3 line 78: "Remaining: **time-sync** tolerance + drift detection." RECOVERY-RETENTION line 2: "Time-sync and security remain scheduled for WP-O3." The absent thing: (1) a numeric NTP-offset tolerance, (2) a hard action (quarantine/degrade) when exceeded — not just a health "flag," (3) any statement that correlation/event-building must account for or reject skewed sensors, (4) a persona exercising a drifting-clock sensor near the 600 s boundary.

**proposed_alternative.** Promote *one* piece of WP-O3 ahead of first-facility: a numeric `max_ntp_offset_ms` tolerance (start conservative, e.g. 1000 ms) and a **hard gate** — a sensor whose heartbeat `ntp_offset_ms` exceeds it (or is `null`, meaning NTP not measurable) is auto-**degraded** (its data quarantined or excluded from correlation, alert fired), not merely "flagged" on a banner. Add a fixture persona: sensor with linear clock drift crossing the 600 s boundary → assert events do not spuriously split/merge relative to a synced twin, OR (if you accept that this is unfixable without the sync contract) assert the sensor is *excluded* and flagged rather than silently corrupting the dataset. Note in EVENT-CONSTRUCTION that 600 s grouping is offset-invariant but *not* drift-invariant, so drift detection is load-bearing for burst correctness, not just cross-sensor CV.

**downstream_impact.** R1, R8 (correlation), EVENT-CONSTRUCTION (600 s), D-055 (provisional correlation ships now, so this is not deferrable to a "future Level 2"), D-057/`ntp_offset_ms`, HEALTH-CHECKS Layer-3, WP-O3, RUNBOOK (needs a "clock drift" alert response). Extends the REVIEW-BRIEF's own "Time-sync … under-specified?" flag with the specific claim that *correlation ships in this build* so the deferral of the time-sync contract is not safe. **Promote the tolerance+gate before first facility (M5), not "later."**

**confidence.** high

### FINDING-REL.5 · severity: major · target: L5 / D-045 "next timer tick retries" — the poison-that-isn't-poison and the partial-week wedge

**steelman.** D-045's reframe is elegant and I want to protect its core (see §3): "`pending/` IS the retry queue by construction; failed run exits leaving files; timer cadence = backoff; DLQ narrows to genuinely poison files only." Retiring the circuit breaker deletes a real bug class (the historic breaker/watcher mechanics). The disk ceiling (L5, WP-A3) bounds the blast radius so unbounded `pending/` growth alerts and evicts. For the common case — WAN down, then WAN up — this is strictly better than a resident uploader: no state machine to wedge, files just drain on the next tick.

**failure_scenario.** "The next timer tick retries" is only safe if a failed run is **idempotent and makes forward progress on the files it *can* process**. Two wedge modes are unaddressed. (a) **One poison file starves the whole run.** The uploader "scans all four devices' folders, filters/splits/uploads" in one oneshot. If processing is a simple loop and one `pending/` file is genuinely malformed (truncated JSONL from a scanner OOM mid-flush, or a file the edge filter throws on), does the run (i) skip it to DLQ and continue, or (ii) exit non-zero on first error, leaving *every other device's* good files un-uploaded until someone removes the poison file? The spec says "DLQ = poison files only" but never says the run **isolates** a poison file and continues — and D-044's sibling rule for the *pipeline* is explicitly "halt-on-failure = non-zero exit." If the uploader inherits halt-on-failure semantics, one bad file on device `a` blocks upload for `b/c/d` indefinitely, `pending/` grows on all four, and the disk ceiling eventually evicts *good* data to make room while the poison file sits there. (b) **Partial-progress not durable → a chronically slow link never completes a "week."** If a run must upload N backlog files but the link is slow enough that the oneshot is still running when the *next* timer fires, what happens? Two overlapping upload oneshots on the same `pending/` folder can move the same file `pending→processing` twice or race the `completed/` move. Unlike the pipeline (which has an explicit run lock, D-044), **the uploader has no documented run lock** — D-044's lock is for `run_level1`; D-045 says nothing about preventing a second `uplink-upload` oneshot from starting while the first is mid-drain of a large backlog. systemd timers with `OnUnitActiveSec` can and do overlap a long-running oneshot unless the unit is `Type=oneshot` with `RefuseManualStart`/a lock; the spec does not require it.

**evidence.** DECISION-LOG D-045: "failed run exits leaving files; timer cadence = backoff; DLQ narrows to genuinely poison files only" — no statement of intra-run poison isolation or partial-progress durability. D-044 (sibling): "halt-on-failure maps to exit codes … overlap prevented by a run lock" — the lock is scoped to the *pipeline*, and the halt-on-failure philosophy, if copied to the uploader, is exactly the wrong choice. EAR-AGENT-SPEC line 30: "`pending/` is the retry queue … DLQ = poison files only" — silent on continue-past-poison. LOCAL-HOST-DATA-FLOW line 11: one uploader "scans all four devices' folders" — one process, four folders, no per-folder isolation guarantee. The absent thing: (1) "the uploader isolates a failed file to DLQ and **continues** with the rest of the batch (never halt-on-first-error)," (2) an uploader-side **run lock / `flock`** preventing overlapping oneshots, (3) durability of `processing→completed` progress so a killed mid-drain run doesn't re-process.

**proposed_alternative.** State in D-045 / L5 that the uploader is **best-effort-per-file, not halt-on-failure**: each file succeeds, fails-to-DLQ, or is left in `pending/` independently; one poison file never blocks the batch (this is the *opposite* of D-044's pipeline halt-on-failure, and the asymmetry should be explicit because they look symmetric). Add an **uploader run lock** (`flock` on a per-host lockfile; if held, the tick exits immediately — the queue persists, next tick tries) so a slow backlog drain cannot be raced by the next timer. Add a persona: `pending/` containing 1 poison + 3 good files across two devices → poison → DLQ + alert; the 3 good files upload same run. Add: overlapping-tick attempt is a no-op.

**downstream_impact.** D-044 vs D-045 asymmetry (must be documented, not assumed symmetric), L5, EAR-AGENT-SPEC WP-A3/A4, disk ceiling (a wedged run makes the ceiling fire on *good* data — the eviction policy then matters more), DLQ retention (Q-09), P17 conservation persona (a poison file must not silently vanish from the conservation count), kill-WAN drill (WP-A5 should add a poison file to the drill). **Fix during M2 build; document the halt-on-failure asymmetry now.**

**confidence.** high

### FINDING-REL.6 · severity: major · target: RECOVERY-RETENTION "stale lock … older than the timer period, broken automatically" / D-044 run lock / Q-19 cadence

**steelman.** The run lock (D-044) is the right primitive: "overlap prevented by a run lock; a stale lock (host crash mid-run) older than the timer period is broken automatically with a `notify()`." This is the standard cron-mutex pattern, it degrades safely (a crashed run leaves a stale lock, the next scheduled run notices it is older than the period and breaks it), and it alerts so the operator knows a run was interrupted. For a single-operator system this is appropriately simple.

**failure_scenario.** "Older than the timer period" is the wrong staleness criterion when **a legitimate run can outlive the timer period** — and at fleet scale it routinely can. Q-19 leans 2–4×/day, i.e. a timer period of ~6–12 h; that is safe. But Q-22 (fleet-scale load) is *deferred*, and the pipeline is "per-agent stages parallelized with a process pool" over up to 60 sensors × multi-day windows with an *expensive Pass-2* whose compute budget is only sized in WP-S5. Suppose the operator, chasing dashboard freshness, sets the timer to hourly (Q-19 is OPEN — nothing forbids it), and one night a large backlog (post-WAN-outage, 5 days × 60 sensors) makes a legitimate run take 90 minutes. The lock is now "older than the timer period" (1 h) **while the run is still healthy and executing**. The next tick's auto-break **steals the lock from a live run**, starts a second `run_level1` against the same per-agent DuckDB files, and now two processes open→work→close the same `l1/{facility}/{agent_id}.duckdb` — DuckDB single-writer semantics mean one corrupts or errors, the "delete + rebuild" recovery kicks in, the rebuild is *also* over the huge backlog and *also* outlives the period, and you have a **thundering-herd of self-stealing runs** that never completes. The disposability that makes L1 safe (D-023) is exactly what lets this loop: every corrupted file is cheerfully deleted and rebuilt, forever.

**evidence.** RECOVERY-RETENTION-v1 line 12: "a stale lock (host crash mid-run) **older than the timer period** is broken automatically with a `notify()`." D-044: "overlap prevented by a run lock … Run cadence: OPEN (Q-19)." Q-19 (OPEN-QUESTIONS line 25): "how often should the ephemeral Level 1 job run — hourly, or a few times daily?" — hourly is on the table. Q-22 (line 31): fleet-scale load gate **deferred** — so nobody has measured whether a run *can* exceed an hour. RFANALYSIS-SCOPE line 60: per-agent process pool, "concurrency bounded by pool size × memory_limit" — a bounded pool means a large fleet serializes, extending wall-clock time. WP-S5: Pass-2 budget only sized later. The absent thing: a liveness signal on the lock (heartbeat/pid-liveness) distinct from wall-clock age; a rule that the stale-lock threshold must be **≥ max observed run duration**, not "the timer period"; any coupling between Q-19 (cadence) and measured run time.

**proposed_alternative.** Break the stale lock on **process liveness, not wall-clock age**: the lock file carries the pid (and/or a heartbeat mtime the running job touches every N minutes); the next tick breaks it only if the pid is dead OR the heartbeat is stale by a margin — never merely because wall-clock exceeded the timer period. Alternatively/additionally, make the timer `Type=oneshot` with systemd's own `RefuseManualStart` + a `flock` so an overrunning run *blocks* the next tick (which exits as a no-op) rather than *racing* it — a run that outlives its period should skip the next slot, not be murdered. Add the invariant "stale-lock-age-threshold ≥ P95 run duration measured under fleet-scale load (Q-22)" and **un-defer the Q-22 load gate at least far enough to know the max run time before setting Q-19 cadence**. Persona: a run whose duration exceeds the timer period must complete uninterrupted; the overlapping tick is a no-op.

**downstream_impact.** D-044 (lock semantics), Q-19 (cadence cannot be chosen safely without knowing max run time), Q-22 (deferral is unsafe for *this* reason specifically — extend it), D-023 (disposability enables the loop), RECOVERY-RETENTION (stale-lock rule), HEALTH-CHECKS Pipeline check ("last L1 run recent < 2× cadence AND exit 0" would show perpetual failure but not diagnose the self-steal), RUNBOOK (needs a "run outran its timer" response). **Fix the lock semantics before M3 (WP-S4 builds the lock); extend Q-22 deferral note now.**

**confidence.** high

### FINDING-REL.7 · severity: major · target: L5 disk ceiling / eviction policy — un-uploaded field data is the only true loss window, and eviction deletes it silently-ish

**steelman.** The disk ceiling is correctly identified as the blast-radius bound (Risk register: "disk ceiling bounds the blast radius"), it is a v5 build requirement not an open bug (L5 RESOLVED note, D-017), it is checked "at the start of every upload run," and it alerts (D-049 disk-ceiling breach → ntfy). BACKUP-DR is honest that agent `pending/`/DLQ is "Transient (un-uploaded field data!) … protected by disk ceiling + heartbeat visibility." The heartbeat carries `disk_pct` and `status: ceiling` so backlog growth is visible per tick. For a bounded outage this is a reasonable, alerting-first design.

**failure_scenario.** The eviction *policy* is named ("cap size + oldest-drop policy, or alert-and-hold" — L5 OPEN item #1) but **never decided**, and the two branches have opposite failure modes for a *surveillance* system. If eviction is **oldest-drop**: during a multi-day WAN outage the disk fills, and the ceiling evicts the *oldest* `pending/` files — which are exactly the field observations from the start of the outage, permanently, before the server ever saw them. For a tracker-detection system the *earliest* evidence of a device's arrival (first-seen — a core scoring factor, "new+stationary") is the most valuable, and oldest-drop deletes precisely that. The operator gets a `status: ceiling` alert but the *data is already gone* and RUNBOOK only says "If eviction ran, note the window in the run log" — no recovery, because there is none. If instead the policy is **alert-and-hold** (stop accepting new writes when full): then the *scanners* (resident, `Restart=always`) keep producing and their buffered writes to `pending/` fail — does the scanner crash-loop, drop in memory, or block? Unspecified. Either the newest data is lost (symmetric problem) or the scanner wedges. And the ceiling is checked "at the start of every upload run" (~5 min) — but the *scanners* write continuously between runs, so a fast-filling disk (50 MB rotation × 4 devices) can blow past the ceiling *between* checks, and the check is on the uploader, not the writer.

**evidence.** LOCAL-HOST-DATA-FLOW L5 OPEN (line 235): "Decision needed: cap size + oldest-drop policy, or alert-and-hold. **Recommend resolving before fleet rollout.**" (Still open.) RECOVERY-RETENTION line 29: "Agent pending/ | until uploaded (ceiling-bounded) | D-045" — no policy. BACKUP-DR line 13: "Agent `pending/`/DLQ | Transient (un-uploaded field data!) | none — protected by disk ceiling." RUNBOOK line 15: "If eviction ran, note the window in the run log." EAR-AGENT-SPEC line 30: "disk ceiling checked at run start, eviction policy + alert" — policy unnamed. Q-10 (line 16): hardware baseline "sizes the disk ceiling default" — still open, so the *headroom* is unknown too. The absent thing: (1) the eviction policy decision, (2) which end (oldest/newest) is dropped and the surveillance-data justification, (3) what a resident scanner does when the disk is full between upload runs, (4) whether the ceiling is enforced at *write* time, not just upload time.

**proposed_alternative.** Decide the policy explicitly and, for a surveillance system, prefer **alert-and-hold with graceful scanner backpressure over oldest-drop**: when the ceiling is hit, scanners *pause capture* (log + heartbeat `status: ceiling` + alert) rather than silently deleting the earliest evidence — a *gap* in capture is honest and visible; a silent oldest-drop is invisible evidence destruction. If oldest-drop is chosen anyway (disk truly cannot hold a plausible outage), drop **context feed (`ctx_`) first, signal feed (`obs_`) last, and DLQ never** — a tiered eviction that sheds the least-valuable data first, and *emit a durable "capture gap" context record* so the server knows data was dropped (never a silent hole). Enforce the ceiling at **write time in the scanner**, not only at upload-run start, so continuous writes cannot overrun between ticks. Tie the ceiling default to Q-10 (hardware) and Q-14 (WAN quality → plausible outage length → required buffer size).

**downstream_impact.** L5, D-045, BACKUP-DR (loss-window claim), RECOVERY-RETENTION, RUNBOOK (eviction response), EAR-AGENT-SPEC WP-A3, Q-10, Q-14, HEALTH-CHECKS (a capture gap must surface distinct from "sensor dead"), the scanner (resident) gains a backpressure behavior it does not currently have, `observation-context` schema (add a `capture_gap` record_type if you go the durable-gap route). **Resolve before fleet rollout (as L5 itself already recommends) — extend that open item with the tiered/backpressure proposal.**

**confidence.** high

### FINDING-REL.8 · severity: major · target: R5.0 onboarding gate under partial delivery / D-024 QA isolation — a promoted sensor with in-flight backlog

**steelman.** The onboarding gate (R5.0) is a genuine strength: "sync iterates `Sensor.objects.filter(status='active')`; a 'discover pending' function surfaces new prefixes but cannot enqueue them," and QA isolation is *structural* via the handoff partition ("a `qa` agent's partition is simply not globbed by production Level 2," D-024). Auto-discovery-≠-auto-production is enforced by construction, and the lifecycle `pending→qa→active→disabled` is clean. P18 tests that a `qa` sensor's events appear in no production output.

**failure_scenario.** The gate governs *which prefixes are fetched*, but status is mutable mid-stream and the pipeline is a **full rebuild over the window** — the interaction is unspecified. Sequence: `cht-rf3a` is in `qa`; its `pending/` has been draining a 3-day backlog into R2 under its prefix; the operator reviews it (RUNBOOK onboard step 3-4), likes it, and promotes to `active`. The next `run_level1` fetches `active` prefixes including `cht-rf3a` and rebuilds its events **over the whole window** — including the 3 days of data that were captured *while it was qa*. So data collected under `qa` status (explicitly "not in production reports or correlation," R5.0) **enters production events retroactively** the instant the sensor flips to `active`, because the rebuild is time-windowed, not status-windowed, and the ledger already landed those files during the qa isolation runs (or does qa isolation *not* land them to the same per-agent DuckDB? — unspecified: R5.0 says qa is "synced and analyzed in isolation (staging context)," but D-023's per-agent DuckDB file is keyed by `agent_id`, not by status, so is there a *separate* qa DuckDB file or the same one?). If the same file, promotion re-labels historical qa data as production with no boundary. If a separate qa file, promotion must *migrate or re-sync*, and skip-if-exists (R5.1) means the production run won't re-download files the qa run already fetched — so production may have **no landing rows** for the pre-promotion window and silently start events at the promotion instant, losing the very onboarding-window data the operator just validated.

**evidence.** DATA-WORKFLOW-RULES-v3 R5.0 line 135-138: qa "synced and analyzed **in isolation** (staging context) … **Not in production reports or correlation.**" R7 line 198: "full rebuild over the configured window." D-023 / RFANALYSIS-SCOPE line 41: "`l1/{facility}/{agent_id}.duckdb`" — keyed by agent_id, no status dimension. R5.1 line 156: "skip-if-exists." EVENT-DATASET-HANDOFF line 18: "a `qa` agent's partition is simply not globbed by production Level 2" — handles the *handoff* read side but not the *landing/rebuild* side when status changes. P18 (fixture) tests a *statically* qa sensor, not a *promotion transition*. The absent thing: what happens to already-landed qa-window data at the qa→active transition; whether qa uses a separate DuckDB file/partition; whether promotion re-syncs or re-labels; any persona for the transition.

**proposed_alternative.** Specify the transition explicitly. Option A (clean boundary): qa data lands in a **status-isolated DuckDB file** (`l1/{facility}/{agent_id}.qa.duckdb`) and a separate qa handoff partition; promotion sets an `active_since` timestamp on the Sensor and production events are built only for `observed_at ≥ active_since` — pre-promotion data stays in the qa artifact, never silently becoming production. Option B (accept retroactive inclusion but make it explicit): document that promotion *does* fold the onboarding window into production and add a persona asserting exactly that, so it is a decision not an accident — but then R5.0's "qa data never enters production" NEVER is violated and must be reworded. Either way add `active_since` to the Sensor model and a P18b promotion-transition persona.

**downstream_impact.** R5.0 (the "qa never enters production" NEVER), R5.1 (skip-if-exists vs re-sync on promotion), R7 (window vs status), D-023 (per-agent file keyed by status or not), D-024/handoff (qa partition globbing), Sensor model (add `active_since`), P18 (static → add transition persona), RUNBOOK onboard procedure. **Resolve before M3 (WP-S2 builds the gate).**

**confidence.** med

### FINDING-REL.9 · severity: minor · target: R4 "atomic PUT" reliance / R2 eventual-consistency of LIST for the sync anti-join

**steelman.** R4 correctly relies on S3/R2 single-object PUT atomicity ("an object is either fully present or absent") — this is a real, documented guarantee of the platform, and the design leans on it rather than reinventing it. The sync anti-joins a LIST against the ledger to find new objects (R5.1), which is the standard idempotent-ingest pattern. Cloudflare R2 offers strong read-after-write for object reads.

**failure_scenario.** The sync's newness test is **LIST-based** ("list → anti-join ledger → land new only"), and LIST is where object stores are weakest for freshness. An agent PUTs `obs_…000042.gz` at T; the server's pipeline LISTs the prefix at T+2s. If R2's bucket LIST is not guaranteed to reflect a just-written key immediately (list-after-write is a weaker guarantee than read-after-write on many S3-compatible stores, and R2's list consistency is not asserted anywhere in these docs), the object is *missed this run* and picked up next run — usually harmless (the pipeline is periodic), **but** combined with FINDING-REL.2's undefined window, an object that first becomes LIST-visible *after* it has aged out of the rebuild window would land but never group. More concretely: the "post-load assertion `observations` row delta == Σ ledger row_counts" (R5.1) assumes LIST returns a stable, complete set within a run; if LIST is paginated and an object is added mid-pagination, the count assertion could spuriously fail (false alert) or a partial page could be treated as complete.

**evidence.** DATA-WORKFLOW-RULES-v3 R4 line 121: "Atomicity is a property of S3/R2 PUT — relied upon, documented." R5.1 line 156: "list → anti-join ledger → land new only." R4 CONSUMER MAY ASSUME line 116: "Listing a prefix yields only complete objects. Object size is stable once listed." The absent thing: any statement about **LIST consistency/freshness** (as opposed to PUT atomicity and object read consistency) — the design documents PUT atomicity but is silent on whether a just-PUT key is guaranteed to appear in the very next LIST, and on LIST pagination stability during the count assertion.

**proposed_alternative.** State the LIST-consistency assumption explicitly and make the design robust to a missed-then-later-visible object: this is already true *if* FINDING-REL.2 is fixed (window ≥ backlog age), so this finding is largely subsumed — but add a one-line note in R4/R5.1 that newness is eventually-consistent via LIST and that an object missed in run N is caught in run N+1, and ensure the row-delta assertion is computed over the *same* LIST snapshot it landed from (not a re-LIST), so pagination churn can't cause a false alert. Low cost, prevents a confusing intermittent alert.

**downstream_impact.** R4, R5.1 (count assertion robustness), HEALTH-CHECKS "Capture advancing" check (a LIST lag could read as "none arriving"), FINDING-REL.2 (subsumes the serious version of this). **Documentation fix; before M3.**

**confidence.** low

### FINDING-REL.10 · severity: minor · target: D-049 single-channel alerting / HEALTH-CHECKS self-monitoring — the alert path is a single point of failure with no independent liveness

**steelman.** D-049's one-`notify()`-to-ntfy design is deliberately minimal for a single operator: "One channel, one env var, no alerting stack to run." HEALTH-CHECKS adds real defense-in-depth: state-change-only alerting (no spam), a non-zero exit as a backstop "if `notify()` itself is broken," and self-monitoring ("if no `HealthSweep` row lands within 2× cadence, the dashboard banner turns amber … web `/health/` reports it"). The timer `OnFailure=` is a second path. This is thoughtful for the scale.

**failure_scenario.** All alerting flows through `ALERT_URL` (one ntfy topic), and every backstop still terminates at *the same channel or a dashboard nobody is looking at*. If the ntfy topic/token is misconfigured, the topic is deleted, or the operator's phone has the app muted, then: the pipeline's `OnFailure=` calls `notify()` → same dead channel; the health sweep's red exit → `OnFailure=` → same dead channel; the self-monitoring amber is *on a dashboard the single operator is not watching at 3am*. There is **no independent proof-of-life** — nothing tells the operator "alerting itself is up." So the failure mode is: alerting silently breaks, then a real incident (a facility WAN outage filling disks toward eviction) occurs, and **no alert arrives for either the incident or the broken alerting**. For a system whose only human is asleep and whose data loss window is "un-uploaded `pending/`," a silent alerting outage converts a recoverable incident into data loss.

**evidence.** DECISION-LOG D-049: "one `notify()` utility everywhere, POSTing to `ALERT_URL` (an ntfy topic) … One channel, one env var." HEALTH-CHECKS line 29-31: backstops all route through `notify()` or the dashboard banner. The absent thing: a **heartbeat/dead-man's-switch on the alert channel itself** — a periodic "I am alive" ping the operator would *notice the absence of* (ntfy supports scheduled/expected messages; healthchecks.io-style dead-man switches exist). No "test the alert path" step in provisioning beyond a one-time probe.

**proposed_alternative.** Add a **dead-man's-switch**: a lightweight host timer POSTs a heartbeat to an external monitor (healthchecks.io free tier, or a second ntfy "expected-daily" message) whose *absence* pages the operator — so a broken `ALERT_URL` is itself detectable. Keep D-049's simplicity for the primary path; this is one extra timer and one external dependency, justified because the alert channel is currently a silent SPOF. Add "verify alert path end-to-end (send + receive on phone)" to HOST-PROVISIONING and to `ear-agent check`.

**downstream_impact.** D-049, HEALTH-CHECKS Layer-3/self-monitoring, HOST-PROVISIONING-CHECKLIST, `ear-agent check` (WP-A5), the SRE reviewer's single-operator-load concern. **Low cost; add during M5/WP-O1.**

### FINDING-REL.11 · severity: minor · target: L3 `processing/` crash-recovery (unbuilt OPEN item) as a build-gating dependency for exactly-once

**steelman.** The design honestly flags this as OPEN ("confirm files stuck in `processing/` after a crash are re-queued, not orphaned" — L3 line 147) and EAR-AGENT-SPEC line 27 lists "crash recovery re-queues `processing/`; no orphan states" as a contract-critical behavior with a persona hook. So it is not forgotten; it is scheduled.

**failure_scenario.** This is the *substrate* FINDING-REL.3 depends on: crash-re-queue is the mechanism that re-PUTs a file after a crash, and if it is naive (move everything in `processing/` back to `pending/` on start) it will re-PUT a file that was *already successfully PUT but not yet moved to `completed/`* — triggering exactly the duplicate/etag problem in REL.3. Conversely, if crash-recovery is *skipped* (orphan-in-`processing/`), a file that was moved to `processing/` and then the run died *before* PUT is **never uploaded** — silent loss inside the "durable" local buffer. The OPEN status means neither the re-queue trigger nor its idempotency is pinned, yet P16 (late arrival) and the "exactly-once" claim both assume it works.

**evidence.** L3 line 147 OPEN (unresolved). EAR-AGENT-SPEC line 27 (contract-critical, but implementation unspecified). No persona in the enumerated list (P1–P19) explicitly exercises "file in `processing/` at crash → exactly-once outcome" (P16 is late-*arrival*, not crash-in-processing). The absent thing: the re-queue trigger (on scanner start? uploader start? both?), and its pairing with deterministic re-PUT (REL.3).

**proposed_alternative.** Resolve the OPEN item as a contract, not a "confirm": on uploader start, files in `processing/` are moved back to `pending/` **and** re-PUT idempotently (relies on REL.3's deterministic gzip + content-hash ledger key so a re-PUT of an already-uploaded file is a server-side no-op). Add the missing persona (crash-in-`processing/` → exactly-once). Gate WP-A3 on it.

**downstream_impact.** L3, EAR-AGENT-SPEC WP-A3, FINDING-REL.3 (direct dependency), P16 and any exactly-once claim. **Fix during M2.**

### FINDING-REL.12 · severity: minor · target: D-002 R2-primary / facility WAN-partition behavior on the server-fetch side (asymmetry with agent side)

**steelman.** D-002 correctly makes R2 the rendezvous for a multi-facility, often-remote-server topology ("Store-and-forward becomes load-bearing"). The *agent* side of a WAN partition is well-handled: `pending/` buffers, ceiling bounds, heartbeat shows backlog. The RUNBOOK "R2 outage playbook" covers both sides. HEALTH-CHECKS notes "many stale in one facility → WAN suspicion noted."

**failure_scenario.** The *server*-side partition is under-specified in one respect: the server fetches *all facilities* from one R2 bucket (central topology, D-004 still OPEN). If the **server↔R2** link partitions (not a facility's link — the server's own link, or R2 itself degrades for LIST), the pipeline "runs fail/produce nothing new — silence alerts for the window" (RUNBOOK). But a *partial* R2 degradation — LIST works, GET is throttled/failing for some objects (R2 rate limits, or a large post-outage backlog fetch hitting egress limits) — is not "fail cleanly." The sync would land *some* agents and fail others mid-run; the pipeline is per-agent so it might export handoff partitions for the agents that synced and none for the throttled ones, producing a **partial event dataset that looks complete** (HEALTH-CHECKS "handoff advancing per active agent" would flag the missing ones, but the *dashboard* renders the partial set as the current picture). At facility scale (Q-14 WAN unknown, Q-22 load deferred), a 5-day × 60-sensor backlog fetch is exactly the kind of load that hits R2 egress/rate limits and produces this partial-fetch state.

**evidence.** D-002; D-004 OPEN (central vs per-facility). RUNBOOK "R2 outage playbook" line 32: assumes clean fail ("pipeline runs fail/produce nothing new"). HEALTH-CHECKS "Handoff advancing" flags stale-per-agent but the dashboard banner is green/amber/red at the *system* level. RFANALYSIS-SCOPE line 55-60: per-agent chains, some can succeed while others fail. The absent thing: behavior on *partial* R2 fetch failure (rate-limit/throttle mid-run), and whether a run that synced only some active agents is marked degraded vs success; retry/backoff on the *server* sync (the harvested `gittest2 sync.py` "typed, retry" is mentioned but the retry semantics under sustained throttle are not).

**proposed_alternative.** Specify: a `run_level1` that fails to sync any *active* agent's expected objects (detectable: sensor is `active`, heartbeat fresh at the agent, but server sync got 0 new for it under throttle) marks the run **degraded** (distinct exit code) and the dashboard banner reflects "partial fetch — N agents not synced," never green. Add server-side sync backoff/retry with a per-run budget and a "throttled" alert distinct from "outage." Tie the backlog-fetch load expectation to Q-22 (un-defer enough to know egress limits) and Q-14 (WAN quality).

**downstream_impact.** D-002, D-004 (central topology concentrates this risk), RUNBOOK, HEALTH-CHECKS (partial-fetch verdict), RFANALYSIS-SCOPE (sync retry/degrade semantics), Q-14, Q-22, dashboard status banner honesty. **Document before M3; the throttle behavior matters most for the central topology (Q-04).**

**confidence.** med

## 3. WHAT TO PROTECT (guard against improvement-churn during build)

- **D-045 core: `pending/` IS the retry queue; circuit breaker retired; DLQ = poison only.** This is the strongest reliability decision in the set. It deletes an entire bug class (watcher/breaker state machines) by replacing a resident stateful process with a stateless idempotent timer oneshot whose backoff *is* the timer cadence. My findings REL.1/REL.5 attack its *edges* (heartbeat authorship, poison isolation, run lock) — none of them justify reintroducing a resident uploader or a circuit breaker. Protect the oneshot-timer model; fix the edges within it.
- **D-023 + D-024: per-agent disposable L1 DuckDB + atomic partitioned Parquet handoff.** "Delete + rebuild, never edit in place" (RECOVERY-RETENTION) makes recovery fall out of the architecture rather than requiring surgical repair, and the atomic temp-write+rename makes a half-written partition unobservable. This is exactly right for a batch forensic pipeline. REL.6 attacks the *lock* that overlaps runs, not the disposability — keep the disposability.
- **R5.1 same-transaction land+ledger + row-delta assertion.** "Crash cannot record an unfinished load" plus "row delta == Σ ledger row_counts" is a genuinely strong idempotency-at-ingest design. REL.3 attacks the *agent-side* etag/gzip edge, not this server-side transaction — protect the same-transaction rule and the post-load assertion.
- **R5.0 onboarding gate as a coded allowlist (active-only iteration; pending surfaced-not-enqueued).** Auto-discovery-≠-auto-production, enforced structurally by per-agent R2 folders, is a real improvement over the donor's auto-discover behavior. REL.8 attacks the *promotion transition*, not the gate — keep the gate.
- **R0 change discipline + the feed split (obs_/ctx_) making false-events structurally unrepresentable.** Not strictly my lens, but it is the reason contract violations become *impossible* rather than *forbidden*; do not let a build session "simplify" the two-feed split back into one.
- **Health "from real signals of real work, never synthetic probes."** The canary-is-the-chain principle (HEALTH-CHECKS) is the right instinct; REL.1/REL.12 attack specific checks' semantics, not the principle.

## 4. COVERAGE STATEMENT

**Examined and found sound (held under this lens):**
- **R0.1 meaning-immutability + versioning.** The `schema_version=v5`-only, reject-don't-COALESCE stance (D-021) is the correct antidote to the historic silent-drift class; nothing to add from a reliability angle.
- **Atomic partition export (D-024).** Temp-write+rename + "orphaned temp dirs swept at every job start" (RECOVERY-RETENTION) genuinely makes partial-export unobservable; I could not construct a partial-read failure that survives the rename semantics (the LIST-consistency edge is REL.9, minor).
- **Same-transaction land+ledger (R5.1).** Server-side crash between land and ledger is correctly prevented; the residual risk is agent-side (REL.3), not here.
- **Determinism as a testable contract (R7).** "Two runs on identical data+config produce identical events" with a determinism test is the right backbone for rebuild-based recovery — *given* the window is defined (REL.2).
- **Staged fleet updates (D-051), never fleet-wide.** Facility-by-facility with heartbeat verification before proceeding is the correct blast-radius control for updates; nothing to add.
- **BACKUP-DR sacred/rebuildable split + two drills.** The classification is honest (it explicitly names agent `pending/` as the only true loss window), and the server-rebuild drill doubling as a determinism check is elegant.
