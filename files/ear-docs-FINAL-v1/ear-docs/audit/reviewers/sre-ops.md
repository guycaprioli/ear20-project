# EAR System — Independent Review · SRE / Operations Lens

**Reviewer question:** Can ONE operator + AI sessions actually *run* this — across multiple facilities, across time, across the failure modes this very design creates?

**Verdict in one line:** The *build* is well-instrumented; the *operate* story is the weakest link, and the specific gap is not any single mechanism but the **unstaffed-on-call reality**: a single human, a single ntfy topic, state-change-only alerting, and DR drills with no owner or cadence. The design is honest about hardware failure and dishonest-by-omission about human availability. Findings below are ranked by risk to *operating* the system, not to building it.

### FINDING-OPS.1 · severity: blocker · target: "one operator + AI sessions" operating model (REVIEW-BRIEF §single-operator; GREENFIELD risk register row 5; D-049)

**steelman.** The design is admirably honest that this is a single-operator system and leans into it everywhere: state-change-only alerting to avoid fatigue (HEALTH-CHECKS-v1:29), `pending/` as a self-draining retry queue so a WAN outage needs *no* human action (RUNBOOK R2-outage playbook: "Agents: nothing to do"), recovery-by-rebuild so there is never a surgical 3am repair (RECOVERY-RETENTION Invariant), and the risk register explicitly names "Single maintainer bus-factor" with the mitigation "Everything in ear-docs; decision log; fixture encodes intent executably." The whole architecture is a deliberate campaign to *minimize* the operator's required availability. That is the correct instinct and most of it is real.

**failure_scenario.** It is a Saturday. A facility's WAN goes down at a site whose L4.5 rules were *not* yet propagated (a mid-rollout state, D-051 is staged facility-by-facility). Over 30 hours, `pending/` on all 12 hosts at that site grows. Two hosts cross the disk ceiling; eviction runs; field data is silently dropped per the ceiling policy. `notify()` fired once at the first ceiling breach (D-049 emitter list). The operator is on a fishing trip with no signal. He returns Monday to a single stale ntfy line and evicted data that is *impossible to reconstruct* — it was the un-uploaded `pending/` (BACKUP-DR row: "Transient — un-uploaded field data!"). For a surveillance system whose entire value is the 10% of traffic that matters, an unbounded-availability assumption on a single human is a data-integrity risk, not merely an inconvenience.

**evidence.** Nowhere in the corpus is there an *availability model* for the human. D-049 gives "one channel, one human, no alerting stack to run" but never states response-time expectations, escalation, or what happens when the human is unreachable. RUNBOOK-v1:32 for R2 outage says "silence alerts for the window" — i.e. the operator is expected to be *awake and present* to silence alerts during an outage, contradicting the "nothing to do" framing. There is no on-call schedule, no second contact, no dead-man's-switch that escalates on operator silence. The plan's calendar (GREENFIELD §5) treats "part-time, single operator" as a scheduling constraint on *building*, never as an availability constraint on *running*. The system creates failure modes (ceiling eviction, silent data loss) whose only backstop is a human who is explicitly modeled as part-time.

**proposed_alternative.** (1) Write an explicit **operating-availability contract** into RUNBOOK-v1: what is the maximum tolerable time-to-acknowledge for each alert class, and what is the operator's committed response window? This is a decision (call it D-062), not a doc detail. (2) Change the disk-ceiling policy from silent eviction to **alert-and-hold** as the default (Q-09/L5-open-item-1 leaves this open — resolve it toward *hold*), sizing `pending/` disk so a realistic worst-case outage-plus-operator-absence (e.g. 72h) never evicts. Eviction of un-reconstructable field data should require the disk to *actually* fill, not a soft ceiling tripping while the operator is unreachable. (3) Add a **repeat-until-acknowledged** escalation for the *data-loss-imminent* alert class only (ceiling-approaching), distinct from the state-change-once discipline that is correct for everything else. (4) Name a **second human contact** for the data-loss and total-outage classes even if that person only forwards to the operator — a bus factor of exactly 1 with no backup contact is the finding the risk register names but does not actually mitigate ("everything in ear-docs" mitigates *knowledge* loss, not *availability* loss; these are different failures).

**downstream_impact.** D-049 (add escalation class), Q-09 (resolve toward hold), Q-10 (disk-ceiling sizing must be derived from the availability window, not just hardware), HOST-PROVISIONING-CHECKLIST (disk ceiling default), RUNBOOK (availability contract + escalation), BACKUP-DR (the "transient loss window" is now bounded by operator-availability, which must be stated).

**confidence.** high

### FINDING-OPS.2 · severity: major · target: D-049 / HEALTH-CHECKS-v1 — a single ntfy topic cannot distinguish a facility-WAN outage from noise, and the correlation logic that would is only advisory

**steelman.** D-049's "one channel, one env var, no alerting stack to run" is exactly right for a single operator — a PagerDuty/Alertmanager deployment would be more infrastructure to run than the system it guards, the definition of operational own-goal. And the design *does* think about the WAN case: HEALTH-CHECKS Layer 3 says "many stale in one facility → **WAN suspicion noted**", and RUNBOOK dead-heartbeat response leads with "Check facility WAN first (multiple agents dark = WAN)." The correlation intelligence exists.

**failure_scenario.** A facility with 12 hosts (48 sensors) loses its WAN. On the next `check_health` sweep, all 48 heartbeats go stale within one 15-minute window. State-change alerting (HEALTH-CHECKS:29) fires `notify()` on "any new red." The question is: does the operator's phone get **one** push ("facility ral: WAN outage suspected, 48 sensors dark") or **48** pushes (one per sensor going red)? The doc says the sweep "produces a verdict per check" and Agent-liveness is *one* check that flags per-sensor ("that sensor flagged") — so the natural implementation emits 48 red sub-verdicts. If notify() fires per red sub-verdict, that is 48 phone pushes for one root cause: alert fatigue in its purest form, and the *next* real single-sensor failure during the outage window is buried. If notify() fires once per sweep regardless, then a WAN outage and a single dead sensor produce an *identical* single push and the operator cannot triage from the notification alone — he must open the dashboard every time, defeating the "phone push" value. The doc says "WAN suspicion noted" but "noted" is a dashboard annotation, not an alert-routing decision.

**evidence.** HEALTH-CHECKS-v1:20 — "stale → **that sensor flagged**; many stale in one facility → **WAN suspicion noted**." The "noted" is passive; there is no rule that says "if ≥N sensors in one facility go stale in one sweep, emit ONE facility-level WAN alert and SUPPRESS the per-sensor alerts." HEALTH-CHECKS:29 defines state-change alerting as "on any new red" — cardinality of "a new red" is undefined (per-check? per-sensor? per-sweep?). D-049's emitter list is a flat list of *conditions*, with no de-duplication or roll-up rule between them. RUNBOOK's "multiple agents dark = WAN" is a *human* diagnostic step performed *after* opening the dashboard, not an *alerting* behavior — the intelligence is on the wrong side of the notification boundary.

**proposed_alternative.** Specify alert **roll-up/aggregation semantics** as a first-class part of D-059, not an afterthought: (1) the sweep computes a **facility-level WAN verdict** *before* emitting per-sensor alerts; if ≥N same-facility sensors are stale in one sweep, emit exactly one `notify()` ("facility X: probable WAN outage, K/M sensors dark") and suppress the individual sensor reds for that facility until the WAN verdict clears. (2) Define the notify() cardinality contract explicitly: at most one push per *root cause* per sweep, with the root-cause grouping being (facility-WAN | facility-R2-auth | single-sensor | pipeline | server-infra). (3) Encode the WAN-vs-single-sensor discrimination as a fixture assertion — P14 already stages heartbeat-fresh/stale (HEALTH-CHECKS:36); extend it with a **"whole-facility stale → exactly one facility-level notify, not N"** persona. This is the single highest-leverage alerting fix and it is nearly free pre-code.

**downstream_impact.** D-059 (add roll-up semantics + fixture assertion), D-049 (notify() gains a de-dup/grouping key), HEALTH-CHECKS-v1 (Layer 3 Agent-liveness check must produce a facility-level verdict, not only per-sensor), fixture P14 (new sweep-cardinality persona), DASHBOARD J1 (the "conclude WAN vs host in ≤4 clicks" journey becomes "the alert already told you," raising the bar).

**confidence.** high

### FINDING-OPS.3 · severity: major · target: BACKUP-DR-v1 drills — specified but with no CADENCE and no OWNER; self-attesting only

**steelman.** BACKUP-DR-v1 is unusually strong for a pre-code doc: it names both drills concretely (Postgres restore to scratch container; full server rebuild with a *determinism check that doubles as restore validation* — a genuinely elegant two-birds design), it declares "WP-O2 exit requires both," and the sacred-vs-rebuildable table is the whole strategy legibly in one place. The determinism-as-restore-validation is the kind of insight most DR plans lack. Backups themselves are automated (nightly timer, notify on failure).

**failure_scenario.** WP-O2 runs once during M5. Both drills pass. The disposition log records it. Eleven months later, a Postgres minor-version bump has landed via a routine `compose pull` (D-051 server updates), and the nightly `pg_dump` format is now subtly incompatible with the older `pg_restore` path baked into the runbook muscle memory. Nobody has run the restore drill since M5 because **no doc says when to run it again**. The dev box's NVMe fails (the single-box reality, D-047). The operator reaches for the restore procedure under pressure — the one time it matters — and discovers the drill has silently rotted. The determinism check that "doubles as restore validation" only validates restore *when someone runs it*, and nothing schedules that.

**evidence.** BACKUP-DR-v1:21 — "**Drills (WP-O2 exit requires both)**" ties both drills to a *one-time build gate*, never to a recurring cadence. There is no "quarterly restore drill" timer, no `OnCalendar=` unit, no line item in any recurring ops loop. Contrast HEALTH-CHECKS-v1:33 which *does* schedule the fixture suite ("weekly timer") and mentions "the BACKUP-DR restore drills (per plan)" — but "per plan" points back to WP-O2, which is a build milestone, not a schedule. The RUNBOOK's recurring cadences are quarantine review (weekly) and confirmed-device logging (per-event); DR drilling appears in *neither*. Ownership: the whole system has exactly one owner, so "who runs the drill" is trivially the operator — but there is no mechanism that *reminds* the sole operator, and an un-reminded recurring task on a part-time single operator (FINDING-OPS.1) is a task that will not happen.

**proposed_alternative.** (1) Add a **DR-drill cadence** as an explicit line: restore drill quarterly, full-rebuild drill semi-annually (or after any Postgres major-version bump — tie it to the D-051 release process as a post-condition: "any release that bumps the postgres image → restore drill before the next release"). (2) Make it a **scheduled, self-nagging** task the same way HealthSweep self-monitors: a `HealthSweep`-style "last successful restore drill" freshness check that turns the dashboard banner amber if the last drill is older than the cadence — reuse the exact self-monitoring pattern D-059 already invented for the watchdog. That closes the "un-reminded task never happens" loop using machinery that already exists. (3) The nightly-backup automation notifies *on failure* (BACKUP-DR:26) — but a backup that *succeeds* while producing a *non-restorable* dump is the more dangerous case; the restore drill is the only thing that catches it, which is exactly why it must recur.

**downstream_impact.** BACKUP-DR-v1 (add cadence + ownership), D-051 (release process gains a "postgres-version-bump → restore drill" post-condition), D-059 (HealthSweep gains a drill-freshness check — reuses existing self-monitoring), RUNBOOK (add DR drill to recurring cadences alongside quarantine review), Q-21 (the second-provider snapshot decision interacts: if declined, the drill matters *more*).

**confidence.** high

### FINDING-OPS.4 · severity: major · target: HEALTH-CHECKS-v1 self-monitoring watchdog — does NOT cover "the whole box is down"

**steelman.** The self-monitoring design is genuinely clever and the doc is self-aware about it: "the sweep is itself watched — if no `HealthSweep` row lands within 2× its cadence, the dashboard banner turns amber... **Watchdog for the watchdog, cheaply**" (HEALTH-CHECKS:31). It avoids the infinite-regress trap (who watches the watcher-watcher) by having the web endpoint report it too. For a single-box dev/deploy this is the right frugality.

**failure_scenario.** The server host loses power, or the NVMe dies, or the whole box hangs (D-047: dev *and* prod may be a single box; Q-11 leaves prod deploy open). Now: Postgres is down, so no `HealthSweep` row can be written. The web `/health/` endpoint is down, so it cannot "report" the silence. The dashboard is down, so the banner cannot turn amber. Every layer of the self-monitoring watchdog **lives inside the thing it is monitoring**. The operator's phone receives *nothing* — state-change alerting fires on *new reds written to the DB*, and a dead DB writes no reds. The failure is indistinguishable from "all quiet." This is the classic dead-man's-switch inversion: the system alerts on *bad news it can generate*, but total failure is the *absence* of the system, which by construction generates no news. The 48-sensor facility whose data was piling up in `pending/` is now piling up against a server that will never come back until a human notices — and the notification channel that would tell the human is part of the dead server.

**evidence.** HEALTH-CHECKS-v1:31 — the entire self-monitoring mechanism is: `HealthSweep` row absence → dashboard banner amber + web `/health/` reports it. All three (HealthSweep write, dashboard, `/health/`) require the server to be *up*. D-049's dead-heartbeat emitter is "checked at the start of each pipeline run + a lightweight host timer" — but the pipeline run and the host timer both run *on the server host* (D-044 ephemeral job; the "lightweight host timer" is a systemd timer *on the same box*). There is no *external* observer. The `OnFailure=` backstop (HEALTH-CHECKS:29) fires when the timer *runs and the command fails* — it does not fire when the *box is off and the timer never runs at all*. Absence-of-signal is invisible to every mechanism described.

**proposed_alternative.** Add one **external liveness beacon** — the only watchdog that must live outside the box: (1) cheapest option — a **dead-man's-switch on a free external service** (e.g. healthchecks.io, an ntfy scheduled reminder, or a Cloudflare Worker cron since R2/Cloudflare is already in the stack): the server pings an external URL every sweep; if the *external* service stops hearing pings for 2× cadence, *it* alerts the operator. This is the standard, near-zero-cost pattern and it is the *only* thing that catches total-box-death. (2) Since R2 is already sacred infrastructure the agents reach independently, an even more aligned option: the agents' heartbeats keep landing in R2 during a *server* outage (agents don't need the server — D-045, RUNBOOK R2-outage: "Agents: nothing to do"). A **tiny external checker** (Cloudflare Worker on a cron, reading R2) that notices "fresh agent heartbeats exist in R2 but no HealthSweep/server-side pull for >X" detects exactly the server-down-while-fleet-alive case the internal watchdog is blind to. Either way: the watchdog-for-the-watchdog must be *off the box*, and the doc's "cheaply" claim should not be used to justify keeping it on-box.

**downstream_impact.** D-059 (add external beacon as a fifth, out-of-band layer), HEALTH-CHECKS-v1 (Layer 3 self-monitoring section must state the total-outage blind spot and its external cover), D-049 (the beacon is a second alert path — acceptable, it is *the* backstop), Q-11 (server topology/hosting choice should ensure the beacon target is independent of the deploy host), interacts with FINDING-OPS.1 (this is the mechanism that would have paged the operator on the fishing trip).

**confidence.** high

### FINDING-OPS.5 · severity: major · target: GREENFIELD §5 "additional facilities repeat WP-O2 only" + RUNBOOK onboarding — the manual per-sensor promotion load does not scale to the stated fleet

**steelman.** The onboarding gate is a genuine safety feature, not bureaucracy: R5.0/D-022's `pending→qa→active` prevents a misconfigured or rogue sensor from auto-contaminating production analysis, and the qa isolation step (RUNBOOK step 3: "noise floor sane vs siblings, event shapes plausible, quarantine near-zero") is exactly the human judgment that catches a bad mount or wrong floor before it poisons the corpus. Making onboarding *deliberately* manual is defensible for a surveillance system where a bad sensor is worse than a slow rollout. And GREENFIELD honestly scopes per-facility repeat to "WP-O2 only."

**failure_scenario.** The design targets "7–15 hosts" per facility (LOCAL-HOST-DATA-FLOW:14) × 4 devices = **28–60 sensors per facility**, across N facilities. Onboarding one facility means, per HOST-PROVISIONING-CHECKLIST, *per host*: record 4 bearings, 4 serials, set 4 noise floors, run kill-WAN test (first host per facility), then per the RUNBOOK: verify each of 4 sensors pending → promote to qa → wait for a pipeline run → review isolation results (noise floor vs siblings, event shapes, quarantine) → promote to active → log in inventory. For a 12-host facility that is ~48 sensors each passing a multi-step, judgment-requiring, pipeline-run-gated promotion, all driven by one part-time human. At 2–4 pipeline runs/day (Q-19), the qa-isolation-review step alone means a sensor cannot go active faster than the next run — a 48-sensor facility is *days* of promotion work even if each review is 5 minutes, because they serialize behind human attention and run cadence. "Repeat WP-O2 only" hides that WP-O2 is O(sensors), and sensors is the number that grows fastest.

**evidence.** GREENFIELD §5 — "Additional facilities repeat WP-O2 only" presents facility scale-out as a *fixed* cost. But WP-O2 content (RUNBOOK onboard + HOST-PROVISIONING-CHECKLIST) is *per-sensor* work: RUNBOOK-v1:2–8 describes a per-sensor pending→qa→active flow with human review at qa. HOST-PROVISIONING is "copy per host" (checklist header) with per-device fields (bearings/serials/floors ×4). Nothing in the corpus offers **batch** operations: no "promote all qa sensors in facility X that passed isolation," no bulk floor-setting, no template-derived bulk provisioning beyond copy-paste. The DASHBOARD J2 journey onboards *one* host and asserts "no admin shell" — good — but demonstrates the single-host path, not the 12-host reality. Q-14 asks "how many hosts each" but no doc converts that answer into an onboarding-time budget or a batch path.

**proposed_alternative.** (1) Add **batch onboarding affordances** to the ops dashboard scope (WP-O1): "promote all qa sensors in facility X whose isolation review is green" as one action; a facility-onboarding checklist *view* that tracks all N sensors' promotion state so nothing is dropped. (2) Make qa-isolation review **assisted, not manual-per-sensor**: the sweep already computes per-sensor condition (noise floor, quarantine rate) — surface an automated "isolation verdict" (floor within X of siblings, quarantine < threshold, events non-empty) that the operator *confirms* rather than *derives*, turning 48 investigations into 48 one-click confirmations plus deep-dives only on outliers. (3) State an explicit **onboarding-time budget** per facility (derived from Q-14 answers) in the plan so "repeat WP-O2 only" is not read as O(1). (4) Consider whether the *first* host per facility needs full manual scrutiny but subsequent identical hosts at the same site can inherit a validated profile (same room type, same floor), reducing per-sensor judgment to per-*host-type* judgment.

**downstream_impact.** GREENFIELD §5 (facility scale-out cost is O(sensors), state it), WP-O1 dashboard scope (batch promote + onboarding tracker), WP-O2 (assisted isolation verdict), RUNBOOK onboard section (batch path), DASHBOARD-DESIGN Onboarding queue screen (bulk actions), Q-14 (answer feeds an onboarding-time budget), interacts with FINDING-OPS.1 (this load lands on the same single human whose availability is already the binding constraint).

**confidence.** high

### FINDING-OPS.6 · severity: major · target: D-051 fleet updates — no rollback procedure, and the staged-rollout leaves facilities on mixed versions with no consistency contract

**steelman.** D-051 is right to stage facility-by-facility and to *verify heartbeat carries the new `agent_version` before the next facility* — that is a real canary discipline, and having `agent_version` in the heartbeat schema (heartbeat.v1:51) makes the verification observable rather than assumed. Git-tag releases + `ear-agent-update <tag>` is boring and uniform, exactly what a single maintainer needs (D-038 rationale). The staged approach means a bad release is caught at facility 1, not fleet-wide.

**failure_scenario.** Operator tags `v1.4.0`, runs `ear-agent-update v1.4.0` on facility `ral`. Heartbeats come back carrying `agent_version: 1.4.0` — the verification D-051 requires **passes**, because the *agent starts and reports*. But `v1.4.0` has a subtle edge-filter regression that mis-classifies a class of FORWARD rows as FILTER — data is being *quietly summarized-away*, and this is invisible in the heartbeat (which reports liveness, not correctness). By the time the next pipeline run's quarantine/event shapes reveal it (RUNBOOK quarantine review is *weekly*), `ral` has lost days of tracker-candidate fidelity. Now the operator wants to roll `ral` back to `v1.3.0`. **There is no documented rollback.** `ear-agent-update <tag>` presumably works with an older tag (pip install into venv), but nothing says so, nothing addresses whether the older agent's config renders against the *current* ear-fleet state, whether in-flight `pending/` files written by the new version replay cleanly through the old, or whether a config-version mismatch results. Meanwhile facility `ral` and un-upgraded facilities are running *different agent versions writing to the same R2 bucket and the same server pipeline* — and no doc states that mixed-version fleets are contract-safe.

**evidence.** D-051 describes the *forward* update path in full but says nothing about **rollback**: no `ear-agent-update <previous-tag>`, no rollback drill, no "revert" verb. The heartbeat verification (D-051: "verify heartbeats carry the new `agent_version`") confirms *the agent restarted*, not *the agent is behaving correctly* — a liveness check masquerading as a release-quality gate. The RUNBOOK Releases section repeats the forward path only. On mixed-version safety: the system *should* be safe because `schema_version=v5` is the wire contract and both versions emit v5 (R0.2, D-052) — but this is *implied*, never *stated as a release invariant*, and "staged facility-by-facility" *guarantees* a mixed-version window on every release, so the safety of that window is load-bearing and undocumented. RUNBOOK Releases:29 — "Any contract-touching release: fixture suite green first" gates the *build*, not the *field behavior* of the deployed mix.

**proposed_alternative.** (1) Document a **rollback procedure** in RUNBOOK Releases: `ear-agent-update <previous-tag>` semantics, config-render compatibility, and what to do with `pending/` files written by the newer version (they are v5, so they replay — but *say so*). Add a rollback drill to WP-O2 exit alongside the DR drills. (2) State the **mixed-version invariant** explicitly as a release rule: "all released agent versions emit `schema_version=v5` and are simultaneously ingestable; the server never assumes fleet-uniform agent versions" — and *test it* (a fixture persona: two agents, two versions, same facility, both ingest correctly). (3) Strengthen the canary from liveness to *behavior*: after upgrading facility 1, before proceeding, check not just `agent_version` freshness but **quarantine rate and FORWARD/FILTER ratio deltas vs the pre-upgrade baseline** for that facility — the sweep already has the data; a behavioral canary catches the silent-mis-classify case the version-string check misses. (4) Symmetric gap on the server side: D-051 server update is "GHCR tag bump + compose pull && up -d" with no rollback and no statement of what happens if the new image's *pipeline* misbehaves after the *web* is already serving on it.

**downstream_impact.** D-051 (add rollback + mixed-version invariant + behavioral canary), RUNBOOK Releases (rollback procedure), WP-O2 (rollback drill), fixture (mixed-version-ingest persona), D-052/R0.2 (the v5-uniformity that makes rollback safe becomes a *stated* release invariant, not an implicit one), interacts with FINDING-OPS.3 (rollback is a drill like DR — needs the same cadence/ownership treatment).

**confidence.** high

### FINDING-OPS.7 · severity: major · target: single-operator 6-month-gap survivability — the decision log captures WHY but not the operational RUNNING STATE; cold-return has no restart procedure

**steelman.** The corpus is *extraordinarily* disciplined about capturing intent: DECISION-LOG-v1 records every decision with rationale and a "reopening requires new evidence, never silence" rule (D-log header); the fixture "encodes intent executably" (risk register); contracts are the law with named enforcement per rule (R0.3). ### FINDING-OPS.10 · severity: minor · target: HEALTH-CHECKS Layer 3 "capture advancing" + "handoff advancing" — "expected window" per active sensor is undefined and per-band variable, risking false reds or missed misses

**steelman.** The "capture advancing" check is a genuinely strong idea and correctly distinguishes host-down from RF-quiet: "new obs_/ctx_ objects per active sensor within expected window ... none arriving while heartbeat fresh = capture problem (distinct from host-down)" (HEALTH-CHECKS:22). This catches the insidious case where the agent is *alive and heartbeating* but its scanner has silently stopped producing — a failure a naive liveness check misses entirely. The whole "health from real signals of real work" principle (HEALTH-CHECKS:4) is the right philosophy.

**failure_scenario.** "Expected window" is the load-bearing undefined term. A high-band sensor (b/d) in an RF-quiet area might legitimately produce *few or zero* FORWARD `obs_` objects for long stretches — the target device is a *power-saving tracker that wakes rarely* (D-028 low-duty-cycle exception; persona P15 "deep-sleep survival"). If "capture advancing" expects `obs_` objects "within expected window" and a genuinely-quiet sensor produces none because *there is nothing to forward* (only `ctx_` heartbeats), the check either (a) false-reds a healthy quiet sensor — training the operator to ignore that red (fatigue), or (b) is tuned loose enough to accept quiet, in which case a *genuinely broken* scanner producing nothing is *also* accepted — a miss. The `obs_` feed is FORWARD-only (D-012) and FORWARD volume is *exactly* the thing that's near-zero for the primary target's quiet periods. Basing "capture advancing" on `obs_` arrival conflates "sensor working" with "sensor seeing signal," which the entire feed-split architecture (D-012) was designed to *separate*.

**evidence.** HEALTH-CHECKS:22 — "new `obs_`/`ctx_` objects per active sensor within expected window" — "expected window" is never quantified and is inherently per-band/per-environment variable (a quiet racked-area high-band sensor vs a busy open-warehouse low-band one). The check lumps `obs_` and `ctx_` together, but `ctx_` (heartbeats, summaries, config stamps) arrives *every uploader tick regardless of signal* while `obs_` arrives *only when there's FORWARD-worthy signal* — these have fundamentally different expected cadences. D-028's low-duty-cycle exception and P15 (deep-sleep) establish that near-zero FORWARD volume is *expected and correct* for the primary target, directly colliding with an `obs_`-arrival-based liveness check. PARAMETER-TUNING notes per-facility RF environments differ (§Per-facility awareness), so a single "expected window" cannot be global.

**proposed_alternative.** (1) Base "capture advancing" on the **`ctx_` feed** (which arrives every tick by construction — heartbeat + config stamp ride it, D-012/L4) for the *liveness* judgment, and treat `obs_` volume as a *separate, informational* signal ("this sensor is quiet") that is baseline-relative, never a hard red. A sensor is "advancing" if its `ctx_`/heartbeat is fresh; whether it's *seeing signal* is a different question the operator interprets against the sensor's own history, not a fleet-global window. (2) Make "expected window" **per-sensor, derived from that sensor's own recent cadence** (self-baselining) rather than a global constant — the sweep has the history. (3) Explicitly reconcile this check with D-028/P15 in the doc: a healthy sensor can have zero FORWARD output; state that zero-`obs_` is not a red as long as `ctx_` advances and the noise floor is sane.

**downstream_impact.** HEALTH-CHECKS-v1 Layer 3 (split obs_ liveness from signal-volume; define per-sensor window), fixture P14/P15 (a persona: quiet-but-healthy sensor → not red), D-028/P15 alignment (quiet is correct for the primary target), PARAMETER-TUNING (per-facility window baselines), DASHBOARD Sensors board ("last data" column semantics — is it last `obs_` or last `ctx_`? they differ, and the operator's worst-first sort depends on it).

**confidence.** med

### FINDING-OPS.13 · severity: minor · target: Q-19 pipeline cadence OPEN — dashboard freshness, alert timeliness, and "expected window" health checks all silently depend on an unset number

**steelman.** Deferring pipeline cadence (Q-19, D-044) is defensible: forensic detection is inherently a batch/periodic workload, not real-time, and "2–4×/day is plenty" (Q-19) is a reasonable provisional that the runsheet lets S14 proceed on as a DEFAULT. ### FINDING-OPS.14 · severity: minor · target: HOST-PROVISIONING-CHECKLIST kill-WAN test scoped "first host per facility at minimum" — the store-and-forward safety net is only proven once per site

During a real WAN outage it hits the disk ceiling and evicts (FINDING-OPS.1) while host #1 (larger disk, tested) rides it out fine. The operator's mental model — "we tested SAF at this facility" — is now wrong for the host that actually failed.

**evidence.** HOST-PROVISIONING-CHECKLIST:28 — "Kill-WAN 10-min test ... *(first host per facility at minimum)*." The 10-min test also can't exercise the *ceiling* (a 10-minute outage won't fill any reasonable disk) — it proves "buffers and drains," not "evicts correctly and alerts when full," which is the *actually dangerous* path (FINDING-OPS.1). Disk-ceiling default is set per Q-10/host, and hosts can differ. The checklist records per-host disk (`disk ≥ ___ GB free`) but doesn't tie the kill-WAN coverage to disk-size equivalence.

**proposed_alternative.** (1) Run the kill-WAN buffer/drain test on **every host** (it's 10 minutes and catches per-host config/NTP/disk-mount issues) — or, if keeping the "first host" optimization, scope it to "first host *of each hardware/disk profile* per facility," so a smaller-disk host gets its own test. (2) Separately, add a **ceiling-behavior test** (not a WAN-duration test) to the checklist that *forces* the disk-ceiling path in a controlled way (e.g. pre-fill to near-ceiling and verify eviction-or-hold + alert fires) — because the 10-minute drill proves buffering but never proves the ceiling behavior, which is the path with data-loss consequences. (3) Record in ear-fleet which hosts have had *which* drill, so coverage is legible (feeds FINDING-OPS.7's fleet-state artifact).

**downstream_impact.** HOST-PROVISIONING-CHECKLIST (per-profile kill-WAN; add ceiling-behavior test), FINDING-OPS.1 (ceiling behavior is the data-loss path — testing it is part of resolving that finding), Q-10 (per-host disk variance), ear-fleet (per-host drill-coverage record → fleet-state artifact, FINDING-OPS.7).

**confidence.** low

### FINDING-OPS.15 · severity: minor · target: D-058 verification protocol "Guy re-runs independently" per session — an unbudgeted, serializing operator-time cost across 21+ sessions, and a single-point human bottleneck on the whole build

**steelman.** D-058's "agent claims are not evidence; reports are" + "Guy re-runs independently" is *excellent* engineering discipline — it's the antidote to the "Opus sessions drift from contracts" risk (risk register) and to AI over-claiming (the exact failure mode the REVIEW-BRIEF says the author can't self-detect). "Thirty seconds of trust-but-verify per session" (BUILD-RUNSHEET:43) is framed as cheap, and the five human-oracle points (S04/S18/S19d/S19e/S20) are genuinely un-delegable field-truth judgments.

**failure_scenario.** The verification cost is framed as "30 seconds per session" but the *real* operator-time cost of the build is much larger and *fully serialized through one human*: every one of 21+ sessions requires Guy to (a) paste the bootstrap, (b) potentially answer ASK-gated questions *before* the session can run (S15 ASK Q-03, S18 ASK Q-13, S21 ASK Q-04/14/15/21), (c) re-run the gate in a fresh shell, and (d) for five sessions, exercise deep human-oracle judgment (S18 corpus-vs-known-truth is *not* 30 seconds — it's reviewing whether the event dataset matches his field knowledge of 20595). No session can *complete* without the operator, and the operator is *part-time* (GREENFIELD §5). The M0–M5 calendar ("5–7 weeks part-time") assumes this human gating never becomes the bottleneck — but with a strictly-serial "no session passes until Guy re-runs," the *operator's availability*, not Opus throughput, sets the build pace. A 6-month operator gap (FINDING-OPS.7) doesn't just pause *running* — it pauses *building*, because nothing merges without the human re-run. The build model has the same bus-factor-of-1 as the run model, and D-058 *tightens* it by design.

**evidence.** D-058 / BUILD-RUNSHEET:40–45 — "A session is marked complete only after Guy runs the same gate command." Every session close depends on the operator. The runsheet's ASK gates (S15/S18/S21) block sessions on operator answers. The five human-oracle points (S04/S18/S19d/S19e/S20) are explicitly "cannot be delegated." Q-24 ("is the cumulative spend budgeted") acknowledges the *API* cost but *not* the *operator-time* cost, which is the scarcer resource for a part-time single operator. The "5–7 weeks part-time" calendar makes no allowance for operator-availability as the rate-limiter.

**proposed_alternative.** (1) Acknowledge in the plan that **operator availability, not Opus throughput, is the build's rate-limiter** — and budget it (a companion to Q-24: "operator hours per week available for build gating"). This doesn't change the *discipline* (which is right) but makes the calendar honest. (2) Batch the human-oracle points: S18/S19d/S19e are all "Guy reviews classification/corpus sanity" — cluster them so the operator's deep-judgment sessions are *scheduled* blocks, not scattered blockers. (3) For the *non-oracle* re-runs (most sessions), consider whether the "committed gate report + regression floor in CI" (D-058 items 2+5) is *sufficient* evidence to let a session merge, with Guy's independent re-run as an *asynchronous audit* rather than a *synchronous blocker* — preserving the trust-but-verify without making every merge wait on one human. The five true-oracle points stay synchronous; the rote re-runs need not. (4) State the build's own bus-factor: if the operator is unavailable, the build *stops* (unlike the running system, which the whole design keeps alive without him) — a candid note that the two bus-factors differ.

**downstream_impact.** D-058 (distinguish oracle-gates from rote re-runs; async-audit option), BUILD-RUNSHEET (batch oracle points; state operator-availability as rate-limiter), GREENFIELD §5 calendar (honest operator-hours budget), Q-24 (extend to operator-time, not just API spend), FINDING-OPS.7 (build bus-factor = run bus-factor, both = 1).

**confidence.** low

## 3. WHAT TO PROTECT

These choices are strong enough that "improvements" during build would likely make ops *worse* — guard them.

- **Recovery-by-rebuild / per-agent disposability (D-023/D-024, RECOVERY-RETENTION Invariant "no recovery procedure ever edits data in place").** This is the single best operational decision in the corpus. It converts every recovery into "delete + rebuild" or "restore + re-run" — no surgical, error-prone, 3am in-place repair ever exists. For a single operator this is worth more than any amount of monitoring. Do not let a build session "optimize" by adding an incremental/in-place recovery path; the disposability *is* the safety.

- **"Health from real signals of real work, never synthetic probes" (HEALTH-CHECKS:4).** Injecting synthetic records into production data to test liveness is a classic ops anti-pattern (synthetic data contaminating the real corpus). Judging health from actual heartbeats/object-arrivals/run-outcomes is correct and should be defended against any "let's add a synthetic canary transaction" proposal.

- **`pending/` as the retry queue by construction; circuit-breaker retired (D-045, L5 v2 reframe).** Deleting the file-watcher and circuit-breaker (whole bug classes gone) in favor of "failed run exits, timer retries, cadence = backoff" is a genuine simplification that removes moving parts a single operator would otherwise have to reason about. Guard against a build session "restoring" the circuit breaker for perceived robustness — it was correctly removed.

- **State-change-only alerting with a periodic reminder (HEALTH-CHECKS:29).** The core instinct — don't re-alert on known-persisting failures — is exactly right for a one-human channel and directly fights fatigue. FINDING-OPS.2/OPS.11 refine *cardinality and suppression* around it, but the *state-change* core must survive; do not let anyone revert to level-triggered spam.

- **Feed split making the false-CRITICAL bug structurally unrepresentable (D-012).** Operationally this means a whole class of alert (false tracker from a summary row) can *never* fire — structural prevention beats any runbook. Protect it; the `obs_`/`ctx_` physical separation is load-bearing for alert trustworthiness.

- **Onboarding gate pending→qa→active (D-022/R5.0).** Even though FINDING-OPS.5 attacks its *scalability*, the *gate itself* is correct and must survive — a surveillance system auto-ingesting an unvetted sensor is worse than a slow rollout. Batch it (OPS.5), don't remove it.

- **sops+age over a secrets service (D-050).** For one operator, not running a Vault is the right call. FINDING-OPS.7/OPS.12 add *custody and rotation practice* around it, but the *choice of tool* is correctly frugal — protect it against "let's stand up Vault" churn.

- **The two DR drills as concrete, and determinism-check-doubles-as-restore-validation (BACKUP-DR:21–23).** The drills are well-designed; FINDING-OPS.3 only adds *cadence/ownership*. The determinism-as-restore-validation is genuinely clever — protect it.

## 4. COVERAGE STATEMENT

I examined the full operational surface enumerated in §1. Where I found choices sound, why they held:

- **Container liveness (Layer 1, HEALTH-CHECKS:6).** Standard compose healthchecks + Docker restart + restart-loop surfacing in Layer 3 — conventional and correct; no finding. Restart-looping being caught one layer up is the right escalation.

- **`ear-agent check` self-check (Layer 2, HEALTH-CHECKS:9).** The on-host checklist (scanner units ×4, serials present, NTP, disk, queue/DLQ, R2 probe, config render) is a solid provisioning/diagnosis tool. Sound as a *manual/on-demand* tool; its *scheduled* value is limited (it needs SSH or a timer on a possibly-dark host) but that's what Layer 3's remote sweep is for — the layering is coherent. No finding.

- **Heartbeat schema (heartbeat.v1.schema.json).** Well-chosen required fields; `additionalProperties:false` prevents silent drift; `ntp_offset_ms` (D-057) gives continuous clock-drift visibility, correctly seeding the WP-O3 time-sync work. `queue_depth` + `disk_pct` riding the heartbeat make backlog growth visible per tick — good. `last_error` optional is a reasonable minimal error surface. No finding; this is a clean contract. (Note: the *interpretation* of heartbeat freshness is where FINDING-OPS.2/OPS.10 land, not the schema itself.)

- **Retention lifecycle (RECOVERY-RETENTION table).** Enforced by lifecycle policy + job-start pruning, never ad-hoc; "nothing feeding an unreviewed quarantine or unreplayed DLQ is ever purged" is exactly the right invariant; poison DLQ never auto-deleted (Q-09) is correct. Sound — the only gap is L2's tables (FINDING-OPS.9), not the L1 discipline.

- **R2-outage playbook agent side (RUNBOOK:32).** "Agents: nothing to do — pending/ buffers, ceiling protects" is correct given SAF is load-bearing; the *server*-side "silence alerts" is where the gap is (FINDING-OPS.11), not the agent behavior.

- **L4.5 rule-change procedure (RUNBOOK:22).** Add rule in ear-fleet → commit → re-render → applies next uploader tick (D-045), with verification (signal volume drops, ctx heartbeats appear) — clean, versioned, observable. The forced-summary-never-silent-drop discipline (D-013/L4.5) means a mis-scoped rule *degrades gracefully* (still emits summary) rather than losing data — genuinely well-designed. No finding.

- **Quarantine review cadence (RUNBOOK:19).** Weekly, grouped by reason×agent, spike-attribution logic (one agent = host problem; fleet-wide-after-release = the release), never-delete-without-disposition — this is a *model* recurring-ops procedure and is exactly what the DR drill (FINDING-OPS.3) and unclassified-queue triage (FINDING-OPS.9) should imitate. Protected by inclusion in the recurring set.

- **Confirmed-device corpus logging (RUNBOOK:25, PARAMETER-TUNING).** "Always append, no exceptions, including negatives" as the ground-truth for future tuning that "cannot be reconstructed later" is the right discipline; making it a day-one operational habit (not a deferred feature) is correct. No finding — this is well-placed.

- **Backup automation (BACKUP-DR:26).** Nightly pg_dump → local + R2 → prune → notify-on-failure is standard and correct; the gap is *restore-drill cadence* (FINDING-OPS.3), not the backup itself. Off-box nightly (to R2) correctly addresses the single-box risk for Postgres specifically.

- **Container parity dev=prod (D-047).** Same images/compose, MinIO as the sole substitution via one env var — this eliminates the "works in dev, breaks in prod" class and is the right call; the SPOF concern (FINDING-OPS.8) is about the *box*, not the *parity*, which is sound.

Areas I deliberately did **not** deep-dive (out of lens or owned by another reviewer, flagged for handoff): the *legal/privacy* gate (Q-23 — a you-decision, correctly flagged, gates deployment harder than any ops risk); *security depth* (access control, audit, R2 blast-radius internals — Security reviewer owns; I raised only the *operational* rotation/custody/revocation procedures in OPS.7/OPS.12); the *RF/methodology* correctness of what the pipeline computes; the *data-engineering* scale behavior of the pipeline internals (though FINDING-OPS.13 touches cadence-coupling and Q-22's deferred fleet-load-gate is an ops-adjacent risk I note here: deferring the fleet-scale load gate until "before a facility depends on run cadence" is *acceptable* only if that gate is actually run before first-facility go-live and not silently skipped — it should be an explicit S21+ blocker, not a someday item).

**Top 5 risks to operating this system, ranked:**
1. **Unstaffed on-call reality** (OPS.1) — single part-time human is the sole backstop for data-loss-imminent failures with no escalation/second-contact.
2. **Total-box-death is invisible** (OPS.4) — the watchdog lives inside the thing it watches; no external beacon.
3. **DR/rollback drills never recur** (OPS.3/OPS.6) — proven once at build, then rot silently until the one time they matter.
4. **Alert cardinality undefined** (OPS.2/OPS.11) — WAN-outage vs single-sensor indistinguishable at the notification; no maintenance-window suppression.
5. **6-month-gap state reconstruction** (OPS.7) — decision log preserves *why*, not *current running state* or *secret-key custody*; cold-return has no runbook.

**Change before repo creation (structural, D-030-class):** the server-topology / SPOF decision (Q-11/D-004, FINDING-OPS.8) — whether prod shares the dev box materially changes the operational blast radius and is cheapest to decide now. Everything else in my findings is a doc/decision addition that can land during build (ideally folded via the D-060 disposition process before the ops WPs S14/S19/S21+ execute).
