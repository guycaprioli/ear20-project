### FINDING-RF.1 · severity: blocker · target: uplink-only premise vs band plan (methodology §17; LOCAL-HOST L0 line 45; Q-13)

**steelman.** The entire event model rests on a clean physical story: the HackRF sweeps an *uplink* range, so it hears devices transmitting *to* towers, and gaps inside a transmission are the sweep looking elsewhere, not signal absence (methodology §17). D-001/L0 fix the band groups (a/c low 662–850, b/d high 1709–1911) so an EARFCN maps deterministically to at most two same-band radios; R8 forbids cross-band correlation "physically impossible to share an EARFCN." The design has clearly been operated for ~6 months against these ranges, so the ranges "work" in the sense that real trackers (20595 class) were found in them. The band plan is presented as a confirm-only item (Q-13), not an open design question — the steelman is "this is field-validated, leave it."

**failure_scenario.** The high band 1709–1911 MHz is **not an uplink-only range**. In the 3GPP FDD plan: LTE Band 3 uplink is 1710–1785, Band 3 *downlink* is 1805–1880; Band 2 uplink is 1850–1910; Band 2 *downlink* is 1930–1990; Band 1 downlink is 2110–2170. The stated window 1709–1911 therefore spans **Band 3 uplink (1710–1785), a guard region, Band 3 downlink (1805–1880), and Band 2 uplink (1850–1910)** — i.e. it captures tower→device downlink in the 1805–1880 segment mixed with device→tower uplink elsewhere. A tower's downlink beacon is *continuous, high-power, present-every-hour* — exactly the G4 infrastructure signature the noise filter is tuned to strip. But if the sweep is catching downlink, then "gaps within a transmission are the sweep looking elsewhere, not signal absence" (the load-bearing undersampling claim) is **false for the downlink portion**: a downlink carrier really is continuous, and the undersampling correction (sweep-collapse reconstructing "one ~10s transmission") is modeling a physics that does not apply. The low band 662–850 is similarly mixed: LTE Band 5 uplink is 824–849 (downlink 869–894), Band 12/13/14/17 uplink live around 698–716, Band 71 uplink 663–698 with downlink 617–652 — so 662–850 spans several uplink sub-bands *and* the 758–803 downlink block (Band 14/13/29 downlink region). The premise "captures only the uplink channel" (methodology §17) is asserted as physical fact but the chosen ranges do not enforce it.

**evidence.** methodology line 17: "The HackRF captures only the **uplink** channel — one side of the two-way device↔tower conversation." LOCAL-HOST L0 line 45 / line 24: "a, c = low band (662–850 MHz); b, d = high band (1709–1911 MHz)." R8 line 218: "never across band groups (physically impossible to share an EARFCN)." **What is absent:** any statement of *which LTE operating bands / EARFCN ranges* these MHz windows correspond to, any confirmation the windows are uplink-only, and any handling of the possibility that part of each window is downlink. `frequency→EARFCN` mapping (L1 line 91) is asserted but the EARFCN table is never shown; an EARFCN in a downlink sub-band is a downlink EARFCN. The doc even hints the numbers are shaky: line 45 says "A given EARFCN appears on at most the two same-band devices of a host" — true for uplink, but if a downlink EARFCN is being mapped, the whole `(facility, band_group, earfcn)` identity key is mislabeled.

**proposed_alternative.** Before repo creation, add to the methodology a **band-plan appendix** that lists, for each MHz window, the exact 3GPP operating bands and their uplink/downlink sub-ranges, and either (a) narrows the sweep ranges to *uplink-only* sub-bands (e.g. low band to the actual uplink blocks, dropping 758–803 downlink; high band to 1710–1785 + 1850–1910, dropping the 1805–1880 downlink block), or (b) if downlink is deliberately included, **rewrites the undersampling premise** to say "predominantly uplink, but the high band includes downlink carriers which are continuous, not bursty" and adds a discriminator so downlink carriers are classified as G4 by *frequency identity* (they sit in known downlink EARFCNs) rather than relying on the presence×activity filter to guess. Make Q-13 a design question, not a confirm. Tie the frequency→EARFCN mapping table into `contracts/schemas/` so the uplink/downlink status of every EARFCN is machine-checkable.

**downstream_impact.** Touches the entire event model (undersampling → sweep-collapse → 600s burst → events), D-026, the noise filter D-028 (downlink carriers change the infrastructure baseline), the taxonomy G4/G6 (backhaul/infra classes), R8 correlation identity, the HANDOFF `earfcn`/`band_group` fields, and P3/P4 fixture personas (infrastructure chatterer). If the ranges are downlink-contaminated, `avg_power_dbm` semantics and the "stationary vs moving" power-variance inference (taxonomy) also shift.

**confidence.** med — the 3GPP band boundaries are standard and the MHz windows demonstrably straddle uplink and downlink; what I cannot verify from the docs is whether Guy's sweep config already restricts to uplink EARFCNs *inside* these windows (the EARFCN table is never shown). That uncertainty is itself the finding: the premise is stated as physics but the evidence for it is absent from the contract set.

### FINDING-RF.2 · severity: blocker · target: sweep-collapse window 2s vs actual revisit time (methodology §22; Q-06)

**steelman.** 2s is not invented — it is the value earnest-v3 actually used in production (methodology line 159, "earnest-v3 used 2s"), so it is field-derived, not a guess. The doc is honest that it is provisional ("confirm target value, likely tied to sweep revisit time," line 22) and exposes it as a parameter (RFANALYSIS-SCOPE line 66, PARAMETER-TUNING). Q-06 explicitly asks for the real revisit time. So the steelman is: a validated default with a flagged confirm and a tunable seam is exactly the right posture pre-code.

**failure_scenario.** The sweep-collapse window is the boundary that decides "these N sweep-hits are *one transmission*" vs "these are *separate transmissions*." It is physically meaningful **only if it is tied to the sweep revisit time** — the time for the HackRF to sweep its whole band and return to a given EARFCN. That revisit time is a *derived* quantity: `revisit ≈ (band_span / bin_width) × dwell_per_bin`, and it **differs between the low band (188 MHz span, 662–850) and the high band (202 MHz span, 1709–1911)**, and it changes with `bin_width` (a per-agent capture-config field, LOCAL-HOST L0.5). A single hardcoded 2s default therefore cannot be correct for both bands simultaneously, and cannot be correct across two agents with different `bin_width`. Concrete failure: agent `ral-rf1a` (low band, wide bins) has revisit ~1.5s; `ral-rf1b` (high band, narrow bins) has revisit ~4s. With a global 2s window, the high-band agent's *single* 10s transmission (revisited every 4s → hits at t=0,4,8) gets **split into 3 "transmissions"** because consecutive hits are >2s apart — inflating `detection_count`, corrupting `consolidation_confidence`, and (worse) potentially manufacturing a false periodicity at the sweep-revisit cadence downstream in L2 CV analysis. This is the sweep-cadence-as-signal failure mode, one level down from the R2 false-CRITICAL bug the feed split fixes.

**evidence.** methodology line 22: "the consolidation window; earnest-v3 used 2s — confirm target value, likely tied to sweep revisit time." Line 159 marks it "confirm target value (tie to sweep revisit)." Q-06: "is 2 s right, or should it derive from the actual sweep revisit time? What IS the sweep revisit time per band with 4 devices?" **What is absent:** no formula tying the window to `band_span`, `bin_width`, and dwell; no per-band default (one number for two physically different sweeps); no fixture persona that varies revisit time and asserts sweep-collapse stays correct (P1 fixes bursts "captured as 4 sweep hits ~2s apart" — it *bakes in* the 2s assumption rather than testing sensitivity to it). The band-dependence is acknowledged in the question ("per band") but not in the parameter model (RFANALYSIS-SCOPE line 66 lists a single "sweep-collapse window").

**proposed_alternative.** Make the sweep-collapse window **derived, not a flat default**: `collapse_window = k × revisit_time(band, bin_width)` with `k≈2–3` (allow one missed revisit), and compute `revisit_time` per agent from its capture config. Store the *derived* value in the run config snapshot so it is auditable. If a flat default must ship first, make it **per-band** (`collapse_window_low`, `collapse_window_high`) not global. Add a fixture persona: same transmission, two agents with different revisit times, assert both collapse to one detection. Extend Q-06 to demand the actual `hackrf_sweep` parameters (band span, `-w` bin width, sample count) for each band so revisit can be computed, not guessed.

**downstream_impact.** D-026 (sweep-collapse ≠ burst-grouping — the "seconds scale" is exactly this window), `detection_count` and `consolidation_confidence` in HANDOFF, P1/P15 fixture derivations (which encode 2s), the L2 CV/periodicity computation (a mis-set window injects a spurious period), and PARAMETER-TUNING's cross-sensor-agreement proxy (two same-band sensors with different bin_width would systematically disagree if the window is not per-agent).

**confidence.** high — the band-dependence and bin_width-dependence of revisit time is basic sweep physics and the docs themselves flag the tie without wiring it in. The exact numbers are illustrative; the structural gap (one global window for two physically different sweeps) is certain.

### FINDING-RF.3 · severity: major · target: 600s burst gap vs G1 deep-sleep sub-pattern (D-026; methodology §26; taxonomy G1; P15)

**steelman.** 600s is called out as *validated for the primary use case* and explicitly *not universal* (methodology line 27), exposed as a tunable analysis parameter with per-use-case override (line 28), and protected from becoming a hardcoded SQL literal. D-026 records it as "validated, tunable, never hardcoded." The doc even anticipates the exact objection: "a device whose normal cadence is minutes-apart could be wrongly merged, a chattier one over-split." And PARAMETER-TUNING treats it as an A/B target. So the design is honest that 600s is use-case-tuned. The steelman is strong: it is a default, not a law.

**failure_scenario.** The problem is not that 600s is a default — it is that **the same 600s is applied across three G1 sub-patterns whose natural cadences differ by three orders of magnitude**, and the burst-gap threshold *defines the event*, so a wrong gap doesn't just mistune a score — it changes *what gets handed to Level 2 at all*. Walk the deep-sleep case (P15, the primary target's hardest case): a 20595-class deep-sleep tracker wakes every "multi-hour to multi-day" interval (taxonomy line 13). With a 600s gap, **every single wake is its own event** — correct. But now consider the *interval reporter* sub-pattern at, say, a 30-minute cadence (Q-T1's candidate value): 30 min = 1800s > 600s, so again every report is its own event — fine. Now the *motion-activated* sub-pattern: "bursts cluster when the tagged asset moves — gaps regular within active periods, silent otherwise" (line 13). Within an active driving period a motion tracker may report every 60–120s; between trips it is silent for hours. With a 600s gap, the reports *within a trip* (120s apart) correctly merge into one "trip event," but if the tracker's in-motion cadence is >600s (e.g. a power-saving motion tracker reporting every 15 min while moving), **each in-motion report becomes its own event and the "trip" is shattered into singletons** — destroying the very burst structure that distinguishes motion-activated from interval. Conversely a chatty G2 active tracker at 5-min (300s) cadence has *all* its reports merged into one gigantic multi-hour "event," collapsing its periodicity to nothing. The single 600s value cannot be right for a class whose members legitimately span 60s to multi-day inter-report gaps.

**evidence.** taxonomy line 13 enumerates three sub-patterns with wildly different cadences (fixed 30/60min; multi-hour to multi-day; motion-clustered) all under G1, all handed the same 600s gap. methodology line 27: "a device whose normal cadence is minutes-apart could be wrongly merged, a chattier one over-split" — the doc *names* the failure but does not resolve it beyond "it's tunable." Q-T1 is still open: "Real wake intervals you've observed per sub-pattern (30 min? hourly? the 20595 cadence?)" — meaning the actual cadences that would tell you whether 600s splits or merges each sub-pattern **are not yet known**, yet 600s is marked SETTLED (methodology line 158). **What is absent:** any per-sub-pattern gap policy, and any acknowledgment that "event = burst group" is only well-defined *relative to a device's own cadence* — a fixed global gap cannot honor a class with 3-order-of-magnitude cadence spread.

**proposed_alternative.** Downgrade 600s from SETTLED to *validated-default-pending-Q-T1*. More substantively: the burst gap is arguably the **wrong shape of parameter** for a device population this heterogeneous. Consider an **adaptive / per-stream gap**: sessionize using a gap that scales with the stream's own observed inter-detection spacing (e.g. break a burst when a gap exceeds `f × median_intra_stream_gap`), which naturally separates "within-trip" from "between-trip" without a magic global constant. If a global constant must ship, at minimum split it per interval-class once Q-T1 answers, and add a fixture persona for the *motion-activated* sub-pattern (P1/P15 cover deep-sleep and a repeating 5-min-window burster, but no persona exercises the >600s-in-motion shatter case). Note this interacts with FINDING-RF.7: since the event *is* the L2 analysis unit, a mis-set gap can manufacture or destroy periodicity before L2 ever runs.

**downstream_impact.** D-026, HANDOFF `events`/`event_gaps` (gap semantics), taxonomy G1 sub-pattern discrimination, L2 CV/periodicity and four-factor scoring (PERIODIC factor), P15 and a missing motion-activated persona, PARAMETER-TUNING (600s A/B). Directly bears on Q-T1.

**confidence.** high — the contradiction between "three sub-patterns spanning 60s–multi-day" and "one 600s gap marked SETTLED" is on the page, and Q-T1 confirms the cadences needed to defend 600s are not yet known.

### FINDING-RF.4 · severity: major · target: presence_ratio 0.95 vs 0.8, and min-events, across docs (D-028; methodology §105/127; PARAMETER-TUNING line 42/43)

**steelman.** Two docs disagreeing on a tunable is not fatal for a tunable — the whole point of externalizing these into `rfanalysis.toml` (RFANALYSIS-SCOPE line 66) is that the number can move. And the methodology is explicitly the authoritative contract for the filter (D-028 points at it), while PARAMETER-TUNING is a "forward-looking requirement… not yet built" (its own header) — so one could argue the methodology's 0.95 wins and PARAMETER-TUNING's 0.8 is just a stale illustrative figure. D-052 even establishes that schemas are the tie-breaker over prose. Steelman: the hierarchy resolves it.

**failure_scenario.** The `presence_ratio` is not a cosmetic knob — it sets `infra_hours = ceil(effective_hours × presence_ratio)`, the threshold above which a device is called "present nearly always" and routed toward the infrastructure/low-duty branch (methodology line 105). At **0.95**, over a 24h window `infra_hours = ceil(24×0.95) = 23` — a device must appear in 23 of 24 hours to trip presence. At **0.8**, `infra_hours = ceil(24×0.8) = 20` — appear in 20 of 24 hours. Now the primary target: a G1 *interval reporter* beaconing hourly appears in ~24/24 hours → trips presence at *either* threshold → falls into the low-duty-cycle exception (good). But a G1 *interval reporter* on a **90-minute cadence** appears in only ~16 of 24 hours (a beacon every 90 min touches 16 distinct hour buckets). At 0.8 (infra_hours=20) it does **not** trip presence, so it never reaches the low-duty exception path — it is evaluated only by `total activity < MIN_OBSERVATIONS`, and with ~16 short bursts it may pass as signal *or* be dropped as noise depending on MIN_OBSERVATIONS. The exact same device under the two documented values takes two different filter paths. Because the fixture *defines truth* (D-057) and P2 hard-codes "present in ~100% of hours," the fixture will happily pass at 0.95 while a real 90-min tracker silently behaves differently — and a build engineer reading PARAMETER-TUNING would set 0.8. This is precisely the "silent contract violation via a number" class the whole R0 discipline exists to prevent.

**evidence.** methodology line 105 & table line 127: `presence_ratio` default **0.95**. PARAMETER-TUNING line 42: "Presence / infrastructure ratio | Analysis/filter | **~0.8** | Noise filtering." That is a 15-point disagreement on a threshold that decides whether the *primary target* enters the exception path. Separately: PARAMETER-TUNING line 43 lists "Min events for analysis | Analysis | **3** | Below this, no meaningful CV," while P15 (fixture) has a deep-sleep device with **4 bursts across 10 days** and demands "events built (min-events rules must not erase it)." Four events is *one* above the min-of-3 floor — a deep-sleep tracker that produced 3 wakes in the window (entirely plausible for a multi-day cadence) sits **exactly at the erase boundary**, and the two docs don't agree on whether min-events lives in Level 1 (where it could drop the stream before handoff) or Level 2 (analysis-only). **What is absent:** a single authoritative parameter table; the methodology's own tunable table (line 124) omits presence_ratio's alternate value and omits min-events entirely.

**proposed_alternative.** Before build: reconcile to **one** parameter table (I recommend it live in `contracts/schemas/` as a machine-readable defaults file per D-052/D-034, imported by both the methodology prose and PARAMETER-TUNING, so prose can never drift from it). Pick the value deliberately: 0.95 is safer for *not* over-calling infrastructure but risks letting sub-95% chatterers through; 0.8 catches more infra but risks routing sub-daily trackers away from the exception — given the system's stated bias (miss-a-tracker is worst, Q-05), **0.95 is the right default and PARAMETER-TUNING's 0.8 should be corrected**. Critically, add a fixture persona for a *sub-95%-presence* tracker (e.g. 90-min cadence, ~16/24 hours) that asserts it survives — P2 at ~100% presence does not exercise the boundary. And nail down that **min-events is a Level 2 analysis filter, never a Level 1 handoff filter** (else it violates HANDOFF's guarantee that low-duty trackers are preserved) — P15 implies this but no contract states it.

**downstream_impact.** D-028, the low-duty-cycle exception, HANDOFF "noise and infrastructure already removed… including the low-duty-cycle exception," P2 and P15 fixture derivations, PARAMETER-TUNING table, Q-05, and the taxonomy G1 detection guarantee. The min-events ambiguity touches the L1/L2 boundary (D-029).

**confidence.** high — both numeric disagreements are directly quotable from two live (non-archive) documents.

### FINDING-RF.5 · severity: major · target: stage-order contradiction across four contracts (methodology §19; R7 line 196; HANDOFF §8; RFANALYSIS-SCOPE §54)

**steelman.** All four documents agree on the *set* of stages (power-bin, sweep-collapse, burst-group, noise-filter) and on the two invariants that actually matter operationally: filtered devices don't flow forward (INNER JOIN filter='pass'), and events are built only from passing FORWARD signal. One could argue the ordering differences are presentational — the pipeline "obviously" filters before it spends effort building events, as the methodology prose says (line 100: "before event construction… so infrastructure and noise are removed before event-building effort is spent"). Steelman: the intent is consistent even if the diagrams differ.

**failure_scenario.** The four documents specify **four different orderings**, and the ordering is not cosmetic — it changes results. The variants:
- methodology numbered list (§19, lines 19–24): power-bin → **sweep-collapse (2) → burst-group (3) → noise-filter (4)**. Here noise filtering is **last**.
- methodology prose (§100): noise filter sits "After power-bin grouping, **before** event construction" — i.e. noise-filter **before** sweep-collapse/burst-group. This **contradicts the numbered list in the same document.**
- R7 (line 196): power clustering → **noise filtering → sweep-collapse → burst grouping**. Noise-filter **second**, before sweep-collapse.
- HANDOFF (§8): power-bin grouping → **noise filtering → sweep-collapse → burst grouping**. Matches R7.
- RFANALYSIS-SCOPE (§54): lists `filter_decisions` then `events` (sweep-collapse then burst) — matches R7/HANDOFF ordering (filter before events).

So the *methodology's own numbered recipe* (the document that is supposed to be authoritative for this algorithm) puts noise-filtering **after** event construction, while every other document (and the methodology's own prose) puts it **before**. This is not academic: the noise filter is computed on **hourly presence × total activity** (methodology line 102), which is a per-`(agent, earfcn, power_bin)` aggregate over raw observations. If you sweep-collapse and burst-group *first* (methodology numbered list), the "total activity" counted by the filter is over collapsed detections, not raw observations, changing `SUM(observation_count)` and therefore changing which devices cross `MIN_OBSERVATIONS` and the `timeframe×40` line. A deep-sleep tracker (P15) that has, say, 12 raw sweep-hits collapsing to 4 detections could pass a raw-count MIN_OBSERVATIONS but fail a detection-count one — **the P15 "survives noise filter" assertion depends on which order the build engineer picks**, and the fixture would encode whichever the author happened to compute by hand.

**evidence.** Quoted line-for-line above: methodology lines 19–24 vs line 100; R7 line 196; HANDOFF line 8; RFANALYSIS-SCOPE line 54. The methodology internally contradicts itself (numbered list vs prose). D-026 records "sweep-collapse ≠ burst-grouping (two scales)" but says nothing about where noise-filter sits relative to them. **What is absent:** a single canonical ordering, and a statement of *what the noise filter's activity count is computed over* (raw observations vs collapsed detections vs FORWARD rows) — which the ordering silently determines.

**proposed_alternative.** Fix the canonical order in **one** place (R7 is the natural home as the pipeline-stage contract) and make the other three reference it rather than restating it. The correct order on the merits is **power-bin → noise-filter (on raw observation counts) → sweep-collapse → burst-group**, matching R7/HANDOFF/RFANALYSIS-SCOPE and the methodology *prose*; correct the methodology's numbered list (§19) which is the outlier. Explicitly state that the noise filter's "total activity" is `SUM(observation_count)` over **raw enriched observations** (weighted per R2), computed *before* collapse — and add that assertion to P15's derivation so the survival guarantee is order-independent. This is a pure-doc edit now; post-build it is a pipeline rewrite plus a re-derivation of every fixture expected-output.

**downstream_impact.** R7, methodology (self-consistency), HANDOFF, RFANALYSIS-SCOPE, D-026, every fixture persona whose expected output depends on filter counts (P2, P3, P8, P15), and the determinism whole-run assertion (two engineers reading two orderings produce two "correct" event sets).

**confidence.** high — the four orderings and the intra-document contradiction are directly quotable.

### FINDING-RF.6 · severity: major · target: cold-start G1-vs-G5 discrimination over-claims before corpus exists (taxonomy G1/G5; D-055; REVIEW-BRIEF line 24)

**steelman.** The design is unusually self-aware here. The taxonomy explicitly names G5 "THE HARD CONFUSION CLASS" (line 27) and says "This class is why the labeled corpus matters most — G1-vs-G5 is the judgment that physical confirmation trains" (line 30). D-055 marks classification "v1 — corpus tunes it later" and the correlation method "PROVISIONAL… loudly marked provisional." The REVIEW-BRIEF itself flags this as a suspected weak spot (line 24). The four-factor score (NEW + STATIONARY + LOW-DUTY + PERIODIC) is a reasonable prior. Steelman: nothing here *claims* the cold-start classifier is validated; it is explicitly a seed for a corpus-trained future.

**failure_scenario.** The honesty is real but incomplete: the discriminators the cold-start classifier leans on are **the very features that are unavailable or unreliable at cold start**. Walk the G1-vs-G5 decision (taxonomy line 29): the discriminators are *first-seen date*, *duty/interval character*, *inventory correlation*, *permanence (months of unchanging presence)*. At cold start — the first days of a new facility — **"first-seen date" is meaningless** (everything is first-seen "now," because the sensor just started), **"permanence / months of unchanging presence" is unobservable** (there are no months of history), and **"inventory correlation" requires a curated facility allowlist that, like the corpus, does not yet exist**. That strips three of the four G5-discriminators, leaving only "duty/interval character" (mains vs battery) — which is exactly the axis G1 and G5 *overlap* on (both are "periodic like a tracker, stationary like a tracker," line 27). So in the first-facility week — precisely when an operator most needs to trust the tool — **a fixed IoT sensor (G5, benign) and a planted deep-sleep tracker (G1, the target) are near-indistinguishable**, yet the four-factor score will fire NEW (everything is new) + STATIONARY + LOW-DUTY + PERIODIC on *both*, producing a flood of high-scoring false positives that are actually facility telemetry. The dashboard J3 journey ("top score → confirm") would walk the operator straight into confirming benign IoT as trackers, and each such confirmation *poisons the corpus* that is supposed to fix the problem. The system is most confident exactly when it has the least ground to be.

**evidence.** taxonomy line 29 (G5 discriminators = first-seen, duty, inventory, permanence) vs line 14 (G1 signature includes "**sudden first-seen date** (it arrived)") and line 12 four-factor "NEW + STATIONARY + LOW-DUTY + PERIODIC." At cold start, "NEW" is degenerate (all devices are new) and "permanence" (line 29) is unmeasurable. D-055 item (4) marks *correlation* provisional but item (2) rule-based classification is **not** flagged as degraded-at-cold-start — it is described as shipping in the initial dashboard with no cold-start caveat. PARAMETER-TUNING line 23 confirms the corpus "is impossible to reconstruct retroactively" and "all tuning is subjective" without it. **What is absent:** any statement that the first-facility classification output is *low-confidence by construction* until N days of baseline accrue and the inventory allowlist is populated; no "warm-up period" concept; no persona testing the cold-start false-positive flood (P15 tests a single deep-sleep device in isolation, not a facility full of G5 telemetry that all scores high on day one).

**proposed_alternative.** Add an explicit **cold-start / warm-up doctrine** to the taxonomy and D-055: (1) until a facility has ≥ *W* days of history, mark ALL first-seen-based factors as *unavailable* and lower every candidate's confidence accordingly (a "baseline-forming, classifications provisional" banner on the dashboard, mirroring the provisional-correlation discipline); (2) require the facility inventory allowlist to be populated as an **onboarding step** (like the Sensor registry gate R5.0) *before* G5 discrimination is trusted, so known telemetry is suppressed by identity rather than mis-scored; (3) add a fixture persona: a facility with several G5 telemetry devices *and* one true G1, run at cold start, asserting the system does **not** rank the G5s above the G1 with high confidence (or, more honestly, asserting they are all flagged low-confidence pending baseline). Frame the first-facility week (D-056 re-runs journeys on real data) as *corpus-seeding*, not *detection* — the deliverable of week one is a labeled baseline, not verdicts.

**downstream_impact.** taxonomy G1/G5, D-055 (classification + scoring), four-factor score, DASHBOARD J3/J5 journeys and candidate ranking, PARAMETER-TUNING corpus-collection (the corpus is *seeded* by these first confirmations, so their quality is load-bearing), Q-T1/Q-T2 (confusion cases). Extends REVIEW-BRIEF line 24.

### FINDING-RF.7 · severity: major · target: Pass-2 stub + false-split/merge cost undecided lets the primary failure mode ship silently (D-027; methodology Pass-2 OPEN; Q-03/Q-05; P5 xfail)

**steelman.** This is handled about as well as an unresolved algorithm can be pre-code. D-027 LOCKS the two-pass *structure* (the load-bearing performance decision) while leaving only the *calc* open; the seam is explicit; P5 is committed as `xfail` from day one "so the stub can never be mistaken for done" (fixture line 47); RFANALYSIS-SCOPE line 79 forbids the placeholder becoming canonical. The methodology even lays out three named candidate calcs with tradeoffs and a provisional lean (Q-03: centroid-nearest). Steelman: the seam is honest and the structure is right.

**failure_scenario.** The seam is honest but the **default behavior of the stub is undefined in the safe direction**, and that matters because the stub *will run in production* during the entire best-effort window before the real calc is found/chosen. RFANALYSIS-SCOPE line 79 suggests using "the current single-pass fold as placeholder." But that single-pass fold (methodology line 87) "triggers on **bin population**, not **proximity to a boundary**, and is a whole-bin remap" — it is a *different algorithm that can false-merge two genuinely distinct devices* (it folds a low-count bin into its highest-count neighbor regardless of whether they are one device or two). So the placeholder doesn't just "do nothing safely" — it can actively **merge P6's two real devices** (−20 and −44 are far apart so P6 is safe, but two devices at −20 and −22 with unequal counts would be folded together by the population rule). Meanwhile the false-split-vs-merge cost question (Q-05) that would tell you *which way to bias the threshold* is still open — and until it's answered, neither the stub nor the eventual calc has a defined conservatism. The compound risk: the system ships with a placeholder that can false-merge, while the primary *documented* failure mode is false-split (under-detection, methodology line 147), and the parameter that would resolve the tension is deferred. Result: during the exact window when the corpus is being seeded (FINDING-RF.6), power binning is at its least trustworthy, so the seed labels attach to possibly-merged streams.

**evidence.** methodology line 87 (single-pass fold triggers on population, is a whole-bin remap, "**not** the intended two-pass method"); RFANALYSIS-SCOPE line 79 ("e.g. current single-pass fold as placeholder"); methodology lines 147–148 (false-split = primary failure, but "which error is worse… Pending your input"); Q-05 open; P5 line 20 is `xfail` so **no green test guards against the stub's false-merges** — P6 guards far-apart bins but nothing guards near-but-distinct bins under the population-fold placeholder. **What is absent:** a specified *safe* stub behavior (I'd argue the safe stub is **identity / no reconciliation at all** — accept false-splits, which are the known-tolerable error per the working assumption — *not* the population-fold, which introduces the worse error class), and a resolution of Q-05 that sets the placeholder's bias.

**proposed_alternative.** (1) Make the stub **explicitly no-op boundary reconciliation** (Pass-1 only, no merging) rather than the population-fold, so the placeholder errs toward the *documented-tolerable* false-split rather than the worse false-merge; document this as "stub = under-merge on purpose." (2) Resolve Q-05 *before* choosing among boundary-zone/centroid/valley — the cost asymmetry *is* the spec for the calc's conservatism, so it blocks Q-03 more than the doc admits. (3) Add a *green* (not xfail) fixture persona for the stub's own guarantee: near-but-distinct devices (−20/−22, unequal counts) must **not** be merged by the placeholder — this catches the population-fold hazard that P5's xfail and P6's far-apart bins both miss. (4) On the transitive-vs-one-hop open item: specify one-hop only for the stub (transitive chaining across >1 boundary is where a single sweep of jitter could zip three distinct devices into one; it should be the deliberate choice of the *real* calc, never a placeholder default).

**downstream_impact.** D-027, Q-03, Q-05, P5/P6 + a new stub-guard persona, HANDOFF `power_bin` "authoritative post-Pass-2" guarantee (which the stub cannot honestly make — the handoff should mark power_bin *provisional* while P5 is xfail, mirroring the correlation-provisional discipline), PARAMETER-TUNING corpus seeding (FINDING-RF.6), and the false-split cost note in methodology.

**confidence.** high — the placeholder's false-merge hazard and the undefined stub bias are both traceable to quoted lines; the recommendation is a doc-level edit.

### FINDING-RF.8 · severity: minor · target: noise-floor discrepancy and per-agent floor break cross-sensor comparability (LOCAL-HOST L0.5/L1 line 104; Q-07)

**steelman.** The doc handles this maturely: it recognizes the floor is *per-agent* (a −55 sensor is "a fundamentally different instrument" than a −70 one, L0.5 line 64), makes capture config travel with the data (D-014), and flags the −50/−60 constant discrepancy as an open confirm (Q-07). It even notes the per-agent reality is "the strongest argument for capture config traveling with the data" (line 81). Steelman: the physics is understood and the mechanism (config-with-data) is right.

**failure_scenario.** Two issues survive the steelman. First, the −50/−60 discrepancy is described as "confirm the deployed value" (Q-07) but line 104 adds the consequence "the server's −110 references are dead" — meaning somewhere in donor code the floor was assumed at −110, a **60 dBm error** from the real −50. If any surviving logic (score normalization, a "sub-threshold noise" cut, a power-to-proximity heuristic) still references −110, it is off by orders of magnitude and would either pass everything or nothing. This is flagged as a value-confirm but is really a "hunt for dead −110 references" task that isn't a WP. Second, and more physical: cross-sensor correlation and PARAMETER-TUNING's cross-sensor-agreement proxy both assume the **two same-band sensors of a host agree about a device**. But if `ral-rf1a` runs floor −55 and `ral-rf1c` runs −70 (per-agent tuning, legitimately different mounts), the −70 sensor *sees weaker signals the −55 sensor never records* — so for a mid-power device the two sensors will systematically *disagree* on presence, and the "agreement = correctness" proxy (PARAMETER-TUNING line 25) is confounded by the floor difference, not by event-construction quality. The doc says correlation "flags incomparable capture settings" (L0.5 line 77) but never defines *how different two floors may be before they're incomparable*.

**evidence.** LOCAL-HOST line 104: "constants say `NOISE_FLOOR_DBM: -50`, docs/eartest say −60… the server's −110 references are dead." L0.5 line 77: correlation "flags incomparable capture settings" — mechanism named, threshold undefined. PARAMETER-TUNING line 25: cross-sensor agreement as a correctness proxy, with no floor-difference caveat. **What is absent:** a WP/task to grep out and kill dead −110 references; a defined floor-delta tolerance beyond which two same-band sensors are marked non-comparable (parallel to the L4.5 override-mismatch flag P19, which handles a *different* comparability hazard).

**proposed_alternative.** (1) Turn "the −110 references are dead" into an explicit build task: on greenfield, assert no floor constant other than the per-agent config value exists (a grep-gate in CI). (2) Define a floor-delta comparability rule: if two same-band sensors' `noise_floor_dbm` differ by more than `bin_size` (or a stated dBm), mark their correlation *non-comparable* and their cross-sensor-agreement proxy *unusable* — and add this to P19-adjacent fixture coverage (P19 covers override mismatch; add floor mismatch). (3) Answer Q-07 with *per-environment* starting floors (open warehouse vs racked) as it already asks, and record them in `ear-fleet` per-host.

**downstream_impact.** LOCAL-HOST L0.5/L1, D-014, R8 correlation, PARAMETER-TUNING cross-sensor proxy, P19 (add a floor-mismatch sibling), Q-07. Interacts with FINDING-RF.6 (cold-start already fragile; floor disagreement adds noise to the seed corpus).

**confidence.** med — the −110 dead-reference risk depends on donor code not fully visible in the doc set; the comparability-threshold gap is certain from the text.

### FINDING-RF.9 · severity: minor · target: power-variance → "stationary vs moving" and cross-sensor power-gradient → "rough position" claims (taxonomy line 12/47; HANDOFF `power_variance`; R9)

**steelman.** The docs are careful about calibration in the right places: R1/R9 forbid treating dBm as absolute, R9 forces proximity language through an "uncalibrated/coarse" formatter, and HANDOFF's "What L2 must NOT assume" explicitly says "proximity/bearing from it is coarse." The taxonomy's use of power *variance* (not absolute power) to infer stationary-vs-moving is a relative measure, which sidesteps calibration. Steelman: the calibration discipline is genuinely good and consistently applied.

**failure_scenario.** Two physics gaps remain under the good calibration discipline. First, "**stable power ⇒ stationary**" (taxonomy line 12, "stable power (planted on a stationary asset → fixed multipath)") is a **necessary-not-sufficient** inference that the doc treats as a discriminator. A device can have stable power while moving (e.g. an asset moving at constant range/geometry relative to the sensor — a tracker on a forklift doing repetitive laps at roughly fixed distance), and a *stationary* device can have unstable power (a fixed sensor behind an intermittently-opened metal door → multipath swings). Multipath in a warehouse (metal racks, forklifts, roll-up doors) makes power variance a **noisy** stationarity proxy, yet it is one leg of the four-factor score (STATIONARY) and a taxonomy dimension. Second, "cross-sensor agreement → power gradient → rough position" (taxonomy line 47) implies triangulation-by-power, but with **uncalibrated, per-agent-different-gain, per-agent-different-floor** sensors (FINDING-RF.8), the two sensors' dBm are not on a common scale — so a "gradient" between them conflates real range difference with gain/floor difference. The R9 formatter labels the *output* coarse, but the taxonomy dimension table (line 47) presents power-gradient positioning as a usable discriminator without noting it requires the two sensors to be gain-matched to mean anything.

**evidence.** taxonomy line 12 ("stable power… → fixed multipath"), line 42 ("Power variance | stationary (G1/G4/G5/G6) vs mobile (G3, moving G2)"), line 47 ("Cross-sensor agreement | … power gradient → rough position"). These sit in the *discriminator* table as if reliable. HANDOFF `power_variance` field is handed over with no note that warehouse multipath makes it a weak stationarity signal. **What is absent:** any caveat that (a) power stability is necessary-not-sufficient for stationarity and multipath-confounded indoors, and (b) cross-sensor power-gradient positioning requires gain/floor-matched sensors to be meaningful.

**proposed_alternative.** (1) Downgrade "power variance → stationary" from a *discriminator* to a *weak prior* in the taxonomy, with an explicit multipath caveat, and ensure the four-factor STATIONARY leg is one input among four, never a gate. (2) On cross-sensor power-gradient positioning, add the precondition "only meaningful between gain/floor-matched same-band sensors" and tie it to the FINDING-RF.8 comparability rule; otherwise mark gradient-position *unavailable* for mismatched pairs. (3) These are Level-2 concerns (D-029 boundary) so they cost nothing to caveat now and prevent an over-claiming dashboard later. Extends Q-T2/Q-T4 (facility-specific confusion, 24/7 operations weakening discriminators).

**downstream_impact.** taxonomy dimension table + four-factor score, HANDOFF `power_variance`, R9 report formatter (already good — extend the caveat upstream to the discriminator), DASHBOARD device-detail (which surfaces "stationary"), Q-T2/Q-T4.

**confidence.** med — the multipath confound and the gain-matching precondition are solid RF physics; whether they bite depends on facility geometry (Q-T4), which is why it is minor not major.

### FINDING-RF.10 · severity: minor · target: sweep undersampling assumes a fixed transmission duration the docs never pin (methodology §17; P1)

**steelman.** The undersampling story is the design's best physical insight and the reason sweep-collapse exists: a ~10s uplink broadcast caught "4 of 5 possible times" as the sweep passes (methodology line 17). It correctly separates two scales (transmission-reconstruction vs episode-assembly). Steelman: the core insight is sound and rare to get right.

**failure_scenario.** The whole sweep-collapse calibration hangs on an assumed **transmission duration** ("~10 seconds," methodology line 17; P1 encodes "one ~10 s transmission captured as 4 sweep hits ~2 s apart"). But LTE uplink transmissions are **not** uniformly ~10s — a periodic tracking report (PUSCH grant for a small position payload) can be *sub-second*; a data session is many seconds; a RACH preamble is milliseconds. If the real target's uplink burst is, say, 0.5s, then with a 1.5–4s sweep revisit (FINDING-RF.2) the sweep will catch each burst **at most once**, and "sweep-collapse merging 4 hits into one transmission" describes a transmission structure the target doesn't have — every hit is already one transmission, and the collapse window would instead wrongly *merge two genuinely separate sub-second reports* that happen to fall within 2s. The 10s figure appears to come from one observed device class and is generalized into the model and baked into P1/P15 derivations, but Q-T1 (wake intervals per sub-pattern) and the actual *burst duration* per sub-pattern are unknown. The model conflates "how long a transmission lasts" (sets whether collapse is even needed) with "how often the device wakes" (sets the burst gap) — both are open (Q-T1) yet the transmission duration is treated as SETTLED via P1.

**evidence.** methodology line 17 ("a ~10-second uplink broadcast might be caught only 4 of 5 possible times"); P1 fixture ("one ~10 s transmission captured as 4 sweep hits ~2 s apart"). Q-T1 asks wake *intervals* but **not burst/transmission duration** — the duration that determines whether sweep-collapse applies at all is not even in the field-truth question set. **What is absent:** a per-sub-pattern transmission-duration figure, and a statement of what sweep-collapse does when transmission duration < revisit time (answer: it should collapse *nothing* and must not merge distinct short reports).

**proposed_alternative.** Add transmission/burst **duration** (not just interval) to Q-T1's field-truth ask, per sub-pattern. In the methodology, state the regime explicitly: sweep-collapse only merges hits when `transmission_duration > revisit_time`; when `transmission_duration ≤ revisit_time` the collapse window must be ≤ revisit (never merging across the gap between two distinct short reports). Add a fixture persona: two sub-second reports 1.5s apart with a 2s collapse window must **not** be merged into one detection (guards the over-merge of short bursts) — P1's 10s/4-hit shape does not test the short-transmission regime.

**downstream_impact.** methodology undersampling model, sweep-collapse window (interacts with FINDING-RF.2), P1/P15 derivations, `detection_count`/`consolidation_confidence` semantics, Q-T1 (add duration). 

**confidence.** med — LTE uplink burst durations genuinely vary from ms to seconds; whether the target class is ~10s or sub-second is exactly what Q-T1-adjacent field truth would settle, and the model currently assumes one value.

## 3. WHAT TO PROTECT

- **The two-pass *structure* (D-027).** The cheap-coarse-over-all + expensive-precise-over-boundary-subset split is the correct answer to the tractability constraint (methodology §49–55). The performance argument is load-bearing and real; do not let a build-time "simplify to one pass" churn touch the structure. Only the *calc* and the stub bias (FINDING-RF.7) are open.
- **The low-duty-cycle exception (D-028).** Presence-**and**-activity with the exception is the single correct insight that keeps the primary target from being filtered as infrastructure. The NEVER-remove framing and P2's "if filtered, build FAILS" gate are exactly right. Guard it against any "presence-only is simpler" regression.
- **The feed split (D-012) as the fix for cadence-as-signal.** Making AGGREGATE/FILTER rows structurally unable to enter event construction (physically separate `ctx_` feed) is a genuinely strong design — it converts a recurring bug class into an unrepresentable state. Protect it; FINDING-RF.2 is a *second, subtler* cadence-as-signal risk (sweep revisit), not a flaw in this fix.
- **Calibration discipline (R1/R9, HANDOFF "must not assume").** "Uncalibrated relative dBm, never absolute, proximity is coarse" is applied consistently and is correct physics. FINDING-RF.9 asks to *extend* the caveat upstream to the discriminator table, not to weaken this.
- **Server owns binning; agent bin is advisory (L4/D-012).** Keeping all jitter-tolerance power grouping server-side, with the agent forwarding raw `power_dbm`, is the right seam — it keeps the RF judgment in one place where the methodology lives. Protect against any "let the agent pre-bin to save bytes" optimization.
- **Sweep-collapse ≠ burst-grouping as two distinct scales (D-026).** The conceptual separation (seconds vs minutes, transmission vs episode) is correct and should not be collapsed into one threshold, even though *both* thresholds need the fixes above.
