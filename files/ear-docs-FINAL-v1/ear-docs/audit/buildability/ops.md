# Buildability — Ops + Health + Dashboard (M5) — DEV ENVIRONMENT bar

**Track:** WP-O1/O2/O3, S14/S19/S19d/S19e. HEALTH-CHECKS-v1, DASHBOARD-DESIGN-v1, RUNBOOK-v1, BACKUP-DR-v1, RECOVERY-RETENTION-v1, HOST-PROVISIONING-CHECKLIST-v1, D-049/D-059.
**Question:** can a Fable-led / Opus+Sonnet team build the health sweep, notify()/ntfy, ops+analysis dashboards, and run UAT J1–J7 against dev data (fixture + replay), from the docs alone?

**Verdict: buildable-with-marked-stubs.** This is the most build-ready track in the corpus. The health layer, dashboards, and UAT journeys are almost entirely Sonnet-class CRUD-over-known-models plus a well-specified sweep, and the dev loop (MinIO + replay, D-046/47/48) means every UAT journey is runnable against dev data *by construction*. The blocking gaps are two undefined tuning knobs (alert cardinality, "expected window") that an agent can and should ship as clearly-labeled defaults, plus one ASK-gate (Q-16) that only touches the last analysis screen. Nothing in this track requires an open algorithm, field truth, or a decision an agent would resolve *wrong*.

---

## 1. Specified well enough to build as-written

- **Health sweep (S14/WP-S4, D-059).** HEALTH-CHECKS-v1 is the strongest ops doc: it gives the exact 10-check table with per-check "functional means / degraded when", the three outputs (persisted `HealthSweep` verdicts JSON → banner; state-change `notify()`; non-zero exit as `OnFailure=` backstop), and the self-monitoring rule (no sweep row in 2× cadence → amber). S14's gate is concrete and testable: *staged stale-sensor/failed-run → correct verdicts + one state-change notify*, and P14 already stages heartbeat-fresh/stale. An agent can build `check_health` + `HealthSweep` model + banner directly.
- **Models the dashboard reads already exist in contract.** RFANALYSIS-SCOPE-v3 pins `Sensor` (agent_id/facility/status pending|qa|active|disabled/serials/bearing/floor) as the onboarding gate + identity SoT, and `AnalysisRun` (run_id/facility/window/config_snapshot/status/stage results/timing). L2 adds `DeviceClassification`/`TrackerCandidate`/`CorrelatedDetection` (WP-L2/L3). The heartbeat schema (`heartbeat.v1.schema.json`) is frozen and gives the exact fields the Sensors board renders (status/disk_pct/queue_depth/dlq_count/ntp_offset_ms). So every dashboard column has a backing field.
- **notify()/ntfy (D-049).** Fully specified for a build: one `notify()` utility POSTing to `ALERT_URL` (ntfy topic), named emitters (disk-ceiling, pipeline non-zero exit via `OnFailure=`, over-filter safeguard, dead-heartbeat), state-change-only with one reminder per configurable period. Buildable and unit-testable against a mock endpoint.
- **Dashboards (S19/WP-O1 ops; S19d/WP-L4 analysis).** DASHBOARD-DESIGN-v1 is a screen-by-screen IA: nav tree, 7 key screens with per-screen column lists and sort ("worst first"), status-color discipline, the two earned visualizations (event timeline, classification bar), Pegasus/shadcn shell as-is (no custom design system). This is Sonnet CRUD over the models above — the DoD ("usable against dev data") is met by the replay loop.
- **UAT journeys J1–J7.** DASHBOARD-DESIGN-v1 gives them as verbatim acceptance scripts with explicit pass criteria (click budgets, "reached conclusion", "corpus row exists"). S19e runs them against dev data (fixture + legacy corpus). Each journey stages its own condition (one stale heartbeat, a pending prefix, a failed run) — exactly what the fixture/replay harness produces.
- **Dev loop is real and switch-only (D-046/47/48, DEV-ENVIRONMENT-v2).** Same container topology as prod; MinIO is the only substitution; `S3_ENDPOINT_URL` is the one switch; replay scanner → MinIO → pipeline → Parquet exercises every contract with no hardware/cloud. DEV-ENVIRONMENT-v2 spells out the bring-up (seed a `tst-` Sensor via admin, place replay dataset, start replay scanner). **This is what makes every UAT journey runnable at the dev bar.**
- **Recovery/retention (WP-O3 pre-written).** RECOVERY-RETENTION-v1 gives the disposability model + retention table with concrete numbers (13-mo obs_, 6-mo ctx_, 90-d landing, 30/180-d dumps). BACKUP-DR-v1 gives the sacred/rebuildable table + two named drills with pass criteria (dump restore renders; rebuild determinism-matches). RUNBOOK-v1 gives day-2 alert responses. All buildable/runnable at dev bar (the drills run against MinIO + a scratch container).

## 2. Blocking gaps

| # | Gap | Block or stub? | Who resolves |
|---|---|---|---|
| O-a | **Alert cardinality undefined (OPS.2).** A facility-WAN outage flips 48 heartbeats stale in one sweep. Docs say the liveness check flags *per sensor* ("that sensor flagged") but also "WAN suspicion noted" — undefined whether notify() fires 48 pushes or 1 facility-rollup. An agent will pick one arbitrarily; per-red firing = 48-push alert storm on first outage (a wrong build the operator will hate), per-sweep firing = WAN and single-sensor look identical. | **STUB.** Agent ships a facility-rollup: group red sub-verdicts by facility, emit one push when ≥N same-facility sensors go red in a sweep ("facility ral: N sensors dark, WAN suspected"), else per-sensor. Mark the N threshold `(dev default, tune in soak)`. Not a data-corruption path; a labeled default is correct seam discipline. | agent-can-stub (Guy confirms N later) |
| O-b | **"Expected window" for capture/handoff advancing undefined (OPS.10).** The obs_-arrival "capture advancing" check false-reds a legitimately quiet deep-sleeper sensor (near-zero FORWARD volume is *correct* for the primary target, D-028/P15) — or is loosened until a genuinely dead scanner passes. "Expected window" is never quantified and is per-band/per-facility. | **STUB, with a correctness caveat.** Agent must base *liveness* on ctx_/heartbeat freshness (arrives every tick by construction), and treat obs_ volume as informational/self-baselined, never a hard red. This is the audit's own fix and it is unambiguous enough to build. If an agent naively keys liveness on obs_ it builds a false-red generator — so this needs a **doc-fix line** in HEALTH-CHECKS (one sentence: "capture-advancing = ctx_ freshness; zero obs_ is not a red") to steer the build, then it's a clean stub with a self-baselined window default. | doc-fix (one line) + agent-can-stub |
| O-c | **External-beacon gap (OPS.4).** Every watchdog layer (HealthSweep write, banner, `/health/`) lives *inside* the box; total-box-death produces no alert. | **STUB / partial at dev bar; production-only for the real backstop.** At the dev bar there is no external beacon requirement — the loop runs on one box and box-death is a dev annoyance, not a missed tracker. The agent can add a healthchecks.io-style dead-man ping as a labeled stub (env var, off by default in dev). The *actual* need (Cloudflare Worker reading R2, or healthchecks.io) is a production reliability item, not a dev-build blocker. | needs-hardware/field (prod backstop); agent-can-stub the ping hook |
| O-d | **Q-16 (analysis summary / report shape) is ASK-gated on S19d and informs J6/J6e.** Runsheet marks S19d "ASK Q-16 (summary page shape)" and S19e "ASK Q-16 informs J6". Facility-summary screen shape + report audience are unanswered. | **Partial block on ONE screen.** Blocks the *final shape* of Facility summary (screen 7) and J6's acceptance detail — not the ops dashboard (S19/WP-O1), not the sweep, not J1/J2/J4/J5/J7. Agent can build the Facility-summary screen per the DASHBOARD-DESIGN default layout ("shape refined by Q-16") and run J6 against it, flagging the shape as provisional. Only the operator-facing report format truly waits on Guy. | Guy-answer (for final shape) / agent-can-stub the default screen |
| O-e | **Heartbeat host-vs-device scope (A6/REL.1/SEAM-8) leaks into the Sensors board + capture check.** The frozen heartbeat schema has no `host`/`scope` field, so host-level `disk_pct` and per-device `queue_depth` can't be disambiguated; if uploader cardinality (1 shared vs 4) is unresolved, the dashboard may mis-scope one host event as four sensor rows. The ops lens declared the schema "clean," which the audit flags as evidence it *reads* unambiguous when it isn't. | **Upstream block, inherited.** This is a pre-repo schema/ADR fix owned by the agent track (heartbeat-semantics ADR + `scope` annotation), not resolvable inside the ops build. If it lands wrong upstream, the Sensors board renders wrong cardinality. Ops can't stub around a frozen-schema ambiguity — it consumes whatever the agent emits. | doc-fix upstream (heartbeat ADR, pre-repo) — **not an ops-track fix** |
| O-f | **Q-19 (run cadence) is DEFAULT, and the sweep's "recent (< 2×/3× cadence)" thresholds depend on it.** HealthSweep freshness verdicts ("< 3× upload cadence", "< 2× cadence") are relative to cadences Q-19 leaves open (hourly vs 2–4×/day). | **STUB.** DEFAULT gate — agent proceeds on the logged provisional (2–4×/day), parameterizes the multipliers, flags in summary. Standard runsheet DEFAULT handling; not a blocker. | agent-can-stub (Guy confirms in soak) |
| O-g | **ntfy topic value (S14 "ASK ntfy topic (or default)").** The concrete topic string. | **STUB.** Runsheet explicitly permits a default. Env-var it. | agent-can-stub |
| O-h | **L2 ops surface has no runbook (OPS.9, SEAM-7).** run_level2 is a second failure class; RUNBOOK/RECOVERY-RETENTION describe only L1. Affects J7-adjacent triage and the L2 health check's response path. | **STUB (dev).** The health check *exists* (Layer 3 L2 row); the missing piece is a recovery *procedure* (L2 = delete-DuckDB + re-run, inferable from disposability) and unclassified-queue cadence. Dev build runs fine; the runbook clause is a cheap doc-add. Not a code blocker for the dashboard/sweep. | doc-fix (cheap) |

**Inherently production/field (do NOT block the dev build):** real facility onboarding (WP-O2 first-facility rollout, per-sensor promotion at 28–60/facility — OPS.5); DR/rollback *drill cadence* and rollback procedure (OPS.3/OPS.6 — the drills themselves are dev-runnable, the *recurrence* is a field discipline); Q-23 legal clearance; the external dead-man's-switch as a real prod backstop (OPS.4); maintenance-window suppression primitive (OPS.11 — nice-to-have, dev works without it); age-key escrow / on-call staffing (OPS.1). None of these stop a dev stack from standing up and passing UAT.

## 3. Dev-env verdict

**The dashboard + health layer is buildable to working dev code, and all seven UAT journeys are runnable against dev data.** The sweep, notify(), ops dashboard, and analysis dashboard are specified to a build-ready level; the models they read are pinned in contract; the replay/MinIO loop stages every journey's precondition without hardware or cloud. The only things that would misdirect a build are two undefined tuning knobs (alert cardinality O-a, "expected window" O-b) — and for both, the audit already states the correct resolution, so an agent ships a clearly-labeled default (P5-xfail-style seam discipline) and proceeds rather than stalling. One cheap doc-fix (O-b's one-line "liveness = ctx_ freshness") meaningfully de-risks the build by steering the agent away from the false-red trap. Q-16 (O-d) is the only genuine ASK-gate and it touches one analysis screen, not the ops/health core. The single gap the ops track cannot fix itself is the upstream heartbeat host/device scope (O-e), which belongs to the agent track's pre-repo schema ADR.

**Net: an S14 + S19 + S19d/S19e build session would produce working dev code and pass UAT, given the two labeled-default stubs and one one-line doc-fix.**
