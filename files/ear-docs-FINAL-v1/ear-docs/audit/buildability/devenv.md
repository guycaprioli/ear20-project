# Buildability Assessment — Dev Environment + Infrastructure (M0 track)

**Question:** Can Fable(lead)+Opus+Sonnet stand up the full dev stack — Strix box layout, bare repos + GitHub mirror, container-per-service compose (proxy/web/pipeline/postgres + MinIO dev profile, GHCR images), the two-address networking rule, and the replay loop (replay agent → MinIO → pipeline → Parquet) — from the docs alone, following the runsheet? Judged at the **dev-env bar** (MinIO substitutes R2, `replay` substitutes HackRF; no hardware, no cloud).

**Verdict: buildable-with-marked-stubs.** The dev-env *scaffolding itself* (box layout, bare repos, compose topology, MinIO substitution, networking, bring-up order) is specified concretely enough to execute nearly as-written — this is the strongest-specified part of the corpus. What blocks a *green replay loop end-to-end* is not the harness; it is two upstream contract contradictions the loop exercises (field drift → 100% quarantine; uploader cardinality) that an agent would resolve wrong. Those are cheap pre-session doc-fixes. The dev *stack* stands up regardless; the *loop's DoD* ("events → Parquet appear") depends on the fixes landing, or on the agent stubbing the schema correctly by reading the JSON schema and ignoring the prose.

---

## 1. Specified well enough to build as-written

- **Container topology / parity table** — DEV-ENVIRONMENT-v2 §"Container topology" + WP-S1 give the exact service set (proxy/web/pipeline/postgres + minio dev profile), which are resident vs ephemeral (D-044), same-image-for-web-and-pipeline, GHCR as the registry, and the single intentional substitution (MinIO for R2, `S3_ENDPOINT_URL` the only switch, dev-profile-gated). An agent can author the compose file directly from this.
- **MinIO-as-R2 substitution** — the switch is named (`S3_ENDPOINT_URL`), the bucket-init sidecar is called out, the substitution is bounded to one service. Concrete and correct; this is the whole "no cloud" claim and it holds.
- **Two-address networking rule** — DEV-ENVIRONMENT-v2 §Networking pins it explicitly: host-side agents → `localhost:9000`, in-network pipeline → `http://minio:9000`, same bucket, "expected not a bug," and says to pin it in `.env` samples. This is the single most error-prone thing about a MinIO dev loop and it is spelled out. Excellent.
- **Bare repos + GitHub mirror (D-048)** — exact `/srv/git/*.git` paths, `git init --bare`, working clones under `~/dev/ear-*`, `git remote add github`, and the session-close push rule. Fully executable by Sonnet at S00.
- **Disk layout + sizing** — byte-budgeted per-directory table, RAM sizing (pool × `memory_limit`, 4×4 GB start), 2 TB NVMe rationale. No invention needed.
- **Bring-up order (S00→S10)** — DEV-ENVIRONMENT-v2 §"Bring-up order" is a numbered 0–5 sequence (base packages → clone+`compose --profile dev up` → agent venv + seed a `tst-` Sensor → start replay scanner → `compose run --rm pipeline run_level1` → fixture suite green). The runsheet mirrors it (S00, S10) with per-session gates. This is genuinely step-by-step.
- **Replay/capture-source seam** — WP-A6 defines `hackrf`/`replay` behind one interface, replay driver (timestamp shift, speed factor, raw-stdout + legacy-JSONL inputs), `--record`. The seam that makes "no hardware" work is a named work package with a DoD.
- **Gate discipline** — every session reduces to one `make gate` command with a committed gate report (D-058); S10's gate is literally "replay agents → MinIO loop runs on the box." Buildability is checkable.

DoD/WP anchors that make it buildable: WP-D0 (step 0 provisioning), WP-D1 (skeletons + CI; schemas already authored in `contracts/schemas/`), WP-A6 (dev harness), WP-S1 (compose per parity table), runsheet S00/S02/S10/S11.

## 3. Net judgment for this track

The dev-environment *infrastructure* — the thing this track owns — is **the best-specified component in the corpus** and stands up essentially as-written: box layout, bare-repo/mirror setup, the parity compose topology, the MinIO substitution, the two-address rule, and the bring-up order are all concrete and executable by Sonnet (S00/S02/S10) with Opus on the contract-critical uploader/pipeline pieces. No open *algorithm* or unstated tunable blocks the harness (the deep-sleeper `MIN_OBSERVATIONS`, Pass-2 calc, sweep-collapse etc. are event-construction gaps that live in other tracks and don't gate the loop *standing up*).

What separates "the stack comes up" from "the replay loop goes green end-to-end" is **G2 (field drift → 100% quarantine)** and, to build the *right* topology, **G3 (uploader cardinality)** and **G1 (repo count at the S00 ASK-gate)**. All three are **cheap pre-session doc-fixes**, not design work — one paired-change commit each, and the audit already flags them as change-before-repo-creation. With those three edits landed before S00/S03, an agent team builds the entire dev environment and a green replay loop from the docs alone. Without them, the containers still run but the loop's DoD fails and the fixture cements a wrong schema as truth.

**Recommendation:** land G1/G2/G3/G7 (four doc-fixes, hours of work) before S00. Then this track is buildable-as-written; G4/G6 ride as clearly-marked stubs per the project's own seam discipline (D-060), and G5 is ordinary greenfield authoring.
