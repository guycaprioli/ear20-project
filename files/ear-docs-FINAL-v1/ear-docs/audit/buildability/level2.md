# Buildability — ear-analysis Level 2 (M4.5, D-055 best-effort)

**Track:** L2 feature extraction (WP-L1), rule-based G1–G6 classifier + four-factor scoring (WP-L2), PROVISIONAL correlation (WP-L3), persistence to Postgres, analysis dashboard (WP-L4). Sessions S19a–e.
**Bar:** dev environment (D-046/47/48 replay loop, MinIO, `run_level2` after L1). No hardware/cloud/field truth required to stand up the stack.
**Verdict in one line:** buildable to working dev code **with marked stubs** — the plumbing, DuckDB extraction, persistence, dashboard, and `run_level2` job are well-specified; the classifier and correlation must ship as clearly-marked provisional stubs graded against synthetic fixture personas, exactly as the project's own seam discipline (P5-xfail / "provisional correlation") already prescribes. Two cheap doc-fixes should land pre-session or an agent will build the wrong thing.

## 1. Specified well enough to build as-written

- **Component shape / boundary (D-042, D-055).** Unambiguous: installable Django app in the ear-server container; DuckDB computes, Postgres serves; ear-analysis reads ONLY the Parquet handoff, writes ONLY its own models; dashboard renders exclusively from Postgres. An agent can lay this out with no invention.
- **Feature extraction (WP-L1, taxonomy §"What Level 2 sees" + classification-dimensions table).** The feature *set* is enumerated concretely: gap mean/stddev/**CV**, duty cycle/total activity, presence ratio across hours/days, power variance, first/last-seen, burst durations, cross-sensor agreement key `(facility, band_group)`. These are standard aggregate SQL over the handoff columns (`observation-signal.v5` + `handoff-events.v1.columns`). DoD is objective and dev-checkable: **"features match hand-computed values for P1–P3"** (S19a), backed by the double-derivation rule (P1/P2/P15 hand-derived in `expected/derivations/`). This is the strongest-specified part of the track — Opus can build it to green.
- **Persistence models (D-055 (5), D-042).** `DeviceClassification`/`TrackerCandidate`/`CorrelatedDetection` named with their roles; standard Django models over Postgres. Buildable.
- **`run_level2` ephemeral job + timer (D-055, D-044/45 pattern).** Same ephemeral-job shape as L1, already exercised in the dev loop. Buildable.
- **Analysis dashboard (WP-L4, DASHBOARD-DESIGN-v1).** Genuinely detailed: nav IA, seven screens with per-row fields, the two earned visualizations (event timeline, classification-breakdown bar), status-color discipline, Pegasus/shadcn shell as-is. UAT journeys J3/J6 are concrete acceptance scripts. Sonnet can build this against fixture + legacy-corpus data.
- **Correlation *seam* (WP-L3, D-055 (4)).** The dev-build target is explicit and modest: "**simple time-overlap join within `(facility, band_group)`, loudly marked provisional**." That is a buildable concrete algorithm for the stub — the *real* method is open, but the stub is fully specified. DoD "same-device-two-sensors fixture case correlates; seam swappable" is dev-checkable.
- **Fixture-as-truth for grading (D-037/D-039/D-057).** P1/P2/P3 (and P15, P18) give the classifier red/green targets that exist without field truth. This is what makes a stubbed classifier *testable* in dev.

## 3. Cold-start / field-truth items that are PRODUCTION-only (do NOT block dev)

- Q-T1..Q-T4 field truth (real wake intervals, confusion cases, missing groups, shift-hours) — needed to *tune* the classifier to truth; the dev build ships provisional rules by design (S19b DEFAULT gate).
- Legacy-corpus sanity review (S19d, "20595→G1") and UAT J1–J7 (S19e) are **human-oracle** points — Guy runs them against dev data; they gate M5, not the ability to produce working L2 code.
- Real G5 inventory/permanence signal (RF.6) — a field/deployment input, not a dev-stack input.

## 4. Can the classifier/correlation ship as marked stubs against synthetic fixtures? — YES

The project already institutionalizes exactly this: P5-xfail seam, "PROVISIONAL correlation loudly marked," DEFAULT gates on Q-T1..T4, provisional-label discipline. WP-L3's correlation stub has a *concrete* dev algorithm (time-overlap join) and a fixture case to grade it. WP-L2's classifier has a buildable decision order and three-to-five persona targets (P1→G1, P2→G1, P3→G4, plus P15→G1 once added, P18 isolation). The four-factor weights and CV/interval cutoffs are agent-invented-and-marked, not blocked. So the answer to the track question is: **not blocked — ships as marked stubs and produces working, fixture-green dev code**, provided the two cheap doc-fixes (G-4 add P15→G1 to S19b + pin few-event CV; G-3 provisional-label marking) land so the agent doesn't cement a silently-wrong deep-sleeper path.

## 5. Dev-env verdict

**buildable-with-marked-stubs.** ~75% of the track is buildable as-written to green dev code (feature extraction, models, `run_level2`, dashboard, correlation-stub). The classifier and four-factor score are open-algorithm surfaces that the agent stubs with invented-but-marked values graded against synthetic personas — sanctioned by the project's seam discipline, so they don't stall the session. The one thing that would make an agent build the *wrong* thing (pass P1/P2/P3 green while silently starving/mislabeling the primary-target deep-sleeper, P15) is a cheap pre-session doc-fix: add `P15→G1` to the S19b gate and pin the few-event CV/min-events behavior. No hardware, cloud, or field truth is required to stand up and grade the L2 dev stack.
