# Buildability Assessment — TRACK: ear-fixture (M1, built first)

**Question:** Can a Claude agent team author the golden fixture (generator + 19 personas + whole-run assertions + double-derivations) to working DEV code from the docs alone, following the runsheet (S01–S05) and bootstrap-v2?

**Dev-env bar:** MinIO substitutes for R2, `replay` source substitutes for HackRF (D-046/47/48). No hardware/cloud needed. Judge against dev, not prod.

**Bottom line:** The fixture *scaffold* (S01/S02 intake + skeletons, and much of S03 ingest personas) is buildable now. But the **double-derivation rule is the whole point of this track**, and it is where the plan stalls. Double-derivation requires hand-computing exact expected values — event counts, conservation sums, filter decisions — **before any code exists**, then Guy signs off P1/P2/P15 at S04. Several values and identities those hand-derivations depend on are unset or self-contradictory in the docs. An agent cannot hand-derive a number the docs don't pin, and where the docs give *two* answers the agent will derive the wrong one and cement it as law (the fixture *defines* truth — D-037/D-044-mode). **S04 (WP-F2: the target-class event personas) is blocked on Guy-answers and pre-session doc-fixes.** S03 is mostly buildable; S05 is buildable.

D-037 (fixture-as-CI-gate), D-039 (built-first), D-057 (P15–P19 + double-derivation) are coherent and mutually consistent.
- **S01 (historic-log intake, Sonnet).** Inventory + classify + manifest of Guy's raw folders. Raw intake only, "validate against nothing yet" — no derivation, no open parameter. Buildable.
- **S02 (repo skeletons + CI stubs, Sonnet).** Empty-green CI, pyproject, lint, fixture hook. Buildable.
- **The generator itself (S03).** Parameterized, seeded, deterministic `.jsonl.gz` synth under `tst-rf1a/`, `tst-rf1b/`, `tst-rf9x/` prefixes, 24 h window. The *input construction* column for every persona is concrete enough to generate. Determinism gate (two runs byte-identical for the *generator*) is a self-contained property. Buildable.
- **Ingest personas whose expected output is a decision, not a computed number** — P10 (idempotency skip), P11 (unregistered → pending, zero ingested), P12 (fake-gzip content-detect), P13 (config-drift provenance), P14 (heartbeat staleness + dead alert), P16 (late-arrival rebuild), P18 (qa isolation), P19 (override-mismatch flag). Their expected outputs are categorical/structural and derivable from the contracts as written. Buildable once the field-name issue below is fixed (they all emit/consume v5 rows).
- **S05 (legacy-corpus adapter, Opus).** Historic JSONL/fake-gzip → v5 shape, validate against `contracts/schemas/`. The v5 schemas exist and are frozen. Buildable (contingent on the same field-name fix, since the adapter's target is the v5 schema — which is actually the *correct* authority, so this one is fine).

The machine-readable schemas (`observation-signal.v5`, `observation-context.v5`, `handoff-events.v1.columns`, `heartbeat.v1`) exist and are the declared tie-breaker — that is a real, load-bearing asset for the fixture.

## What can proceed vs what stalls, by session

- **S01, S02** — Sonnet, buildable now. No blockers.
- **S03 (WP-F1: generator + ingest personas P9–P13)** — buildable **after G4 (field-name) doc-fix**. G3 (conservation identity) partially bites P9 here too; fix it pre-session. Everything else in S03 is categorical/structural.
- **S04 (WP-F2: event personas P1–P8, P15–P19 + whole-run assertions)** — **BLOCKED.** This is the double-derivation heart, gated by Guy's P1/P2/P15 spot-review. It cannot produce correct committed tables until **G1 (MIN_OBSERVATIONS, Guy-answer), G2 (stage-order, doc-fix), G3 (conservation identity, doc-fix), G5 (presence_ratio, doc-fix)** are resolved. G6/G7 are DEFAULT/xfail and do not block. An agent that runs S04 anyway will hand-derive P8/P15 filter decisions and P17 conservation sums from unset/contradictory inputs and commit a *wrong answer as law* — the exact failure the double-derivation rule exists to prevent. This is worse than a stall.
- **S05 (WP-F3: legacy adapter)** — buildable after G4 (its target is the v5 schema, the correct authority).

## The critical-dependency verdict

**The fixture is NOT fully authorable now.** The scaffold and ingest half are; the **target-class event personas (P1/P2/P8/P15/P17) and the whole-run conservation assertion are not**, because their expected values are computed from parameters the docs leave unset (G1 MIN_OBSERVATIONS) or self-contradictory (G2 stage-order, G3 conservation identity, G5 presence_ratio, G4 field name). Double-derivation is precisely the operation that *surfaces* these gaps — you cannot hand-compute a number that isn't pinned — so the plan's own S04 gate ("expected outputs double-derived; Guy spot-reviews P1/P2/P15") will hit a wall on first attempt.

The cost asymmetry is decisive and matches the audit's B-section: G1–G5 are **one-edit fixes now** (four doc-fixes + one Guy value). Left unfixed, they become a full re-derivation of every expected-output table *plus* a pipeline rewrite after the fixture has already cemented the wrong answer into every downstream repo's regression floor (M1-first means everything depends on this). Because the fixture defines truth, a wrong derivation here is not a local bug — it is a wrong contract propagated fleet-wide.

**None of these are hardware/field/legal (prod-only).** All five are resolvable pre-session at a desk: four cheap doc reconciliations the audit already wrote the exact edits for, and one value (MIN_OBSERVATIONS) that only Guy can pin because it encodes the smallest survivable deep-sleeper total — the primary target's survival math. Fix those five and S03/S04/S05 become buildable to working dev code.
