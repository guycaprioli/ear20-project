## Identity & topology
- **D-001 · Facility-scoped identity** `agent_id={facility}-{host}{device}`, lowercase, `^[a-z]{3}-rf\d+[a-d]$`. — Host prefixes collide across facilities; global uniqueness by construction. — Facility immutable; correlation scoped `(facility, band_group)`.
- **D-002 · R2-primary transport** (central Cloudflare hub; per-agent folders). — Multiple facilities + often-remote server make a cloud rendezvous correct; reversed an earlier single-site MinIO lean. — Store-and-forward becomes load-bearing.
- **D-003 · Identity pinned in env, never hostname-derived.** — PC rename must not mint a new sensor. — `FACILITY` + host prefix in `.env` (ear-fleet).
- **D-004 · Server topology (central vs per-facility)** — **OPEN**, Guy, before M3 deploy shape. App built to run either way.

## Agent
- **D-010 · Agent re-platformed to Python** (`ear-agent` v5). — Single Python-centric maintainer; rewrite de-risked by written L-chain contracts + fixture; Phase-1 was gutting the uploader anyway. Initial keep-Node verdict reversed on Guy's stack-alignment call. — Conditions: port-don't-redesign; systemd model kept verbatim; harvest gittest2 patterns.
- **D-011 · Edge filter KEPT** — noise ≈90% of raw traffic (Guy's empirical number); feed preparation is load-bearing. A ship-raw proposal was rejected on this number. — Rationale recorded as a NEVER in R2/L4 so no rewrite "simplifies" it away.
- **D-012 · Feed split** — signal feed `obs_…` (FORWARD only; sole input to event construction) vs context feed `ctx_…` (summaries, L4.5 heartbeats, config stamps). — Makes the historic false-CRITICAL bug structurally unrepresentable. — Two landing paths server-side.
- **D-013 · L4.5 curated summarization override** — per-EARFCN, identity-based, forced-summary (never silent drop), versioned rules, heartbeat always. — Volume control for known infra floods without silent data loss.
- **D-014 · Per-agent capture config travels with data** — batch-stamped + `config_version`. — Server cannot honestly interpret/correlate without capture conditions.
- **D-015 · Agent health heartbeat** to `_health/{agent_id}.json`. — Distinguishes sensor-dead from RF-quiet.
- **D-016 · Process model: 4 scanner processes + 1 shared uploader + 1 shared DLQ per host.** — Matches proven v4.3 model; per-device isolation at folder level. (Docs corrected from an earlier 4×2 claim.)
- **D-017 · Legacy Postgres path not ported** (excision by omission); DLQ replays to R2 only. — Two destinations was a proven hazard (dlqReplay bug).

## Server & pipeline
- **D-020 · rfanalysis spine: Django + embedded DuckDB inside Pegasus host;** NestJS/TimescaleDB rejected. — Batch forensic workload fits DuckDB; Pegasus supplies Celery/Redis/admin/dashboard.
- **D-021 · Two-table ingest** (persistent raw landing → batched enrichment) with quarantine-with-reason; `ignore_errors` banned; folder-authoritative identity; `schema_version=v5` only (greenfield — no COALESCE drift absorbed).
- **D-022 · Onboarding gate**: Sensor registry allowlist, `pending→qa→active→disabled`; discovery never auto-ingests; qa isolated.
- **D-023 · Per-agent Level 1 DuckDB files** (`l1/{facility}/{agent_id}.duckdb`) — zero cross-agent ops in L1 → parallel chains, disposable recovery, physical isolation. Tunables: worker concurrency × `memory_limit`. Open→work→close per task.
- **D-024 · Handoff medium: atomic partitioned Parquet** `handoff/{facility}/{agent_id}/{date}/` → **dedicated Level 2 DuckDB**. — No writer contention; the L1/L2 NEVER physically enforced; temp-write+rename.
- **D-025 · Per-agent chains for bin/consolidation/events (correctness boundary); load/enrich may batch.**
- **D-026 · Event = burst group**; burst gap **600 s** (validated, tunable, never hardcoded); sweep-collapse ≠ burst-grouping (two scales).
- **D-027 · Two-pass power binning LOCKED (Guy)** — Pass-1 fast bucketing over all + Pass-2 edge reconciliation over boundary cases only; split is load-bearing for processing time. Benchmark (WP-S5) sizes the Pass-2 budget only, never questions the structure. Pass-2 *calc* remains OPEN (hunt/choose).
- **D-028 · Noise filter: presence×activity with the low-duty-cycle exception** (present-but-sparse = power-saving tracker = PASS) + signature filter + over-filter safeguard. Exception is NEVER-remove.
- **D-029 · Level 1 / Level 2 hard boundary** — prep stops at the event dataset; CV/scoring/classification/correlation/reports are a separate effort consuming the handoff contract.

- **D-044 · Level 1 execution model: ephemeral pipeline job (Guy)** — hybrid: **resident** = Postgres + Django web (dashboard/admin/onboarding must be reachable between runs); **ephemeral** = the pipeline runs as a scheduled job container (same image, `docker compose run --rm pipeline manage.py run_level1`) that syncs → lands → enriches → bins → filters → builds events → exports Parquet → exits. Non-zero exit = alert; next scheduled run retries (halt-on-failure maps to exit codes; L1 DuckDB files disposable). **Celery + Redis removed from the Level 1 critical path** — per-agent chains become sequential code parallelized with a process pool inside the job; completion-gating is function order; overlap prevented by a run lock; scheduled by systemd timer/cron. Celery may return with Level 2 (beat-scheduled reports) or as a second ephemeral job. — Rationale: Level 1 is periodic batch by design; run-to-completion gives clean memory, pinned image per run, fewer moving parts for a single operator. Run cadence: OPEN (Q-19).

- **D-045 · Agent uploader is an ephemeral timer job (Guy)** — per-host model becomes: **resident** = 4 scanner processes (hardware capture must run continuously); **ephemeral** = `uplink-upload.timer` → oneshot every ~5 min (matches rotation cadence): scan `pending/` → edge filter + feed split → gzip → PUT → write heartbeat → exit. **Consequences:** file-watcher module deleted (whole bug class gone); circuit breaker retired — `pending/` IS the retry queue by construction (failed run exits leaving files; timer cadence = backoff); DLQ narrows to genuinely poison files only; L4.5 rule changes and config reload happen naturally per run; heartbeat rides the upload run (carries queue depth + disk %, so backlog growth is visible); disk-ceiling check runs at a defined point. Heartbeat interval = upload cadence (answers half of Q-09). Symmetric with D-044: resident only what must be (capture / dashboard), ephemeral everything periodic.
